# 主库和从库（读写分离）

> 起因：阅读 [BaseRepository.php](BaseRepository.php) 时看到 `dbFecshopSlave`、`$useSlave` 等字段，引出主从库的概念。

---

## 基本概念

主库和从库是数据库的一种**读写分离**架构。

| 角色 | 英文 | 职责 | 说明 |
|---|---|---|---|
| 主库 | Master | 负责**写**（INSERT / UPDATE / DELETE），也能读 | 数据的"权威源头" |
| 从库 | Slave | 负责**读**（SELECT） | 数据从主库**复制**而来，可有多个 |

数据流向是单向的：

```
       写入
用户 ──────► 主库 (Master)
                  │
                  │ 复制 (binlog 同步)
                  ▼
      读取     从库 (Slave)  ← 可以有多个
用户 ◄────────
```

---

## 为什么要分主从

一句话：**读远多于写**。大多数业务里，读请求（浏览商品、看订单）是写请求（下单、改资料）的几十倍。

- 一台主库扛不住全部读压力 → 把读分给多台从库，主库只管写。
- 从库可以横向扩展（加机器），读性能几乎线性提升。
- 顺带做了容灾：主库挂了，从库还有一份数据。

---

## 在 BaseRepository 里怎么体现

看 [BaseRepository.php:19](BaseRepository.php#L19) 到 [BaseRepository.php:53](BaseRepository.php#L53)：

```php
private static $useSlave = null;   // 标记这次要不要走从库

public function getDb()
{
    // useSlave 为真 → 用从库连接；否则返回 null（走框架默认，通常是主库）
    return self::$useSlave ? Yii::$app->dbFecshopSlave : null;
}

protected function getSlaveDb()
{
    return Yii::$app->dbFecshopSlave;   // 明确拿从库连接
}
```

- `dbFecshopSlave` 就是配置里指向**从库**的那个数据库连接。
- 调用 `instance(true)` 时 `$useSlave = true`，后续查询走从库；默认 `false` 走主库。
- 把"走不走从库"做成一个可控开关，而不是无脑全走从库——原因见下面的"主从延迟"。

---

## 一个关键坑：主从延迟

复制不是瞬时的，从库数据会比主库**慢一点点**（毫秒到秒级）。所以有个经典问题：

> 用户刚下单（写主库），马上跳转到"我的订单"页（读从库）——可能查不到刚下的单，因为还没同步过来。

对这种"写完立刻要读"的场景，通常**强制走主库读**。这也是为什么代码把"走不走从库"做成一个可控开关（`$useSlave`）。

---

## 前端类比

- 主库 ≈ 后端**唯一可写的 API**。
- 从库 ≈ 挂在前面的**多个 CDN / 只读缓存节点**——读走缓存快且能分流，但缓存有更新延迟。
- 强一致的读（写完马上读）≈ 需要**回源**，绕过缓存直接打主库。
