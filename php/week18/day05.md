## 1. 学习内容

### 1.1 为什么要评估

前四天你搭了一条链：BM25 → Hybrid → Rerank → FAQ Agent。每一步你都调了参数（权重、Top-k、temperature）。问题来了：

> 你怎么知道这些调整是变好了还是变差了？

靠感觉不行——今天觉得好，明天换个问题就崩。评估的意义是：**用可复现的数字，客观衡量系统好坏，让每次改动都能被验证。**

小白重点：评估之于 RAG，就像单元测试之于代码。没有测试，你不敢重构；没有评估，你不敢调参。

---

### 1.2 RAG 评估分两层

答错可能出在两个地方，要分开评估：

```text
用户问题 ─► 【检索层】拿对文档了吗？ ─► 【生成层】答对了吗？
                  ↑ 评估点 1              ↑ 评估点 2
```

| 层 | 问什么 | 典型指标 |
|---|---|---|
| 检索层 | 正确文档进 Top-k 了吗 | Recall@k、Hit Rate |
| 生成层 | 最终答案对不对 | 准确率（人工/自动判分） |

小白重点：如果检索层就没召回正确文档，生成层再强也白搭（巧妇难为无米之炊）。所以出问题先查检索层。

---

### 1.3 关键指标解释

**Hit Rate@k**：正确文档出现在 Top-k 里的比例。

```text
Hit Rate@3 = 正确文档进了 Top-3 的题数 / 总题数
```

**准确率（Accuracy）**：最终答案正确的比例。

```text
准确率 = 答对的题数 / 总题数
```

对比：

| 指标 | 衡量什么 | 例子 |
|---|---|---|
| Hit Rate@3 | 检索够不够好 | 10 题里 8 题的答案文档进了 Top-3 → 80% |
| 准确率 | 端到端够不够好 | 10 题里 7 题最终答对 → 70% |

本周完成标准是准确率 > 70%。

---

### 1.4 准备评估集（10 题）

评估集就是一批“问题 + 标准答案 + 应命中的文档 id”：

```python
# eval_set.py
EVAL_SET = [
    {
        "query": "怎么重置密码",
        "expected_doc_id": 0,          # 正确答案应来自 0 号文档
        "expected_keywords": ["设置", "重置", "验证码"],  # 答案应包含的关键点
    },
    {
        "query": "支付失败怎么办",
        "expected_doc_id": 1,
        "expected_keywords": ["余额", "支付方式", "重试"],
    },
    {
        "query": "退款多久到账",
        "expected_doc_id": 2,
        "expected_keywords": ["3-7", "工作日"],
    },
    # ... 补齐到 10 题
]
```

小白重点：评估集要**尽量贴近真实用户问法**，包括口语化、错别字、同义词。这样评估结果才有参考价值。

### 1.5 写自动跑分脚本

```python
from faq_agent import FaqAgent
from eval_set import EVAL_SET


def evaluate(agent: FaqAgent, docs: list[str]):
    hit_count = 0      # 检索命中数
    correct_count = 0  # 答案正确数
    failures = []

    for item in EVAL_SET:
        query = item["query"]

        # ---- 检索层评估：正确文档进 Top-3 了吗 ----
        contexts = agent.retrieve(query, top_k=3)
        retrieved_ids = [c["id"] for c in contexts]
        hit = item["expected_doc_id"] in retrieved_ids
        if hit:
            hit_count += 1

        # ---- 生成层评估：答案含关键点吗（简易自动判分）----
        answer = agent.answer(query)["answer"]
        correct = all(kw in answer for kw in item["expected_keywords"])
        if correct:
            correct_count += 1

        # ---- 记录失败 case，标注失败在哪一层 ----
        if not correct:
            failures.append({
                "query": query,
                "retrieval_ok": hit,
                "answer": answer,
                "reason": "检索未召回" if not hit else "生成错误",
            })

    total = len(EVAL_SET)
    print(f"检索命中率 Hit@3: {hit_count}/{total} = {hit_count/total:.0%}")
    print(f"端到端准确率: {correct_count}/{total} = {correct_count/total:.0%}")
    print("\n失败 case：")
    for f in failures:
        print(f"  Q: {f['query']}  失败层: {f['reason']}")
    return failures
```

小白重点：关键在最后——每个失败 case 都标了 `reason`：如果检索没命中就是“检索未召回”，检索命中了但答案错就是“生成错误”。这一步让你知道该往哪修。

---

### 1.6 失败 case 分析：对症下药

拿到失败清单后，按层归类，各有各的修法：

| 失败层 | 现象 | 优化方向 |
|---|---|---|
| 检索未召回 | 正确文档没进 Top-3 | 调 Hybrid 权重、增大召回 Top-k、优化分词、加同义词 |
| 生成错误 | 文档召回了但答案错 | 改 prompt、降 temperature、加 Rerank 提高文档排序 |

举例分析：

```text
失败 case: "咋改登录密码"
  检索层: 未召回（0 号文档没进 Top-3）
  原因: BM25 分词把"咋改"当成生僻词，没匹配到"重置密码"
  修法: 增加同义词映射（咋改→重置/修改），或提高语义检索权重
```

小白重点：

> 失败分析的价值不在于“我错了 3 题”，而在于“这 3 题错在检索还是生成、下一步该改哪里”。能说出这一句，今天才算真正掌握。

---

### 1.7 自动判分 vs 人工判分

上面的 `all(kw in answer ...)` 是最简易的自动判分（关键词包含）。它有局限：

| 判分方式 | 优点 | 缺点 |
|---|---|---|
| 关键词包含 | 快、可自动化 | 答案换个说法就误判为错 |
| 人工判分 | 准 | 慢、不可规模化 |
| LLM 判分 | 较准、可自动 | 有成本、偶尔不稳 |

小白重点：入门用关键词判分就够跑出大致数字。真要严谨，可以再叠一层“LLM 当裁判”，让 LLM 判断答案和标准答案是否语义一致。

## 4. JS/Node.js 类比

| Python / 评估 | Node.js 类比 | 说明 |
|---|---|---|
| 评估集 EVAL_SET | 测试用例数组 | 每条带输入和期望 |
| evaluate() | 测试 runner（jest/mocha） | 批量跑并统计 |
| Hit Rate / 准确率 | 测试通过率 | 通过数 / 总数 |
| 失败 case 分析 | 失败测试的 stack trace | 定位错在哪 |
| LLM 判分 | 自定义 assert 断言 | 判断结果是否符合预期 |

小白重点：整套评估流程就是给 RAG 系统写测试。Node 里你写 `expect(result).toContain(...)`，这里写 `all(kw in answer ...)`，本质完全一样——用可复现的断言代替“感觉还行”。

## 8. 今日自测题

### 8.1 为什么 RAG 评估要分检索层和生成层？

参考答案：

> ✅ 因为答错可能出在两个地方：检索没召回正确文档，或召回了但生成答错。分层评估才能定位问题——检索层就错了，生成层再强也没用，得先修检索。

---

### 8.2 Hit Rate@3 和准确率有什么区别？

参考答案：

> ✅ Hit Rate@3 衡量“正确文档是否进了 Top-3”，是检索层指标；准确率衡量“最终答案对不对”，是端到端指标。前者好但后者差，说明问题在生成层。

---

### 8.3 一个失败 case 检索命中了但答案错，该怎么优化？

参考答案：

> ✅ 说明问题在生成层。可以改 prompt（约束更严）、降 temperature、或加强 Rerank 让最相关文档排更前，减少 LLM 被无关内容干扰。

---

### 8.4 关键词包含判分有什么坑？

参考答案：

> ✅ 答案换个说法（同义词、改写）就会被误判为错，即使意思对。更严谨可以用 LLM 判断答案与标准答案是否语义一致。

---

### 8.5 评估的核心价值是什么？

参考答案：

> ✅ 用可复现的数字客观衡量系统好坏，让每次调参改动都能被验证，并能定位失败原因、指导下一步优化。就像给代码写测试。
