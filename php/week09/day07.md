### 1.3 必须掌握的权限边界

| 场景 | 必须校验什么 |
|---|---|
| 查询订单详情 | 订单 `user_id` 是否等于当前用户 |
| 发起支付 | 支付单/订单是否属于当前用户 |
| 查询用户资料 | 是否本人或具备管理权限 |
| 修改手机号/邮箱 | 是否本人且完成验证码校验 |
| 申请售后 | 售后对应订单是否属于当前用户 |
| 查看通知 | 通知是否属于当前用户 |

核心原则：

```text
不要相信前端传来的 user_id；后端必须从 token/session 解析当前用户。
```

---

### 1.4 必须掌握的隐私保护点

| 数据 | 风险 | 建议 |
|---|---|---|
| 手机号 | 泄露可被骚扰/撞库 | 返回时脱敏 |
| 邮箱 | 泄露隐私 | 返回时脱敏 |
| 地址 | 个人隐私 | 仅订单必要场景返回 |
| token | 泄露后可冒充登录 | 不打印日志，不放 URL |
| 验证码 | 被绕过登录/注册 | 不打印日志，短 TTL |
| 第三方 open_id | 可关联身份 | 谨慎返回前端 |
| 密码 hash | 泄露后可被离线攻击 | 永不返回，日志不打印 |

## 8. 今日自测题

### 8.1 用户域验收的核心到底是什么？

参考答案：

> ✅ 不是“能登录”，而是登录后 `user_id` 可信、资源归属正确、敏感资料受保护，且订单、支付、售后都不能越权访问。

---

### 8.2 本周有哪三条身份入口链路？

参考答案：

> ✅ quickLogin（手机号/验证码等快捷凭证）、注册 Node 链（手机号/邮箱/密码/验证码）、OAuth 第三方登录（第三方 code）。三条入口不同，但最终都要落到本地 `user_id` + 本系统 token/session。

---

### 8.3 为什么后端不能相信前端传来的 `user_id`？

参考答案：

> ✅ 前端传的 `user_id` 可以被随意篡改，A 用户可能伪造成 B 用户。后端必须从 token/session 解析出当前用户，再按这个 `user_id` 做资源归属校验。

---

### 8.4 查询订单详情时应该怎么防越权？

参考答案：

> ✅ 查询条件里必须同时带上 `user_id`，例如 `findOne(['id' => $orderId, 'user_id' => $currentUserId])`，而不是只按 `order_id` 查，否则会让别人看到不属于自己的订单。

---

### 8.5 哪些用户敏感字段绝对不能打印到日志？

参考答案：

> ✅ token/refresh token、验证码、密码及密码 hash 等。它们一旦进日志就可能被用来冒充登录、绕过注册或离线破解，返回前端时手机号、邮箱等也要按场景脱敏。
