## 1. 学习内容

### 1.1 什么叫「连贯」

连贯 = 后面的轮次能正确使用前面轮次的信息。设计测试用例的关键，是让每一轮**依赖**前面的轮次：

```text
第 1 轮：我叫李雷，帮我推荐一款笔记本
第 2 轮：预算 6000 以内           ← 依赖第1轮的「推荐笔记本」
第 3 轮：主要用来写代码            ← 继续追加约束
第 4 轮：那第二款怎么样            ← 依赖第3轮模型给的列表
第 5 轮：我叫什么名字来着？        ← 直接考第1轮的「李雷」
```

小白重点：

> 好的多轮测试，一定要有「回指」——用「刚才那个」「我说的」「第二个」这种词，逼模型用历史。如果每轮都是独立问题，测不出记忆。

---

### 1.2 组装完整链路

把三天的组件拼起来，一次对话要经过：

```text
用户输入
  → append 到 session
  → 取历史
  → (超长?) 摘要压缩
  → (仍超窗?) 窗口截断
  → 拼成 messages 发给 LLM
  → 回复 append 到 session
```

代码骨架：

```python
class ChatEngine:
    def __init__(self, sm, window=10, max_msgs=20):
        self.sm = sm
        self.window = window
        self.max_msgs = max_msgs

    def send(self, sid: str, user_text: str) -> str:
        self.sm.append(sid, "user", user_text)

        history = self.sm.get_messages(sid)

        # 1. 太长先压缩
        if len(history) > self.max_msgs:
            history = compress(history)
            self.sm.replace(sid, history)

        # 2. 再按窗口截断（只影响发出去的，不改存储）
        context = apply_window(history, self.window)

        # 3. 发给（无状态的）LLM
        reply = call_llm(context)

        self.sm.append(sid, "assistant", reply)
        return reply
```

小白重点：

> 压缩会**改写存储**（replace），截断只**影响这次发出去的内容**（不改存储）。顺序是先压缩、后截断。

### 1.3 用假 LLM 跑通

为了测「记忆链路」而不是「模型智商」，用一个会「复读历史」的假 LLM：

```python
def call_llm(messages: list[dict]) -> str:
    # 假模型：把它看到的所有 user 内容回显出来，方便验证记忆
    seen = [m["content"] for m in messages if m["role"] == "user"]
    return f"[我记得你说过]: {' | '.join(seen)}"
```

这样如果第 5 轮的回复里还包含第 1 轮的「李雷」，就证明历史确实传进去了。

### 1.4 跑一遍 5 轮

```python
sm = SessionManager()
sid = sm.create()
sm.append(sid, "system", "你是购物助手")
engine = ChatEngine(sm)

turns = [
    "我叫李雷，帮我推荐一款笔记本",
    "预算 6000 以内",
    "主要用来写代码",
    "那第二款怎么样",
    "我叫什么名字来着？",
]

for i, t in enumerate(turns, 1):
    reply = engine.send(sid, t)
    print(f"第{i}轮 用户: {t}")
    print(f"第{i}轮 助手: {reply}\n")
```

第 5 轮的假回复里应能看到「李雷」，说明第 1 轮信息还在。

### 1.5 观察每轮实际发出的 messages

调试记忆最有效的手段：把每次发给 LLM 的 `context` 打印出来。

```python
def send(self, sid, user_text):
    self.sm.append(sid, "user", user_text)
    history = self.sm.get_messages(sid)
    context = apply_window(history, self.window)

    print("=== 本轮发给 LLM 的 messages ===")
    for m in context:
        print(f'  {m["role"]}: {m["content"][:30]}')
    ...
```

小白重点：

> 记忆出 bug 时，第一件事永远是「打印发出去的 messages」。90% 的问题是历史没传对，而不是模型太笨。

---

### 1.6 故意制造「断链」

理解正例后，制造反例更能加深印象。三种常见断链：

| 断链原因 | 现象 | 对应哪天的知识 |
|---|---|---|
| 每轮只发当前消息，不带历史 | 模型完全失忆 | Day01：没存/没取历史 |
| 窗口太小（如只留 2 条） | 早期信息被切掉 | Day02：窗口截断 |
| 压缩 prompt 没保护名字 | 摘要里丢了「李雷」 | Day03：摘要 prompt |

练习时，把窗口 `window` 改成 2，重跑，你会看到第 5 轮答不出名字——这就是「断链」的直观感受。

### 1.7 连贯性验收清单

跑完 5 轮，逐条核对：

```text

[ ] 第 5 轮能正确答出「李雷」

```

## 4. JS/Node.js 类比

| Python / AI 后端 | Node.js 类比 | 说明 |
|---|---|---|
| `ChatEngine` | Express 里处理一轮请求的 service | 每次请求带上历史 |
| 每轮带历史发给 LLM | 每次请求带 session cookie | 无状态服务靠外部状态续上 |
| 打印发出去的 messages | 打印请求体做调试 | 定位数据传递问题 |
| 断链（不带历史） | 忘了带 cookie，登录态丢失 | 服务端「认不出你」 |

一句话类比：

> 多轮对话像每次 HTTP 请求都带上 session：LLM（服务端）本身不记事，全靠你每次把「历史」（session）一起交上去，它才「认得」你。
