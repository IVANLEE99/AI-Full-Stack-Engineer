## 1. 学习内容

### 1.1 为什么要写设计文档

代码会过期，设计思路不会。文档的价值：

| 价值 | 说明 |
|---|---|
| 对齐 | 团队知道「记忆」是怎么实现的 |
| 复用 | 换项目也能照着搭 |
| 排错 | 出问题时对照文档找断点 |
| 面试/复盘 | 能讲清一套完整方案 |

小白重点：

> 判断文档好不好只有一个标准：**一个没参与开发的人，能不能照着它把这套 Memory 重新实现出来。**

---

### 1.2 Memory 设计文档骨架

推荐结构：

```text
# Memory 设计文档
## 1. 背景与问题
   - LLM 是无状态的，为什么需要外部记忆
## 2. 总体架构
   - Session 存储 + 上下文窗口 + 摘要压缩（三件套）
   - 一张链路图
## 3. 组件设计
   ### 3.1 SessionManager（存什么、怎么存、sessionId 规则）
   ### 3.2 上下文窗口（为什么截断、保留策略、system 保护）
   ### 3.3 摘要压缩（何时触发、压什么、prompt、风险）
## 4. 数据结构
   - message 的字段、session 的字段
## 5. 关键取舍
   - 窗口大小、压缩阈值、存内存还是存 DB
## 6. 已知风险与 TODO
   - 并发、持久化、token 精确计数
```

---

### 1.3 逐节要点：背景与架构

背景节，用一句话点明矛盾：

```text
LLM 每次调用都是独立的、无状态的（不记得上一句）。
但产品需要「连续对话」。
所以我们在模型外面，用工程手段维护会话状态。
```

架构节，画文字链路图：

```text
用户消息
  │
  ▼
[SessionManager] ── 追加/读取历史
  │
  ▼
history 太长? ──是──▶ [摘要压缩] 改写存储
  │否                     │
  ▼◀─────────────────────┘
[上下文窗口] 截断到最近 N 条（保护 system）
  │
  ▼
拼成 messages ──▶ [LLM] ──▶ 回复
  │
  ▼
回复写回 [SessionManager]
```

---

### 1.4 逐节要点：组件与数据结构

组件节，每个组件写清「职责 + 关键决策」。数据结构节用表：

message 结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| role | string | system / user / assistant |
| content | string | 消息文本 |
| ts | int | 时间戳（可选） |

session 结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | sessionId |
| messages | list | 消息数组 |
| summary | string | 已压缩的历史摘要（可选） |
| updated_at | int | 最后活跃时间 |

---

### 1.5 逐节要点：关键取舍

把「参数怎么定的」写清楚，这是文档最有价值的部分：

| 取舍点 | 选项 | 我的选择与理由 |
|---|---|---|
| 窗口大小 N | 6 / 10 / 20 | 先取 10，够覆盖典型客服对话 |
| 压缩阈值 | 20 / 30 条 | 超 20 条触发，避免频繁调 LLM |
| 存储位置 | 内存 / Redis / DB | 学习期内存；生产用 Redis 带 TTL |
| token 计数 | 估算 / 精确 tokenizer | 先字符/4 估算，生产接精确计数 |

小白重点：

> 文档里写「选了什么」不重要，写「**为什么这么选、放弃了什么**」才是资深的体现。

---

### 1.6 类比打卡：Web 会话 ↔ LLM 记忆

这是今天的「类比日」核心。填完这张表，你就把两套心智模型打通了：

| Web 后端会话 | LLM 记忆 | 相同点 |
|---|---|---|
| session store（Redis） | SessionManager | 都在服务外部存状态 |
| sessionId（cookie 里） | sessionId | 都用一个 id 找回历史 |
| 每次请求带 cookie | 每次调用带历史 messages | 无状态服务靠外部状态续上 |
| session 过期 TTL | 会话闲置清理 | 都要回收避免内存膨胀 |
| session 太大裁剪 | 上下文窗口截断 | 都要控制单次携带的数据量 |
| —— | 摘要压缩 | LLM 特有：可以用模型压缩历史 |

一句话打卡结论：

> LLM 记忆 ≈ 无状态 Web 服务的会话管理，只是多了一个「用模型压缩历史」的独门武器。

## 4. JS/Node.js 类比

| Python / AI 后端 | Node.js 类比 | 说明 |
|---|---|---|
| Memory 设计文档 | 架构 RFC / README | 团队对齐用的设计说明 |
| Session 三件套 | express-session + Redis + 中间件 | 会话管理的完整方案 |
| 类比打卡 | 迁移旧知识到新场景 | 用熟悉的模型理解新东西 |
| 关键取舍表 | ADR（架构决策记录） | 记录「为什么这么选」 |

一句话类比：

> 写 Memory 文档，就像给 Node 团队写一份「我们的 session 方案是怎么设计的」RFC——别人读完能接手。
