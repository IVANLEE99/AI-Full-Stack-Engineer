## 1. 学习内容

### 1.1 支付状态和订单状态有什么区别？

订单状态关注业务订单：

```text
待支付、待发货、已发货、已完成、已取消
```

支付状态关注资金处理：

```text
待支付、处理中、成功、失败、取消、退款中、已退款
```

两者关系：

| 支付状态 | 可能影响订单状态 |
|---|---|
| 支付成功 | 订单从待支付变为已支付/待发货 |
| 支付失败 | 订单通常仍是待支付，允许重试 |
| 支付取消 | 订单可能仍待支付或取消，取决于业务 |
| 已退款 | 订单进入退款/售后相关状态 |

小白重点：支付失败不一定等于订单取消。

---

### 1.2 支付主线状态

一个常见支付流程：

```text
创建支付单
  ↓
待支付
  ↓
调用支付渠道
  ↓
处理中
  ↓
支付成功
```

也可能失败：

```text
处理中
  ├─ 成功 → 支付成功
  ├─ 失败 → 支付失败
  ├─ 用户取消 → 已取消
  └─ 超时 → 已超时
```

如果涉及退款：

```text
支付成功
  ↓
退款中
  ↓
已退款 / 退款失败
```

---

### 1.3 创建、确认、捕获是什么意思？

不同支付平台术语不同，但常见概念包括：

| 概念 | 含义 |
|---|---|
| create | 创建支付意图/支付单 |
| confirm | 确认用户同意支付或提交支付信息 |
| authorize | 授权冻结资金 |
| capture | 真正扣款 |
| complete/succeeded | 支付完成 |
| failed | 支付失败 |
| refunded | 已退款 |

Stripe PaymentIntent 里可能看到：

```text
requires_payment_method
requires_confirmation
requires_action
processing
succeeded
canceled
```

项目内部通常会把第三方状态映射成自己的统一状态。

---

### 1.4 渠道状态如何映射到内部状态？

不同平台状态不一样，所以需要映射。

示例：

| 渠道 | 第三方状态 | 内部支付状态 |
|---|---|---|
| Stripe | `succeeded` | 支付成功 |
| Stripe | `processing` | 处理中 |
| Stripe | `canceled` | 已取消 |
| Braintree | `settled` | 支付成功 |
| Braintree | `failed` | 支付失败 |
| PayPal | `COMPLETED` | 支付成功 |

这就是渠道笔记的价值：让团队知道每个渠道状态如何转成内部状态。

---

### 1.5 支付成功后如何更新订单？

支付成功通常来自回调：

```text
第三方支付平台回调
  ↓
验签
  ↓
找到支付单
  ↓
校验金额/币种/订单号
  ↓
支付单状态改为成功
  ↓
订单状态从待支付改为已支付/待发货
```

注意：订单状态更新要幂等。

如果同一个支付成功回调来两次：

```text
第一次：待支付 → 已支付
第二次：发现已支付 → 不重复处理，直接返回成功
```

---

### 1.6 支付状态机图

基础版：

```text
创建支付单
  ↓
待支付
  ↓ 发起渠道支付
处理中
  ├─ 回调成功 → 支付成功 → 更新订单已支付
  ├─ 回调失败 → 支付失败 → 订单仍待支付/允许重试
  ├─ 用户取消 → 已取消
  └─ 超时未支付 → 已超时

支付成功
  ↓ 申请退款
退款中
  ├─ 退款成功 → 已退款 → 更新订单售后/退款状态
  └─ 退款失败 → 支付成功/退款失败待处理
```

---

### 1.7 异常状态必须覆盖

支付状态机不能只画成功路径。

必须考虑：

| 异常 | 风险 |
|---|---|
| 支付失败 | 用户需要重试或换渠道 |
| 支付超时 | 订单是否关闭？支付单是否失效？ |
| 用户取消 | 前端如何展示？ |
| 重复回调 | 不能重复更新订单/发货 |
| 金额不一致 | 必须拒绝更新成功 |
| 已支付订单再次收到失败回调 | 不能状态回退 |
| 退款失败 | 售后状态如何处理 |

---

### 1.8 状态机与渠道笔记怎么结合？

渠道笔记建议写：

| 渠道 | 创建 API | 成功状态 | 失败状态 | 回调事件 | 内部状态映射 |
|---|---|---|---|---|---|
| Stripe | PaymentIntent create | `succeeded` | `canceled` / failed | `payment_intent.succeeded` | success |
| Braintree | Transaction sale | `settled` | `failed` | webhook transaction | success/fail |
| PayPal | Order capture | `COMPLETED` | `VOIDED` | payment capture completed | success |
