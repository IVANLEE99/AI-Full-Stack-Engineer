### 1.3 支付域必须掌握的 7 个风险

| 风险 | 你要能解释什么 |
|---|---|
| 回调验签 | outer 接口不能只靠 URL 隐藏，必须验证平台签名 |
| 重复回调 | 第三方可能多次通知，同一事件只能处理一次 |
| 金额校验 | 回调金额必须和订单/支付单金额一致 |
| 状态回退 | 已成功支付不能被失败回调改回失败 |
| 前端不可信 | 前端支付成功提示不能作为最终依据 |
| 密钥泄露 | secret key/webhook secret 不能给前端 |
| 渠道差异 | 不同支付平台状态要映射成统一内部状态 |

### 1.5 回调验签和重复回调为什么是重点？

支付回调是外部系统主动请求你的服务。

如果不验签，攻击者可能伪造：

```text
“订单 1001 已支付成功”
```

如果不幂等，支付平台重复通知可能导致：

- 重复更新订单
- 重复发货
- 重复发送消息
- 重复发放权益

所以回调处理必须满足：

```text
可信来源 + 同一事件只处理一次 + 金额一致 + 状态只前进不回退
```

### 1.6 预习 MQ/Webhook 异步处理

下周会进入 MQ/Webhook 或异步处理。

支付成功后不一定所有事情都同步完成：

- 通知订单服务
- 发送支付成功消息
- 发放权益
- 发送邮件/短信
- 触发库存/发货流程
- 写审计日志

这些适合通过 MQ 异步处理。

预习问题：

| 问题 | 为什么重要 |
|---|---|
| 为什么回调接口要快速返回？ | 防止支付平台重试 |
| 为什么支付成功后要发 MQ？ | 解耦订单、通知、权益 |
| MQ 消费失败怎么办？ | 需要重试和死信 |
| 消息重复怎么办？ | 消费端也要幂等 |
| Webhook 和 MQ 有什么关系？ | webhook 触发内部异步流程 |
