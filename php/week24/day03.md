## 1. 学习内容

### 1.1 为什么"可复现"是交付的核心

昨天你写了 README，但文档写得再漂亮，没跑过一遍就不算数。今天要做的是"验收自己的文档"。

一个残酷的事实：

```text
80% 的 README 在别人第一次照着跑时都会卡住。
原因几乎都是：作者跳过了自己早就装好/配好的那一步。
```

比如你本地早就 `cp .env.example .env` 过了，写文档时就忘了写这一步；你本地前端 `dist` 早就 build 好了，别人拉下来根本没有 `dist`。

可复现的意思是：

> 换一个人、换一台机器，照着同一份文档，能得到同样的可运行结果。

用一句话记住：

> 跑通不是终点，"别人也能跑通"才是。

---

### 1.2 准备一个"尽量干净"的验证环境

要检验可复现性，就不能用你那台"什么都装好了"的开发机的既有状态。几种由易到难的方式：

| 方式 | 干净程度 | 适合 |
|---|---|---|
| 新建一个空目录，重新 `git clone` | 中 | 快速验证代码可复现 |
| 删掉本地镜像和数据卷重来 | 较高 | 验证 Docker 构建可复现 |
| 用一台新虚拟机 / 云主机 | 高 | 最接近"别人第一次" |
| 让同学/AI 照着 README 跑 | 高 | 真实的第三方视角 |

最实用的做法组合：

```bash
# 1. 换个干净目录，重新克隆
cd /tmp
git clone <仓库地址> gp-verify
cd gp-verify

# 2. 清掉可能残留的旧镜像和卷（谨慎，会删数据）
docker compose down -v 2>/dev/null || true

# 3. 完全照 README 来，一步都不能跳
```

⚠️ 安全提醒：`docker compose down -v` 会删数据卷。只在"验证专用、无重要数据"的环境里用。

小白重点：验证时最重要的纪律是"不许用脑子里的隐藏知识"。只要你发现自己"手动多做了一步 README 里没写的操作"，立刻记下来补进 README。

---

### 1.3 严格照 README 执行并记录

打开昨天的 README，逐条命令执行，边执行边记录：

```text
步骤 1：cp .env.example .env
  → 结果：OK
步骤 2：编辑 .env 填密钥
  → 结果：OK（填了测试用 Key）
步骤 3：cd frontend && npm run build
  → 结果：❌ 报错，缺 node_modules
  → 修正：README 漏了 npm install，已补
步骤 4：docker compose up -d
  → 结果：OK，5 个容器都起来
步骤 5：访问 http://localhost:8080
  → 结果：OK，看到首页
```

验证记录表模板：

| 步骤 | 命令 | 结果 | 卡点 / 修正 |
|---|---|---|---|
| 克隆 |  |  |  |
| 配环境 |  |  |  |
| 构建前端 |  |  |  |
| 启动 compose |  |  |  |
| 检查状态 |  |  |  |
| 访问首页 |  |  |  |

启动后先确认所有服务都健康：

```bash
docker compose ps
```

期望看到所有服务 `Up`（或 `running`）状态。如果有服务是 `Exited` 或 `Restarting`，说明它崩了，去看日志：

```bash
docker compose logs <服务名>
```

小白重点：MySQL、向量库这类有状态的服务首次启动要初始化，可能要等十几秒。太快访问会报连接失败，稍等再试。真实项目里会用 `healthcheck` + 重试来解决，验证时先手动等一会儿即可。

### 1.4 走通核心业务流程

服务起来只是"活着"，还要验证"功能可用"。以智能文档问答毕业项目为例，核心流程是：

```text
打开首页
  ↓
上传一个文档
  ↓
等待文档解析 + 向量化
  ↓
输入一个问题
  ↓
系统检索 + 调大模型
  ↓
返回带出处的答案
```

逐步验证并记录：

| 流程 | 操作 | 期望结果 | 实际结果 | 通过 |
|---|---|---|---|---|
| 打开首页 | 访问 8080 | 页面正常渲染 |  |  |
| 上传文档 | 选文件上传 | 提示上传成功 |  |  |
| 文档处理 | 等待处理 | 状态变为已就绪 |  |  |
| 提问 | 输入问题 | 返回答案 |  |  |
| 查看出处 | 看引用来源 | 显示来源文档片段 |  |  |

也可以用命令行直接打接口，验证后端本身：

```bash
# 健康检查
curl http://localhost:8080/health

# 提问接口（示例）
curl -X POST http://localhost:8080/api/ask \
  -H "Content-Type: application/json" \
  -d '{"question":"这个文档讲了什么？"}'
```

小白重点：至少测一个"错误情况"，比如上传空文件、问一个文档里没有的问题，看系统是否给出合理提示而不是白屏或 500。演示时能坦诚展示"边界情况也处理了"，比只演"完美路径"更有说服力。

---

### 1.5 设计 Demo 演示脚本

录 Demo 前先想清楚"演什么、演多久"。不要打开录屏就乱点。

一个 2-3 分钟 Demo 的推荐结构：

```text
[0:00-0:20] 一句话介绍：这是什么项目，解决什么问题
[0:20-0:50] 展示一键启动：docker compose up -d + docker compose ps
[0:50-2:00] 走核心流程：上传文档 → 提问 → 看答案和出处
[2:00-2:30] 展示一个边界情况的处理（可选）
[2:30-2:50] 一句话收尾：技术栈 + 亮点
```

演示脚本模板：

| 时间 | 我要说的话 | 我要操作的动作 |
|---|---|---|
| 开头 | "这是一个……系统，解决……" | 展示首页 |
| 启动 | "整套系统用一条命令启动" | 敲 `docker compose up -d` |
| 核心 | "用户上传文档后可以提问" | 上传 + 提问 |
| 收尾 | "后端用 PHP，向量库用……" | 展示架构 |

对比表：

| 做法 | 效果 |
|---|---|
| 不写脚本直接录 | 卡壳、绕路、时长失控 |
| 先写脚本再录 | 流畅、重点突出、时长可控 |

小白重点：Demo 的目标观众是"没看过你代码的人"（面试官、同学、未来的你）。他们只关心"这东西能干什么、跑不跑得起来"，不关心你代码写得多巧。所以 Demo 演"能用"，不演"代码"。

---

### 1.6 录制 Demo（视频或 GIF）

工具选择：

| 类型 | 工具举例 | 适合 |
|---|---|---|
| 屏幕录像 | 系统自带录屏 / OBS | 有旁白、完整演示 |
| GIF | 各类 GIF 录制工具 | 放进 README，几秒钟看重点 |
| 终端录制 | asciinema | 纯命令行操作演示 |

录制建议：

- 分辨率别太大，1080p 足够，文字要看得清。
- 关掉无关通知，避免弹窗入镜。
- 命令行字体调大一点。
- 演示数据用脱敏内容，别出现真实姓名、真实密钥、真实内网地址。

录完把它放进项目：

```text
graduation-project/
├── docs/
│   └── demo.gif        # 或 demo.mp4
└── README.md           # 在顶部嵌入 demo
```

在 README 顶部嵌入（GIF 示例）：

```markdown
# Graduation Project

![演示](docs/demo.gif)
```

小白重点：GIF 适合放 README（打开就自动播放，几秒抓住重点）；完整视频适合面试/答辩讲解。两者可以都留一份。

### 1.7 回看验收：别人看得懂吗？

录完别急着收工，回看一遍，用"别人视角"检查：

| 检查点 | 通过标准 |
|---|---|
| 开头 10 秒 | 能说清"这是什么、干什么" |
| 启动演示 | 真的展示了从命令到服务起来 |
| 核心流程 | 完整走通，没有中途卡死跳过 |
| 画面清晰 | 文字看得清，节奏不拖 |
| 脱敏 | 无真实密钥/姓名/内网地址 |
| 时长 | 控制在 3 分钟内 |

如果回看发现"我自己都觉得没讲清楚"，就重录。Demo 是要长期用的作品集素材，值得多录一次。

## 4. JS/Node.js 类比

| PHP 毕业项目 Demo 验证 | Node.js 类比 | 说明 |
|---|---|---|
| 干净环境 `docker compose up` | 新机器 `npm install && npm start` | 验证可复现 |
| README 遗漏步骤 | 忘了写 `.env` / 前置依赖 | 作者盲区 |
| 走通核心流程 | 跑通关键 API 用例 | 功能可用 |
| 录 Demo GIF 放 README | npm 包 README 里的演示 GIF | 让人秒懂 |
| 健康检查接口 | Express 的 `/health` | 存活探针 |
| 脱敏演示数据 | 用 mock/示例数据 | 不泄露真实信息 |

注意：Node 通常一个进程 `npm start` 就起来；PHP 毕业项目是多容器编排，验证时要确认"所有服务"都起来（`docker compose ps` 全 Up），而不只是后端进程活着。

## 8. 今日自测题

### 8.1 为什么"我本地能跑"不算完成交付？

参考答案：

> ✅ "我本地能跑"只是自证，你的机器上有很多别人没有的隐藏状态（装好的依赖、配好的 .env、build 好的产物）。只有在干净环境里、换一个人照着同一份文档也能跑通，才叫可复现，才算真正交付。

---

### 8.2 怎么准备一个"尽量干净"的验证环境？

参考答案：

> ✅ 换一个空目录重新 `git clone`；或删掉本地旧镜像和数据卷（`docker compose down -v`，注意会删数据）；最彻底是用一台新虚拟机/云主机，或直接让同学/AI 照 README 跑。核心纪律是：不许用脑子里的隐藏知识，多做的每一步都要补进文档。

---

### 8.3 服务 `docker compose up` 后，怎么确认它们真的健康？

参考答案：

> ✅ 用 `docker compose ps` 看状态，所有服务应为 Up/running；如果有 Exited 或 Restarting，说明崩了，用 `docker compose logs <服务名>` 看原因。有状态服务（数据库、向量库）首次启动要初始化，需要等十几秒再访问。

---

### 8.4 录 Demo 前为什么要先写演示脚本？

参考答案：

> ✅ 没有脚本容易卡壳、绕路、时长失控。先写脚本能让演示流畅、重点突出、时长可控。Demo 的观众是没看过代码的人，所以要演"能用"而不是演"代码"，开头 10 秒就要说清"这是什么、干什么"。

---

### 8.5 Demo 里要注意哪些脱敏问题？

参考答案：

> ✅ 演示数据不能出现真实姓名、真实 API 密钥、真实内网域名或本地绝对路径。用占位符和示例数据。录制时关掉无关通知避免弹窗入镜。Demo 是要长期用作作品集的，泄露信息会有安全风险。
