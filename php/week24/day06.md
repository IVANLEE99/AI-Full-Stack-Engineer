### 1.2 整理 Git（一）：清理分支

一个杂乱的仓库，一眼就能看出"没整理过"。先看分支：

```bash
# 看本地所有分支
git branch

# 看远程分支
git branch -r

# 看当前状态
git status
```

清理原则：

| 分支 | 处理方式 |
|---|---|
| `main` / `master` | 保留，作为交付主干 |
| 已合并的功能分支 | 可以删除 |
| 半途而废的试验分支 | 删除或明确标注 |

删除已合并的本地分支：

```bash
# 删除一个已合并分支（安全，未合并会拒绝）
git branch -d old-feature
```

⚠️ 安全提醒：`git branch -D`（大写 D）会强制删除，即使没合并也删。除非确认这个分支的内容不要了，否则用小写 `-d`。删远程分支（`git push origin --delete xxx`）属于影响共享仓库的操作，动手前确认没人还在用。

小白重点：毕业交付通常保持在 `main` 一条干净主干即可。不需要为了"显得复杂"保留一堆废分支，清爽反而更专业。

### 1.3 整理 Git（二）：规范提交历史

提交历史是别人了解你工作过程的窗口。看一眼你的历史：

```bash
git log --oneline -20
```

好的提交信息 vs 差的：

| 差的提交 | 好的提交 |
|---|---|
| `update` | `feat: 新增订单创建接口` |
| `fix bug` | `fix: 修复未登录可访问订单详情的越权问题` |
| `1` / `aaa` / `测试` | `docs: 补充 Docker Compose 部署说明` |
| `保存一下` | `refactor: 将下单逻辑从 Controller 抽到 Service` |

一个常见的提交规范（Conventional Commits）前缀：

```text
feat:     新功能
fix:      修 bug
docs:     文档
refactor: 重构（不改功能）
test:     测试
chore:    杂项/构建/依赖
```

Node 类比：这和 Node 社区广泛使用的 Conventional Commits 完全一致，很多项目还用它自动生成 changelog。PHP 项目同样适用。

小白重点：历史已经写下的乱提交，不必强行改写（改写历史有风险）。重点是**从今往后**每次提交都写清楚。交付时保证最近的提交是规范的、能讲清楚做了什么即可。

### 1.4 整理 Git（三）：检查 .gitignore 和已追踪文件

这一步和脱敏强相关，先确认该忽略的都忽略了。

一个 PHP + 前端 + Docker 项目的 `.gitignore` 参考：

```text
# 依赖
/vendor/
/node_modules/
/frontend/node_modules/

# 环境与密钥
.env
.env.*
!.env.example

# 构建产物
/frontend/dist/

# 日志与缓存
*.log
/runtime/
/storage/logs/

# 编辑器/系统
.DS_Store
.idea/
.vscode/
```

关键：检查有没有"已经被提交进去"的敏感文件。`.gitignore` 只对"还没被追踪"的文件生效，已经提交的不会自动移除：

```bash
# 看当前被 git 追踪的文件里有没有 .env
git ls-files | grep -i env
```

如果发现 `.env` 已经被追踪，要把它从版本控制里移除（但保留本地文件）：

```bash
git rm --cached .env
```

然后确认 `.env` 在 `.gitignore` 里，再提交。

小白重点：`git rm --cached` 只是"不再追踪"，本地文件还在。但要注意，如果密钥曾经被提交过，它仍然存在于**历史提交**里。真正的做法是把泄露的密钥"作废、换新"，而不是以为删了就安全。这一点在 1.7 脱敏部分会再强调。

README 是总入口，应该在开头就给出导航：


| 文档 | 回答的问题 |
|---|---|

### 1.6 整理 Demo：让成果"看得见"

代码和文档是"读"的，Demo 是"看"的。招聘方往往先看 Demo 再决定要不要读代码。

Demo 的几种形式：

| 形式 | 适用 | 放哪里 |
|---|---|---|
| GIF | 展示一个核心流程（登录→下单） | 直接嵌进 README |
| 录屏 MP4 | 完整演示 | 传到网盘/视频平台，README 放链接 |
| 截图 | 关键页面/接口返回 | docs/ 目录 + README 引用 |

在 README 里嵌入 GIF：

```markdown
## 效果演示

![核心流程演示](./docs/demo.gif)
```

小白重点：GIF 最友好，因为 README 里直接能看到，不用点链接、不用下载。挑一个"最能代表项目"的核心流程录成 GIF 放在 README 顶部附近。

### 1.7 脱敏检查：交付前最重要的一步

这是今天的重中之重。公开交付意味着全世界都能看到，任何真实敏感信息都可能被利用。

要脱敏的东西：

| 类别 | 具体例子 | 处理 |
|---|---|---|
| 密钥/token | API Key、数据库密码、JWT secret | 换成占位符，走 .env.example |
| 真实域名/IP | 内网地址、生产域名 | 换成 `example.com` / 代号 |
| 公司/品牌名 | 真实公司名、项目代号 | 换成 `graduation-project` 等代号 |
| 本地绝对路径 | `/Users/你的名字/...` | 换成相对路径或占位 |
| 个人信息 | 手机号、邮箱、姓名 | 用占位符 |
| 测试数据 | 真实用户数据 | 换成假数据 |

用命令扫一遍常见泄露点（示例，按需调整关键词）：

```bash
# 搜可能的密钥关键词（大小写不敏感）
grep -ri -E "api_key|secret|password|token" . \
  --include="*.php" --include="*.yml" --include="*.env*" \
  --exclude-dir=vendor --exclude-dir=node_modules
```

逐条确认：

- 命中的是不是真实密钥？→ 换成占位符
- 是不是 `.env.example` 里的占位符？→ 没问题
- 是不是变量名（如 `$apiKey`）？→ 没问题

⚠️ 重要：如果发现真实密钥**曾经被提交过**，删掉当前文件不够，它还在 Git 历史里。正确做法是：

1. 立即到对应平台**作废/轮换**这个密钥（最关键，等于让泄露的那把钥匙失效）
2. 从当前代码移除，改用 `.env`
3. 如需彻底清理历史，再考虑用工具重写历史（有风险，非必要不做）

小白重点：脱敏的核心心法是"假设有人正在盯着你的仓库找密钥"。宁可多检查一遍。密钥一旦公开就当作已泄露，第一反应永远是"换新的"，而不是"删掉就行"。

### 1.8 用"陌生人视角"做最终验收

最后，假装你是第一次见到这个项目的人，从零走一遍：

```text
1. 打开 GitHub 仓库首页 → README 能看懂这是什么吗？
2. 看到 Demo GIF → 知道它能做什么吗？
3. 按 README 的"快速开始" → 能跑起来吗？
4. 按 DEPLOY.md → 能部署吗？
5. 翻代码 → 结构清晰、能看懂分层吗？
6. 全程 → 有没有看到任何真实密钥/内网信息？
```

验收清单：

| 检查项 | 通过标准 |
|---|---|
| README 自解释 | 陌生人 3 分钟内知道项目是什么、怎么跑 |
| 一键启动 | `docker compose up` 能起来 |
| 文档齐全 | 部署、复盘、规划、Demo 都能找到 |
| Git 干净 | 分支清爽、提交信息可读 |
| 全程脱敏 | 无真实密钥/域名/公司名/个人路径 |
| Demo 可见 | 有 GIF 或录屏链接 |

小白重点：能通过"陌生人视角"验收的项目，才是真正的作品集项目。你自己觉得清楚不算数，陌生人能独立跑通、独立看懂才算。

### 8.2 `.gitignore` 加了 `.env`，就代表密钥安全了吗？

参考答案：

> ✅ 不代表。`.gitignore` 只对还没被追踪的文件生效。如果 `.env` 之前已经提交过，它仍在历史里。要用 `git rm --cached` 停止追踪，更重要的是：一旦密钥曾经公开就当作已泄露，第一反应是去平台作废/轮换密钥，而不是以为删了就安全。

### 8.3 好的提交信息应该怎么写？

参考答案：

> ✅ 用规范前缀 + 清楚描述，比如 `feat: 新增订单创建接口`、`fix: 修复未登录可访问订单详情的越权问题`。避免 `update`、`保存一下`、`aaa` 这类无信息量的提交。这和 Node 社区的 Conventional Commits 一致。

---

### 8.4 为什么 Demo 建议优先用 GIF？

参考答案：

> ✅ 因为 GIF 能直接嵌进 README，读者不用点链接、不用下载就能看到项目效果。招聘方往往先看 Demo 再决定读不读代码，把最能代表项目的核心流程录成 GIF 放在 README 显眼处最友好。

---

### 8.5 什么叫"陌生人视角验收"？为什么重要？

参考答案：

> ✅ 假装自己第一次见到这个项目，只看 README 和文档，从零 clone、启动、部署、读代码，看能不能独立跑通、独立看懂、全程没看到真实密钥。重要是因为作品集是给别人看的，自己觉得清楚不算数，陌生人能独立跑通才算真正合格。
