## 1. 学习内容

### 1.1 什么叫“最终验收”？

最终验收不是简单地问：

```text
我是不是学完了？
```

而是要用证据回答：

```text
我能不能独立交付一个 PHP 后端项目？
```

证据包括：

| 验收维度 | 你要证明什么 |
|---|---|
| 能运行 | 项目可以在本地或服务器启动 |
| 能使用 | 核心功能能从前端/API 跑通 |
| 能解释 | 你能讲清楚接口、数据库、业务流程 |
| 能排错 | 出现报错时知道看日志、配置、数据库 |
| 能部署 | 知道上线步骤、环境变量、回滚方法 |
| 能复盘 | 能总结做得好的地方和不足 |

小白重点：毕业不是“没有问题”，而是“知道问题在哪里，并知道下一步怎么改”。

---

### 1.2 第一步：确认项目能从零启动

你要模拟一个新同事拿到项目后的启动过程。

检查顺序：

```text
拉代码
  ↓
安装依赖
  ↓
准备 .env / 配置文件
  ↓
准备数据库
  ↓
执行迁移或导入 SQL
  ↓
启动服务
  ↓
访问健康检查或首页接口
```

如果是 Composer 项目，常见命令类似：

```bash
composer install
```

如果项目有数据库迁移，可能类似：

```bash
php yii migrate
```

或框架自己的迁移命令。

你需要记录：

| 步骤 | 命令/操作 | 是否成功 | 遇到的问题 |
|---|---|---|---|
| 安装依赖 |  |  |  |
| 配置环境 |  |  |  |
| 初始化数据库 |  |  |  |
| 启动服务 |  |  |  |
| 访问接口 |  |  |  |

注意：如果某一步失败，不要只写“失败”，要写清楚错误信息和解决方法。

---

### 1.3 第二步：按用户视角跑核心流程

后端项目最终是给用户或业务使用的，所以你要从用户视角走流程。

以商城毕业项目为例，核心流程可能是：

```text
注册/登录
  ↓
浏览商品
  ↓
加入购物车
  ↓
创建订单
  ↓
支付或模拟支付
  ↓
查看订单详情
  ↓
申请售后或取消订单
```

你可以用浏览器、Postman、curl 或前端页面验证。

记录表：

| 流程 | 接口/页面 | 期望结果 | 实际结果 | 是否通过 |
|---|---|---|---|---|
| 登录 |  | 返回 token/session |  |  |
| 商品列表 |  | 返回商品数据 |  |  |
| 创建订单 |  | 生成订单号 |  |  |
| 支付 |  | 状态变更 |  |  |
| 订单详情 |  | 能看到订单 |  |  |

小白重点：不要只测“正常情况”，至少要测一个错误情况，例如未登录访问、商品不存在、参数为空。

---

### 1.4 第三步：按后端视角检查工程质量

用户流程跑通后，还要检查后端质量。

重点看这些：

| 检查项 | 你要看什么 |
|---|---|
| API 设计 | 路径、方法、参数、响应是否清晰 |
| 数据库 | 表结构是否能支撑业务，字段命名是否清楚 |
| Service 层 | 核心业务是否集中在服务层，而不是 Controller 里乱写 |
| 错误处理 | 参数错误、业务错误是否有明确错误码/提示 |
| 日志 | 关键失败场景是否能查日志 |
| 配置 | 敏感信息是否没有硬编码到代码里 |
| 安全 | 登录鉴权、权限校验、输入校验是否覆盖 |

你可以用这个问题检查自己：

```text
如果明天有一个新需求，我知道应该改哪个 Controller、哪个 Service、哪张表吗？
```

如果答案是“不知道”，说明你还需要补项目结构图。

---

### 1.5 第四步：按部署视角复述上线流程

毕业项目不能只停留在本地。

你要能说清楚：

1. 代码怎么发布
2. 依赖怎么安装
3. 配置怎么注入
4. 数据库怎么迁移
5. 服务怎么启动/重启
6. 日志在哪里看
7. 出问题怎么回滚

可以整理成：

```text
1. 拉取最新代码
2. composer install --no-dev
3. 配置生产环境变量
4. 执行数据库迁移
5. 重启 PHP-FPM / Swoole / 队列 worker
6. 检查健康接口
7. 观察错误日志
8. 如果失败，回滚到上一个版本
```

如果你还没有真实服务器，也可以写“模拟部署流程”，但要明确哪些步骤是真实执行过的，哪些只是计划。

---

### 1.6 第五步：整理项目讲解稿

毕业验收时，你要能像面试/述职一样介绍项目。

建议准备一个 3 分钟讲解稿：

```text
我的毕业项目是一个 ______ 系统。
它解决的问题是 ______。
核心功能包括 ______、______、______。
后端使用 PHP + ______ 框架，数据库使用 ______。
我负责/实现了 ______。
项目中最难的是 ______，我是通过 ______ 解决的。
如果继续优化，我会优先做 ______。
```

再准备一个技术结构说明：

| 模块 | 作用 | 相关接口 | 相关数据表 |
|---|---|---|---|
| 用户模块 | 登录、注册、用户信息 |  |  |
| 商品模块 | 商品列表、详情 |  |  |
| 订单模块 | 下单、支付状态 |  |  |
| 后台模块 | 管理配置和数据 |  |  |

小白重点：能讲清楚项目，比“代码很多”更重要。

---

### 1.7 第六步：整理遗留问题，而不是假装完美

真实项目一定有遗留问题。毕业验收时，你应该主动列出来。

遗留问题可以分级：

| 等级 | 含义 | 示例 |
|---|---|---|
| P0 | 阻塞上线 | 登录失败、订单无法创建 |
| P1 | 影响核心体验 | 支付状态偶发不同步 |
| P2 | 可后续优化 | 页面加载慢、日志不够清晰 |
| P3 | 学习改进项 | 测试覆盖率不足、文档不完整 |

记录模板：

| 问题 | 等级 | 影响 | 当前方案 | 后续计划 |
|---|---|---|---|---|
|  |  |  |  |  |

这不是扣分项，反而说明你具备工程意识。

## 4. JS/Node.js 类比

| PHP 24 周毕业 | Node.js 学习类比 |
|---|---|
| 能跑通 PHP 项目 | 能跑通 Express/NestJS 项目 |
| 能解释 Controller/Service/Model | 能解释 Router/Service/ORM |
| 能写接口和读数据库 | 能写 REST API + Sequelize/Prisma 查询 |
| 能部署 PHP 服务 | 能部署 Node 服务 |
| 能排查日志和配置 | 能看 Node logs/env/process manager |
| 毕业项目 | 一个完整可演示的 full-stack/backend portfolio project |

注意：PHP 与 Node 的运行模型不同。PHP 常见部署可能涉及 Nginx + PHP-FPM，也可能是 Swoole 常驻进程；Node 通常是一个常驻进程。毕业时你要能说清楚自己项目使用的是哪一种。
