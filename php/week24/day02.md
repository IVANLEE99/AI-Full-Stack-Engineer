## 今日目标

学会给毕业项目写一份"别人照着就能跑起来"的 `README.md`，以及一份"上线时照着就能部署"的 `DEPLOY.md`，把昨天的 Docker Compose 编排变成任何人都能一键启动的完整说明。

今天你要真正掌握这一句话：

> 好的 README 是"新同事拿到仓库后不用问你任何问题就能把项目跑起来"的说明书；好的部署文档是"你休假时别人照着也能安全上线和回滚"的操作手册。文档不是给自己看的备忘，而是给"未来完全不了解这个项目的人"看的接力棒。这就像 npm 包的 README 决定了别人愿不愿意用你的包一样。

## 1. 学习内容

### 1.1 为什么"能跑"不等于"能交付"

你昨天把 `docker compose up` 跑通了，项目在你的电脑上活得好好的。但这里有个陷阱：

```text
只有你知道：
- 要先复制 .env.example 成 .env
- 要先 build 前端才能挂载 dist
- 数据库首次启动要等几秒
- 那个端口被占用了要改成别的
```

这些"只在你脑子里"的知识，一旦你离开，项目就变成一个"谁也不敢碰的黑盒"。

交付的本质是把"你脑子里的隐性知识"变成"写在文档里的显性知识"。

用一句话记住：

> 项目能跑，只证明你行；文档完整，才证明这个项目行。

Node 类比：

| 场景 | 没有文档 | 有文档 |
|---|---|---|
| npm 包 | 别人不知道怎么 import、怎么配参数 | README 里有安装、用法、示例 |
| PHP 项目 | 别人不知道怎么装依赖、配环境 | README 里有完整启动步骤 |
| 上线 | 只有作者会部署 | DEPLOY.md 里谁都能照做 |

---

### 1.2 README 和 DEPLOY 的分工

小白容易把所有东西塞进一个文件。正确的做法是分工：

| 文档 | 读者 | 回答的问题 |
|---|---|---|
| `README.md` | 第一次接触项目的人 | 这是什么？怎么在本地跑起来？ |
| `DEPLOY.md` | 负责上线运维的人 | 怎么部署到服务器？出问题怎么办？ |

一句话区分：

> README 面向"本地开发"，DEPLOY 面向"生产上线"。

小白重点：README 是项目的"门面"，打开仓库第一眼看到的就是它。GitHub 会自动把 README 渲染在仓库首页，所以它决定了别人对项目的第一印象。

---

### 1.3 README 应该包含哪些章节

一个专业的 README 通常包含这些部分：

```text
1. 项目名称 + 一句话简介
2. 功能特性（这个项目能做什么）
3. 技术栈（用了什么）
4. 架构 / 服务组成（有哪些服务）
5. 环境要求（需要装什么）
6. 快速开始（最少几步跑起来）
7. 环境变量说明
8. 常用命令
9. 目录结构
10. 常见问题（FAQ）
11. 许可证 / 作者
```

不是每个项目都要全写，但"快速开始"和"环境要求"是绝对不能少的。

对比表：

| 章节 | 必要性 | 作用 |
|---|---|---|
| 一句话简介 | 必须 | 让人 3 秒知道这是什么 |
| 快速开始 | 必须 | 让人最快跑起来 |
| 环境要求 | 必须 | 避免"版本不对"的坑 |
| 环境变量说明 | 强烈建议 | 避免"少配一个变量启动失败" |
| 目录结构 | 建议 | 帮人快速定位代码 |
| FAQ | 建议 | 减少重复提问 |

### 1.4 写一个完整的 README 示例

下面是毕业项目 `graduation-project` 的 README 示例，全部脱敏、用代号：

```markdown
# Graduation Project — 智能文档问答系统

一个基于 PHP 后端 + 向量检索 + 大模型的智能问答系统。
用户上传文档后，可以用自然语言提问，系统会检索相关内容并生成答案。

## 功能特性

- 文档上传与解析
- 文档向量化与相似检索（RAG）
- 基于大模型的问答生成
- 问答历史记录

## 技术栈

- 后端：PHP 8.3 + PHP-FPM
- 数据库：MySQL 8.0
- 向量库：Qdrant
- 大模型接入：LLM Gateway（统一代理）
- 前端：静态 SPA（Nginx 托管）
- 编排：Docker Compose

## 服务组成

| 服务 | 作用 | 对外端口 |
|---|---|---|
| frontend | 前端页面 | 8080 |
| backend | PHP 业务 API | 仅内网 |
| db | MySQL 业务数据 | 仅内网 |
| vector-db | Qdrant 向量检索 | 仅内网 |
| llm-gateway | 大模型代理 | 仅内网 |

## 环境要求

- Docker 24+ 
- Docker Compose v2+
- （可选）Node 20+，用于本地构建前端

## 快速开始

1. 克隆代码

   ```bash
   git clone <仓库地址>
   cd graduation-project
   ```

2. 准备环境变量

   ```bash
   cp .env.example .env
   # 编辑 .env，填入数据库密码和 LLM_API_KEY
   ```

3. 构建前端（如果 dist 未提供）

   ```bash
   cd frontend && npm install && npm run build && cd ..
   ```

4. 一键启动

   ```bash
   docker compose up -d
   ```

5. 访问

   浏览器打开 http://localhost:8080

## 环境变量说明

见 `.env.example`，主要包含：

| 变量 | 说明 | 示例 |
|---|---|---|
| DB_ROOT_PASSWORD | MySQL root 密码 | change-me |
| DB_NAME | 业务数据库名 | graduation |
| DB_USER | 业务库账号 | app_user |
| DB_PASSWORD | 业务库密码 | change-me-too |
| LLM_API_KEY | 大模型 API Key | sk-xxxx |

## 常用命令

| 命令 | 作用 |
|---|---|
| `docker compose up -d` | 后台启动全部服务 |
| `docker compose ps` | 查看服务状态 |
| `docker compose logs -f backend` | 看后端日志 |
| `docker compose down` | 停止全部服务 |

## 目录结构

​```text
graduation-project/
├── frontend/          # 前端代码
├── backend/           # PHP 后端
│   ├── src/           # 业务代码
│   ├── Dockerfile     # 后端镜像构建
│   └── composer.json  # PHP 依赖
├── docker-compose.yml # 编排文件
├── .env.example       # 环境变量模板
├── README.md          # 本文件
└── DEPLOY.md          # 部署文档
​```

## 常见问题

**Q: 启动后访问 8080 报 502？**
A: 后端还没起完，或数据库还在初始化，等 10 秒再刷新，或看 `docker compose logs backend`。

**Q: 提示 LLM_API_KEY 无效？**
A: 检查 `.env` 里是否填了真实 Key，改完要 `docker compose up -d` 重新加载。
```

小白重点：README 里的"快速开始"要按真实步骤写，最好你自己在一台干净的机器（或删掉本地缓存后）照着走一遍，能跑通才算合格。很多 README 就是因为"作者跳过了自己早就装好的那一步"，导致别人跟着做失败。

### 1.5 环境变量文档：`.env.example` 的作用

`.env.example` 是一个"变量清单模板"，它的作用是：

> 告诉别人"这个项目需要哪些环境变量"，但**不泄露真实值**。

真实的 `.env`（含密码、密钥）绝不提交到 Git；`.env.example`（值是占位符）则提交到 Git，作为模板。

```text
.env          → 加进 .gitignore，不提交（真实密码）
.env.example  → 提交到 Git（占位符模板）
```

`.env.example` 示例：

```text
# 数据库配置
DB_ROOT_PASSWORD=change-me
DB_NAME=graduation
DB_USER=app_user
DB_PASSWORD=change-me-too

# 大模型配置
LLM_API_KEY=sk-your-key-here
LLM_PROVIDER=example-provider
```

Node 类比：这和 Node 项目里 `.env` + `.env.example` 的做法完全一样，`dotenv` 读取 `.env`，`.env.example` 只当模板。

### 1.6 DEPLOY.md 应该包含哪些章节

部署文档面向"上线"，比 README 更严谨。它通常包含：

```text
1. 部署环境说明（服务器、Docker 版本）
2. 首次部署步骤
3. 更新发布步骤（已经上过线，发新版本）
4. 环境变量 / 配置注入方式
5. 数据库迁移说明
6. 健康检查方式
7. 日志查看位置
8. 回滚方案（出事了怎么退回去）
9. 故障排查清单
```

其中最容易被小白忽略、但最重要的两块是：**回滚方案**和**故障排查**。

小白重点：能上线只是及格，能"出问题时快速回滚"才是专业。上线前你必须先想清楚"如果这版有问题，怎么退回上一版"。

### 1.7 写一个完整的 DEPLOY.md 示例

```markdown
# 部署文档 — Graduation Project

## 1. 部署环境

- 一台可运行 Docker 的服务器（Linux）
- Docker 24+ / Docker Compose v2+
- 已开放对外端口：8080（前端）

## 2. 首次部署

​```bash
# 1. 拉取代码
git clone <仓库地址>
cd graduation-project

# 2. 准备生产环境变量（用强密码，不要用示例值）
cp .env.example .env
vim .env

# 3. 构建镜像并后台启动
docker compose build
docker compose up -d

# 4. 初始化数据库（首次需要执行迁移）
docker compose exec backend php migrate.php

# 5. 健康检查
curl http://localhost:8080/health
​```

## 3. 更新发布（已上线，发新版本）

​```bash
git pull
docker compose build backend
docker compose up -d backend      # 只重建后端
docker compose exec backend php migrate.php   # 如有新迁移
curl http://localhost:8080/health
​```

## 4. 配置注入

所有敏感配置走 `.env`，不写死在代码里。
修改 `.env` 后需重新 `docker compose up -d` 才生效。

## 5. 数据库迁移

迁移脚本位于 `backend/migrations/`。
执行：`docker compose exec backend php migrate.php`
迁移前建议先备份：`docker compose exec db mysqldump ...`

## 6. 健康检查

访问 `http://<服务器>:8080/health`，正常应返回：

​```json
{"status":"ok"}
​```

## 7. 日志查看

​```bash
docker compose logs -f backend    # 后端日志
docker compose logs -f db         # 数据库日志
​```

## 8. 回滚方案

​```bash
# 1. 切回上一个稳定版本（tag 或 commit）
git checkout <上一个稳定 tag>

# 2. 重新构建并启动
docker compose build backend
docker compose up -d backend

# 3. 如果数据库迁移不兼容，恢复迁移前的备份
​```

⚠️ 回滚前务必确认：新版本是否改了数据库结构。如果改了，回滚代码的同时可能需要恢复数据库备份。

## 9. 故障排查

| 现象 | 可能原因 | 处理 |
|---|---|---|
| 访问 8080 报 502 | 后端未起 / 崩溃 | `docker compose logs backend` |
| 后端连不上数据库 | db 未就绪 / 密码错 | 检查 `.env`、等待 db 初始化 |
| LLM 调用失败 | API Key 无效 / 网络 | 检查 `LLM_API_KEY`、网关日志 |
| 磁盘满 | 日志/数据卷过大 | 清理旧日志、扩容 |
```

对比表：

| 章节 | README 有吗 | DEPLOY 有吗 |
|---|---|---|
| 项目简介 | ✅ | ❌ |
| 本地快速开始 | ✅ | ❌ |
| 生产首次部署 | ❌ | ✅ |
| 更新发布 | ❌ | ✅ |
| 回滚方案 | ❌ | ✅ |
| 故障排查 | 简单 FAQ | 详细清单 |

### 1.8 用"新同事视角"检验文档

写完文档后，最好的检验方法是模拟一个"完全不了解项目的人"：

```text
假装你今天第一天入职，只有这个仓库，没人可问。
你能只靠 README 把项目跑起来吗？
每卡一次，就是文档缺一条。
```

检验清单：

| 检验点 | 通过标准 |
|---|---|
| 环境要求 | 明确写了需要哪些工具和版本 |
| 依赖安装 | 有可复制的命令 |
| 配置准备 | 说清楚要复制哪个文件、填哪些值 |
| 启动命令 | 一条条能直接复制执行 |
| 验证方式 | 说清楚"跑起来后访问哪里看到什么" |
| 出错处理 | 常见错误有对应解法 |

小白重点：文档里的命令要能"直接复制粘贴运行"，不要写"大概执行一下 xxx"。命令要精确到可执行。

## 4. JS/Node.js 类比

| PHP 项目文档 | Node.js 类比 | 说明 |
|---|---|---|
| `README.md` | npm 包 / 项目的 README | 门面，第一印象 |
| 快速开始章节 | `npm install && npm start` 说明 | 最少几步跑起来 |
| `.env.example` | Node 项目的 `.env.example` | 变量模板 |
| `DEPLOY.md` | 部署 runbook / Ops 文档 | 上线操作手册 |
| 回滚方案 | 蓝绿/回退发布说明 | 出事怎么退 |
| 目录结构说明 | `src/` 结构说明 | 帮人定位代码 |
| 常用命令表 | `package.json` 里的 scripts 说明 | 常用操作速查 |

注意：无论 PHP 还是 Node，好文档的标准是一致的——"读者不用问作者就能自己跑起来"。语言不同，文档的工程价值相同。

## 8. 今日自测题

### 8.1 README 和 DEPLOY.md 有什么区别？

参考答案：

> ✅ README 面向"第一次接触项目、要在本地跑起来的人"，回答"这是什么、怎么快速启动"；DEPLOY.md 面向"负责上线运维的人"，回答"怎么部署到生产、出问题怎么回滚"。README 是门面，DEPLOY 是操作手册。

---

### 8.2 为什么要有 `.env.example`，不能直接提交 `.env`？

参考答案：

> ✅ `.env` 里有真实密码和密钥，提交到 Git 会泄露，必须加进 `.gitignore`。`.env.example` 用占位符列出"需要哪些变量"，作为模板提交到 Git，让别人知道要配哪些值，但不泄露真实值。

---

### 8.3 一份好的 README 至少要有哪两个章节？

参考答案：

> ✅ "环境要求"和"快速开始"。前者避免版本不对的坑，后者让人用最少的步骤把项目跑起来。这两个是绝对不能少的。

---

### 8.4 DEPLOY.md 里为什么必须有"回滚方案"？

参考答案：

> ✅ 上线可能出问题。能上线只是及格，能在出问题时快速退回上一个稳定版本才是专业。回滚方案要提前想清楚，尤其要注意"如果新版改了数据库结构，回滚代码时数据库怎么处理"。

---

### 8.5 怎么检验 README 写得够不够好？

参考答案：

> ✅ 用"新同事视角"：假装自己第一天入职、只有这个仓库、没人可问，只靠 README 尝试把项目跑起来。每卡一次，就是文档缺一条。最好在干净环境（或让别人/AI）真的照着跑一遍验证。
