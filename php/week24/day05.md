## 1. 学习内容

### 1.1 "进阶"到底是往哪进？

小白容易把进阶理解成"再学一个新框架"。但真正的后端进阶是沿着几个能力维度往深走：

```text
入门阶段：能写出功能，让它跑起来
       ↓
进阶阶段：让它跑得快、跑得稳、扛得住、改得动
```

具体拆成四个方向：

| 方向 | 核心问题 | 从"能用"到"用好" |
|---|---|---|
| 性能 | 快不快？扛不扛得住量？ | 能优化慢查询、加缓存、压测 |
| 安全 | 会不会被攻破？数据安全吗？ | 能防注入/越权、管好密钥 |
| 架构 | 改得动吗？加需求会不会崩？ | 能分层、解耦、设计可扩展结构 |
| AI 工程化 | AI 能力靠谱吗？可控吗？ | 能把 RAG/大模型做成稳定可观测的服务 |

小白重点：这四个方向不是"选一个",而是后端工程师都要逐步补齐的。你可以按短板决定先补哪个，但长期都要覆盖。

---

### 1.2 方向一：性能

性能进阶的典型主题：

```text
数据库层：慢查询分析、索引优化、N+1 问题、连接池
缓存层：  Redis 缓存、缓存穿透/击穿/雪崩、缓存更新策略
应用层：  异步任务、消息队列、常驻进程（Swoole）
观测：    压测（ab/wrk）、APM、慢日志
```

| 主题 | 你要能回答 | Node 类比 |
|---|---|---|
| 慢查询优化 | 这条 SQL 为什么慢？加什么索引？ | 同样有慢查询和 explain |
| N+1 查询 | 列表页为什么发了 100 条 SQL？ | ORM 里的 eager loading |
| Redis 缓存 | 什么该缓存？缓存怎么失效？ | 都用 Redis，思路一致 |
| 消息队列 | 哪些操作该异步？ | Bull / RabbitMQ 思路一致 |

小白重点：性能优化的第一原则是"先测量再优化"。不要凭感觉觉得哪里慢，要用慢查询日志、压测数据说话。

---

### 1.3 方向二：安全

安全进阶的典型主题：

```text
输入侧：  SQL 注入、XSS、参数校验、文件上传校验
鉴权侧：  越权（水平/垂直）、token 安全、会话管理
密钥侧：  密钥管理、.env 不入库、加密存储
传输侧：  HTTPS、敏感字段脱敏、日志脱敏
```

| 主题 | 你要能回答 | 前面阶段的呼应 |
|---|---|---|
| SQL 注入 | 为什么必须用预处理语句？ | 呼应 PDO 参数绑定 |
| 越权 | 用户 A 能不能查到用户 B 的订单？ | 呼应鉴权中间件 |
| 密钥管理 | API Key 会不会被推到 GitHub？ | 呼应 Day 01 的 .env |
| 日志脱敏 | 日志里会不会打印密码/token？ | 呼应部署文档 |

小白重点：安全不是最后才加的功能，而是每写一个接口就要想"这个接口会被怎么攻击"。进阶阶段要养成"攻击者视角"。

---

### 1.4 方向三：架构

架构进阶的典型主题：

```text
分层：    Controller / Service / Repository 职责边界
解耦：    依赖注入、接口编程、事件机制
可扩展：  模块化、领域划分、避免上帝类
可维护：  统一异常、统一响应、代码规范、静态检查（PHPStan）
测试：    单元测试、集成测试（PHPUnit）
```

| 主题 | 你要能回答 | Node 类比 |
|---|---|---|
| 分层职责 | 业务逻辑该放 Controller 还是 Service？ | 同样有分层争论 |
| 依赖注入 | 怎么让代码好测、好替换？ | NestJS 的 DI |
| 静态检查 | 怎么在跑之前发现类型错误？ | TypeScript / ESLint |
| 单元测试 | 改了代码怎么知道没改坏？ | Jest |

小白重点：架构能力的信号是"改需求时你有把握"。如果加个需求就担心到处崩，说明架构和测试还要补。

---

### 1.5 方向四：AI 工程化

这是本学习路线的特色方向，前面阶段已经接触过。进阶主题：

```text
检索质量：切分策略、embedding 选型、召回评估
稳定性：  超时/重试、降级、限流、成本控制
可观测：  记录 prompt/响应/token、可回放、可审计
可控性：  幻觉控制、输出结构化、安全过滤
工程化：  LLM Gateway 统一代理、多模型切换、缓存
```

| 主题 | 你要能回答 | 呼应 |
|---|---|---|
| RAG 质量 | 检索不准怎么定位是切分还是 embedding？ | 呼应向量库阶段 |
| 成本控制 | 一次请求花多少 token？怎么降？ | 呼应 LLM Gateway |
| 可观测 | 出了坏结果，能不能回放当时的 prompt？ | 呼应日志 |
| 降级 | 大模型挂了，服务还能不能用？ | 呼应稳定性 |

小白重点：AI 工程化的难点不在"调通一次"，而在"稳定、可控、能观测、成本可算"。进阶就是把 Demo 级的 AI 能力做成生产级服务。

### 1.7 把方向拆成"周级任务 + 产出"

规划最容易失败的写法是这样：

```text
第 1-4 周：学性能
第 5-8 周：学安全
```

太粗，没法执行也没法验收。好的写法是每周都有明确任务和可交付产出：

```markdown
### 第 3 周：数据库性能优化
- 任务：给毕业项目找出 3 条慢查询，用 EXPLAIN 分析
- 任务：为其中 2 条加合适索引，对比优化前后耗时
- 产出：一篇《慢查询优化记录》，含优化前后的执行计划和耗时数据
- 验收：能说清楚每个索引为什么加、快了多少
```

对比：

| 粗糙规划 | 可执行规划 |
|---|---|
| 学数据库优化 | 找 3 条慢查询、加索引、写优化记录 |
| 学安全 | 给项目做一次越权自测，修复发现的问题 |
| 学测试 | 给 Service 层写 10 个单元测试，覆盖核心逻辑 |
| 提升架构 | 把一个"上帝 Controller"重构成分层结构 |

小白重点：判断一条规划好不好，就问"这周结束我能拿出什么给别人看？"。拿不出产出的规划，基本等于没规划。

### 1.8 排出 30 周整体节奏

30 周不用平均分配，可以按短板和目标安排权重。一个参考节奏（你按自己情况调）：

```text
第 1-8 周   架构 + 测试   （先让代码"改得动、测得了"）
第 9-16 周  性能 + 数据库 （让代码"跑得快、扛得住"）
第 17-22 周 安全          （让代码"攻不破"）
第 23-30 周 AI 工程化 + 综合项目（把 AI 能力做成生产级 + 一个大项目收尾）
```

30 周进阶路线表模板：

```markdown
## 30 周进阶路线

| 周次 | 方向 | 本周任务 | 产出 | 验收标准 |
|---|---|---|---|---|
| 1 | 架构 | | | |
| 2 | 架构 | | | |
| 3 | 架构+测试 | | | |
| ... | | | | |
| 30 | 综合 | 完成一个可上线项目 | 上线的项目 + 复盘 | 能独立部署运维 |
```

小白重点：30 周是弹性的。宁可任务排少一点、每周都完成，也不要排得满满当当然后第 3 周就放弃。

### 1.9 给规划加"防半途而废"机制

规划最大的敌人是坚持不下去。加几个机制：

| 机制 | 做法 |
|---|---|
| 周复盘 | 每周花 15 分钟：完成了吗？没完成为什么？ |
| 可见进度 | 用一张表打勾，能看到累计进度 |
| 小步快跑 | 每周任务量控制在"努努力能完成" |
| 产出驱动 | 每周有交付物，避免"学了但没痕迹" |
| 弹性缓冲 | 每 4 周留 1 周做机动/补课 |

小白重点：把"学习"变成"每周交付一个小产出"，是最有效的坚持机制。产出会累积成作品集，而作品集就是你下一份工作的敲门砖。

## 4. JS/Node.js 类比

| PHP 进阶规划 | Node.js / 工程实践类比 | 说明 |
|---|---|---|
| 四大进阶方向 | 后端通用能力维度 | 语言无关，思路一致 |
| 性能优化 | Node 也做慢查询/缓存/压测 | 工具不同，方法相同 |
| 安全 | XSS/注入/越权 | Web 安全通用 |
| 架构分层 | NestJS 分层 + DI | 结构思想一致 |
| 静态检查 | PHPStan ↔ TypeScript/ESLint | 上线前发现问题 |
| 单元测试 | PHPUnit ↔ Jest | 保障可维护性 |
| 30 周路线表 | 项目排期 / roadmap | 拆任务 + 定产出 |
| 周产出驱动 | sprint 交付 | 小步快跑 |

注意：进阶阶段"语言"越来越不重要，"工程能力"越来越重要。PHP 和 Node 在性能、安全、架构上的方法论高度一致，你在一边学到的能迁移到另一边。这正是你 24 周坚持类比学习的长期回报。

## 8. 今日自测题

### 8.1 后端进阶的四个方向是什么？各解决什么问题？

参考答案：

> ✅ 性能（快不快、扛不扛得住量）、安全（会不会被攻破）、架构（改得动吗、加需求会不会崩）、AI 工程化（AI 能力是否稳定可控可观测）。它们不是四选一，而是后端工程师都要逐步补齐，只是按短板决定先后。

---

### 8.2 一条"好的进阶周计划"应该长什么样？

参考答案：

> ✅ 有明确任务、可交付产出、可验收标准。反例是"第 1-4 周学性能"这种太粗的写法。正例是"本周找 3 条慢查询、加 2 个索引、写一篇含执行计划和耗时对比的优化记录"。判断标准：这周结束能不能拿出东西给别人看。

---

### 8.3 进阶规划的优先级应该由什么决定？

参考答案：

> ✅ 由"Day 04 复盘出的短板" + "目标岗位需要什么"共同决定，而不是"我想学什么就排什么"。这样复盘才能真正指导规划，短板才能被优先补齐。

---

### 8.4 性能优化的第一原则是什么？

参考答案：

> ✅ 先测量再优化。不要凭感觉猜哪里慢，要用慢查询日志、压测数据说话。定位到真正的瓶颈再动手，避免优化了不重要的地方。

---

### 8.5 为什么说进阶阶段"语言不重要，工程能力重要"？

参考答案：

> ✅ 性能、安全、架构、测试这些方法论是语言无关的，PHP 和 Node 高度一致。进阶阶段拼的是"能不能设计出可维护、跑得快、攻不破的系统"，这种工程能力可以在语言之间迁移，也正是坚持类比学习的长期回报。
