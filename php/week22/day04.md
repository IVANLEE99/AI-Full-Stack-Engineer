## 今日目标

给运营知识助手接入 RAG（检索增强生成），让它能回答"政策类"问题（退货政策、售后规则），并且**每条回答都附带引用来源**。

今天你要真正掌握这一句话：

> RAG = 先"检索"再"生成"：用户提问时，先从知识库里搜出最相关的政策片段，把这些片段连同问题一起发给大模型，让它"照着资料回答"并标出引用——这样答案有依据、可溯源，而不是模型凭空编。

## 1. 学习内容

### 1.1 RAG 要解决什么问题

大模型不知道"你公司的退货政策是几天"。你有两个办法：

- 办法 A：把政策塞进提示词。但政策文档几万字，全塞进去又贵又超长度。
- 办法 B（RAG）：只把"和当前问题最相关的几段"检索出来塞进去。

```text
用户：过保了还能退货吗？
   │
检索：从政策库里搜出最相关的 2-3 段（如"退货时效""过保处理"）
   │
拼提示词：这些政策片段 + 用户问题 → 发给大模型
   │
大模型：照着片段回答 +「引用了 [1][2]」
```

小白重点：RAG 的核心是"**别让模型背整本书，只给它当前需要的那几页**"。

---

### 1.2 RAG 的四步

```text
1. 切片（Chunk）：把政策文档切成小段（每段几百字）
2. 向量化（Embed）：把每段转成一串数字（向量），存进向量库
3. 检索（Retrieve）：用户提问也转成向量，找最相似的几段
4. 生成（Generate）：把检索到的段落 + 问题发给大模型，生成回答
```

小白重点：W17-W18 已经建过知识库（向量化+存储），本周是"接入"——直接用现成的检索能力，不用重造轮子。

对比一个熟悉的场景：

| RAG 步骤 | 类比 |
|---|---|
| 切片 | 把书拆成一页页 |
| 向量化 | 给每页贴"语义标签" |
| 检索 | 按问题找最相关的几页 |
| 生成 | 让人照着这几页回答 |

---

### 1.3 "引用来源"是怎么来的

引用不是大模型编的，是我们"检索到哪几段，就把哪几段作为来源"：

```php
<?php
// 检索返回的每一段都带来源信息
$retrieved = [
    [
        "id" => "policy-001",
        "title" => "退货政策 - 时效",
        "content" => "商品签收后 7 天内可无理由退货……",
        "score" => 0.92, // 相似度
    ],
    [
        "id" => "policy-003",
        "title" => "售后 - 过保处理",
        "content" => "超过保修期的商品，可付费维修……",
        "score" => 0.85,
    ],
];
```

这两段就是回答的"引用来源 `[1] [2]`"。前端把它们展示成可点击的来源卡片。

小白重点：**引用来源 = 检索命中的片段**。这让答案"可溯源"，用户能核对，是政策类问答的关键。回顾 Day 01/02 里 `sources` 字段，就是装这个的。

### 1.4 用 PHP 实现最小检索（关键词版）

真实项目用向量检索，但为了先跑通、看懂原理，先写个关键词版：

```php
<?php
declare(strict_types=1);

class SimpleRetriever
{
    /** @var array<array{id:string,title:string,content:string}> */
    private array $docs;

    public function __construct()
    {
        // 真实项目：从向量库检索。这里用内置政策库演示
        $this->docs = [
            ["id" => "policy-001", "title" => "退货政策-时效",
             "content" => "商品签收后 7 天内可无理由退货，需保持包装完好。"],
            ["id" => "policy-002", "title" => "退货政策-运费",
             "content" => "无理由退货运费由买家承担，质量问题运费由卖家承担。"],
            ["id" => "policy-003", "title" => "售后-过保处理",
             "content" => "超过保修期的商品可申请付费维修，不支持退货。"],
        ];
    }

    /** 返回最相关的 topK 段 */
    public function retrieve(string $query, int $topK = 2): array
    {
        $scored = [];
        foreach ($this->docs as $doc) {
            $score = $this->keywordScore($query, $doc["content"] . $doc["title"]);
            if ($score > 0) {
                $scored[] = $doc + ["score" => $score];
            }
        }
        // 按相关度降序，取前 topK
        usort($scored, fn($a, $b) => $b["score"] <=> $a["score"]);
        return array_slice($scored, 0, $topK);
    }

    private function keywordScore(string $query, string $text): int
    {
        $score = 0;
        foreach (["退货", "运费", "保修", "过保", "维修", "时效"] as $kw) {
            if (str_contains($query, $kw) && str_contains($text, $kw)) {
                $score++;
            }
        }
        return $score;
    }
}
```

小白重点：关键词版只是脚手架，理解"检索返回带来源的片段"就行。真实项目把 `retrieve` 换成"调向量库"即可，接口不变。

### 1.5 把检索片段拼进提示词

检索到片段后，拼成一段"参考资料"提示词发给大模型：

```php
<?php
declare(strict_types=1);

function buildRagPrompt(string $userQuestion, array $retrieved): string
{
    $context = "";
    foreach ($retrieved as $i => $doc) {
        $n = $i + 1;
        $context .= "[{$n}] {$doc['title']}：{$doc['content']}\n";
    }

    return <<<PROMPT
你是运营知识助手。请**仅根据以下参考资料**回答用户问题。
如果资料里没有答案，就说"暂无相关政策信息"，不要编造。
回答时在引用处标注编号，例如 [1] [2]。

参考资料：
{$context}

用户问题：{$userQuestion}
PROMPT;
}
```

小白重点：提示词里有三个关键约束：
1. **仅根据参考资料**回答（防瞎编）
2. 没有就说"暂无"（防幻觉）
3. 标注引用编号（可溯源）

`<<<PROMPT ... PROMPT` 是 PHP 的 heredoc 语法，写多行字符串很方便，类似 JS 的模板字符串反引号。

### 1.6 完整 RAG 问答流程

把检索、拼提示词、调模型串起来：

```php
<?php
declare(strict_types=1);

function ragAnswer(string $userQuestion): array
{
    $retriever = new SimpleRetriever();
    $retrieved = $retriever->retrieve($userQuestion, topK: 2);

    // 没检索到：直接返回，不调模型瞎编
    if (empty($retrieved)) {
        return [
            "answer" => "暂无相关政策信息，建议咨询人工客服。",
            "sources" => [],
        ];
    }

    $prompt = buildRagPrompt($userQuestion, $retrieved);
    $llmResp = callLlm(["messages" => [["role" => "user", "content" => $prompt]]]);

    // 把检索片段作为引用来源返回给前端
    $sources = array_map(fn($d) => [
        "id" => $d["id"],
        "title" => $d["title"],
        "snippet" => $d["content"],
    ], $retrieved);

    return [
        "answer" => $llmResp["content"],
        "sources" => $sources,
    ];
}
```

小白重点：返回结构 `{answer, sources}` 和 Day 01/02 完全对齐——前端不用改，直接就能展示来源卡片。这就是"契约统一"的好处。

---

### 1.7 前端展示引用来源

回顾 Day 01 的 sources 展示，RAG 回答的来源就长这样：

```vue
<div v-if="msg.sources?.length" class="sources">
  <div class="sources-title">📚 引用来源</div>
  <a
    v-for="(s, i) in msg.sources"
    :key="s.id"
    class="source-card"
    :title="s.snippet"
  >
    <span class="source-num">[{{ i + 1 }}]</span>
    {{ s.title }}
  </a>
</div>
```

配合 CSS，鼠标悬停能看到片段原文（`title` 属性），点击可跳转到政策详情页。

小白重点：**引用来源让用户信任答案**。政策问答如果没有来源，用户不敢照着做。这是产品可用性的关键。

---

### 1.8 RAG vs Tool（Day 03）的区别

学员常混淆 RAG 和 Tool，对比一下：

| 对比项 | RAG（今天） | Tool（Day 03） |
|---|---|---|
| 解决什么 | "非结构化知识"问答（政策/文档） | "结构化实时数据"查询（订单/库存） |
| 怎么工作 | 检索文档片段 → 拼提示词 → 生成 | 大模型调函数 → 查库 → 喂回 |
| 输出重点 | 带引用来源的回答 | 带真实数据的回答 |
| 谁触发 | 我们检索后主动拼进提示词 | 大模型决定调不调 |
| 典型问题 | "退货政策是几天？" | "我订单 SN-8842 到哪了？" |

小白重点：一个助手往往**两者都要**——问政策走 RAG，查订单走 Tool。Day 05 联调时会把它们放到同一个对话里。

## 4. JS/Node.js 类比

| RAG 概念 | Node 类比 | 说明 |
|---|---|---|
| 检索 retrieve | 全文搜索 / Elasticsearch 查询 | 找最相关的文档 |
| 向量化 embed | 把文本转成 number[] | 语义相似度基础 |
| topK | `LIMIT k` | 只取最相关的几条 |
| 拼提示词 | 模板字符串拼接 | 把资料塞进 prompt |
| heredoc `<<<PROMPT` | 反引号模板字符串 | 多行字符串 |
| sources 引用 | 搜索结果高亮/链接 | 让答案可溯源 |
| 防幻觉约束 | 输入校验/兜底逻辑 | 没数据就不乱答 |

一句话类比：

> RAG 就像"先用搜索引擎查资料，再让人照着资料写答案并附参考文献"——检索保证有依据，引用保证可溯源。

## 8. 今日自测题

### 8.1 RAG 是什么？

参考答案：

> ✅ RAG（检索增强生成）= 先检索再生成。用户提问时先从知识库搜出最相关的片段，把片段和问题一起发给大模型，让它照着资料回答并标引用，保证答案有依据、可溯源。

---

### 8.2 RAG 的四步是什么？

参考答案：

> ✅ 切片（把文档切成小段）→ 向量化（转成向量存库）→ 检索（按问题找最相似片段）→ 生成（把片段+问题给模型生成回答）。

---

### 8.3 "引用来源"从哪来？

参考答案：

> ✅ 引用来源就是检索命中的那几个片段。我们把检索结果作为 `sources` 返回给前端展示，让用户能核对答案依据。引用不是模型编的。

---

### 8.4 怎么防止模型瞎编？

参考答案：

> ✅ 三招：提示词里要求"仅根据参考资料回答"；资料里没有就说"暂无相关信息"；检索为空时直接返回兜底提示，不调模型。

---

### 8.5 RAG 和 Tool 有什么区别？

参考答案：

> ✅ RAG 处理非结构化知识（政策/文档），靠检索片段拼提示词；Tool 处理结构化实时数据（订单/库存），靠大模型调函数查库。前者答政策问题，后者查实时数据。
