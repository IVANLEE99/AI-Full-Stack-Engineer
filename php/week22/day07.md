### 1.5 预习：什么是 Agent 平台化

现在你的助手是"一个写死的应用"。平台化的目标是让它变成**可配置、可复用、可上架**的能力单元。

对比：

| 维度 | 现在（MVP） | 平台化后 |
|---|---|---|
| 工具 | 代码里硬编码订单/商品查询 | 可注册的 Tool 清单，动态挂载 |
| 知识 | 固定几篇政策文件 | 可配置的知识源，按需挂载 |
| 助手 | 一个 Vue+PHP 应用 | 一个 Skill/Agent 定义，平台加载调度 |
| 复用 | 只能自己用 | 别人配置后即可用，能上架 |

小白重点：平台化的核心思想是**把"变化的部分"抽象成配置**（工具、知识、提示词、模型），把"不变的部分"沉淀成引擎。

---

### 1.6 预习：Skill 格式长什么样

平台化常见做法是把一个能力定义成 Skill（技能）。它通常是一份声明式配置，描述"这个技能是什么、需要什么工具、什么时候触发"：

```yaml
# 示例：一个 Skill 定义（预习用，脱敏）
name: ops-knowledge-assistant
description: 运营知识助手，回答政策问题并查询订单商品
triggers:
  - 退货政策
  - 查订单
  - 发货
tools:
  - order_query      # 订单查询 Tool
  - product_query    # 商品查询 Tool
knowledge:
  - policy_docs      # 政策知识源（RAG）
prompt: |
  你是运营知识助手。回答政策问题必须引用来源；
  查询类问题调用对应 Tool；答案未知时诚实说明。
```

对照你这周做的东西：`tools` 就是 Day 03 的 MCP、`knowledge` 就是 Day 04 的 RAG、`prompt` 就是你在 PHP 层组装的系统提示。平台化只是把它们从代码里搬进配置里。

小白重点：先不用懂全部字段，理解"一个 Skill = 触发词 + 工具 + 知识 + 提示词"这个结构即可。

---

### 1.7 预习：从 MVP 到平台要抽象什么

下一阶段你会逐步把这些"写死的"变成"可配置的"：

| 写死的 | 抽象成 |
|---|---|
| PHP 里 if 判断要不要查订单 | 工具路由 / 意图识别可配置 |
| 固定的系统提示词 | Skill 里的 prompt 字段 |
| 只有订单/商品两个工具 | 可注册的 Tool 注册表 |
| 只有政策一个知识库 | 可挂载的多知识源 |
| 一个前端页面 | 平台统一对话入口，多 Skill 复用 |

小白重点：不用现在就实现，今天只要能说出"哪些地方该抽象"，就为下阶段打好了预习基础。

## 4. JS/Node.js 类比

| 平台化概念 | Node/前端类比 | 说明 |
|---|---|---|
| Skill 定义 | package.json / 插件清单 | 声明式描述一个能力 |
| Tool 注册表 | npm 依赖注册 / 插件系统 | 可动态挂载的能力 |
| 知识源挂载 | 数据源配置 | 按需接入不同数据 |
| 平台加载 Skill | 框架加载插件 | 引擎不变，插件可换 |
| 从 MVP 到平台 | 从脚本到框架 | 抽象出可复用引擎 |

一句话类比：

> 从 MVP 到平台，就像从"写一个 Node 脚本"到"写一个能加载插件的框架"——脚本只解决一个问题，框架让别人也能用你的能力解决他们的问题。
