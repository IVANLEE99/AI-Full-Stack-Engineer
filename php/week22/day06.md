## 1. 学习内容

### 1.1 什么叫 MVP 达标

MVP = Minimum Viable Product（最小可行产品）。达标不看功能数量，看**闭环是否完整、价值是否讲清**。

我们这个"运营知识助手"的 MVP 达标线：

| 维度 | 达标标准 |
|---|---|
| 政策问答 | 能回答退货/发货等政策，**带引用来源** |
| 订单查询 | 能查到指定订单的真实状态 |
| 多轮追问 | 记得上一轮上下文，能追问 |
| 流式体验 | 回答逐字出现，不是干等一大坨 |
| 异常处理 | 查不到/出错时有友好提示，不白屏 |
| 可演示 | 有一段能完整跑通的脚本 |

小白重点：达标 ≠ 完美。MVP 允许"只支持几个订单号""知识库只有几篇政策"，但这条闭环必须稳。

对比 Node 项目里"能 demo 的最小版本"思路一致：先跑通主流程，边角后置。

### 1.2 设计黄金演示脚本

演示不能即兴发挥，要有固定脚本，确保每次都跑通：

```text
【第 1 轮 · 问政策】
用户：你们的退货政策是几天？
助手：（流式）根据平台政策，普通商品支持 7 天无理由退货……
     📎 来源：《退换货政策》第 2 条

【第 2 轮 · 查订单（Tool）】
用户：帮我查下订单 SN-8842 现在到哪了
助手：（流式）订单 SN-8842 当前状态为"运输中"，预计明天送达。

【第 3 轮 · 追问（多轮上下文）】
用户：那这个订单还能退吗？
助手：（结合上文订单 + 政策）该订单目前处于运输中，签收后 7 天内可申请
     无理由退货。📎 来源：《退换货政策》第 2 条
```

小白重点：这段脚本同时展示了**三个卖点**——RAG 引用、MCP 工具、多轮上下文。第 3 轮最亮眼，因为它把政策和订单串起来了。

---

### 1.3 打磨黄金链路

对照脚本，逐轮打磨体验。前端对话组件的核心状态：

```js
// 前端：一条消息的数据结构（对齐前五天的契约）
const message = {
  role: "assistant",       // user / assistant
  content: "",             // 流式累加的正文
  sources: [],             // 引用来源 [{ title, snippet }]
  status: "streaming",     // pending / streaming / done / error
};
```

发送逻辑（复用 Day 05 的 streamChat）：

```js
async function send(text) {
  messages.value.push({ role: "user", content: text });
  const reply = reactive({ role: "assistant", content: "", sources: [], status: "streaming" });
  messages.value.push(reply);

  try {
    await streamChat(text, (chunk) => {
      if (chunk.delta) reply.content += chunk.delta;      // 逐字
      if (chunk.sources) reply.sources = chunk.sources;   // 引用
    });
    reply.status = "done";
  } catch (e) {
    reply.status = "error";
    reply.content = "抱歉，我暂时无法回答，请稍后再试。"; // 友好兜底
  }
}
```

小白重点：`status` 字段驱动 UI（转圈/正文/来源卡/错误提示）。这是让 demo "看起来专业"的关键。

---

### 1.4 补齐边界与异常

演示最怕当场翻车。提前处理这几类：

| 场景 | 处理 |
|---|---|
| 订单号查不到 | "没有找到该订单，请确认订单号" |
| 政策库没相关内容 | "这个问题我暂时没有依据，建议咨询人工客服" |
| LLM/网络超时 | "响应超时，请重试"，不要白屏 |
| 空输入 | 发送按钮禁用 |
| 连续快速发送 | streaming 期间禁用输入 |

PHP 侧兜底示例：

```php
<?php
try {
    $result = $mcp->call("order_query", ["order_sn" => $sn]);
    if (empty($result["data"])) {
        $toolResult = "未找到订单 {$sn}，可能订单号有误。";
    } else {
        $toolResult = json_encode($result["data"], JSON_UNESCAPED_UNICODE);
    }
} catch (\Throwable $e) {
    error_log("MCP 查询失败: " . $e->getMessage());
    $toolResult = "订单系统暂时不可用，请稍后再试。"; // fallback，不抛给前端 500
}
```

小白重点：**Tool 失败要 fallback，不能让整个对话崩掉**。这正是本周 README 强调的"Tool 失败 fallback"能力对齐点。

### 1.5 加"完成度高"的细节

这些小细节让 MVP 从"能跑"变成"像个产品"：

- 加载态：助手消息 pending 时显示"正在思考…"三个点动画
- 来源卡片：引用来源用卡片展示，可点开看片段
- 逐字光标：流式时正文末尾加一个闪烁光标
- 空状态：首次进入显示几个示例问题按钮
- 自动滚动：新消息出现时滚到底部

来源卡片模板：

```vue
<div v-if="msg.sources?.length" class="sources">
  <div class="sources-title">📎 引用来源</div>
  <div v-for="(s, i) in msg.sources" :key="i" class="source-card">
    <span class="source-title">{{ s.title }}</span>
    <p class="source-snippet">{{ s.snippet }}</p>
  </div>
</div>
```

小白重点：不要在 demo 前一晚才加这些。功能通了就该顺手补，越到后面越没时间。

### 1.6 走查 + 掐时间

录制前完整走一遍脚本，掐时间（目标 2-3 分钟）：

```text
□ 打开页面，展示空状态和示例问题
□ 第 1 轮：问政策 → 看逐字 → 看来源卡
□ 第 2 轮：查订单 → 看查到真实状态
□ 第 3 轮：追问 → 看上下文串联
□ 演示一个异常：查一个不存在的订单 → 看友好提示
□ 全程无报错、无白屏、无卡死
```

小白重点：走查时如果哪一步磕绊，现在修，别指望录的时候运气好。

### 1.7 录制 Demo

录制要点：

- 屏幕录制（保证清晰），可加简短旁白说明每一步在演示什么
- 按脚本走，不即兴
- 出错就重录，不将就
- 录完自己看一遍，确认三个卖点都清晰展示了

小白重点：Demo 是你毕业项目的"门面"。招聘方/评审往往先看 demo 再看代码。一段流畅的 2 分钟 demo 胜过千行 README。

## 4. JS/Node.js 类比

| MVP 概念 | Node/前端类比 | 说明 |
|---|---|---|
| MVP | 最小可交付版本 | 主流程通，边角后置 |
| 演示脚本 | E2E 测试用例 | 固定输入固定路径 |
| status 字段 | 组件状态机 | 驱动 UI 显示 |
| Tool fallback | try/catch + 兜底返回 | 失败不崩 |
| 来源卡片 | 列表渲染组件 | v-for 展示 |

一句话类比：

> MVP 就像餐厅试营业——菜单只上几道招牌菜，但每道都要能稳定出品、摆盘好看，让客人吃到就懂你的价值，而不是把全部菜品都上齐却道道半生不熟。

## 8. 今日自测题

### 8.1 MVP 达标看什么？

参考答案：

> ✅ 看闭环是否完整、价值是否讲清，不看功能数量。允许订单/政策库很小，但"问政策带引用→查订单→追问"这条闭环必须稳。

---

### 8.2 为什么要写死演示脚本？

参考答案：

> ✅ 演示即兴发挥容易翻车。固定脚本能保证每次跑通，并且能刻意安排展示三个卖点（RAG 引用、MCP 工具、多轮上下文），像 E2E 测试用例一样确定。

---

### 8.3 Tool 查询失败该怎么处理？

参考答案：

> ✅ try/catch 捕获，返回友好兜底文案（如"订单系统暂时不可用"），绝不把 500 抛给前端导致对话崩掉。这是本周强调的"Tool 失败 fallback"能力。

---

### 8.4 演示脚本第 3 轮追问为什么最重要？

参考答案：

> ✅ 因为它把政策（RAG）和订单（Tool）串起来，还用到多轮上下文，一次展示了产品的综合能力，是最能打动评审的一轮。

---

### 8.5 哪些细节能让 MVP "看起来像产品"？

参考答案：

> ✅ 加载态、逐字流式光标、引用来源卡片、示例问题空状态、新消息自动滚动到底部。这些不改变核心功能，但显著提升完成度观感。
