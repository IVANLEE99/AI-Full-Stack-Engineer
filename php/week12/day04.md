## 今日目标

系统理解跨服务调用中的稳定性和安全性设计：服务间鉴权、超时、重试、熔断、幂等和错误降级。能判断哪些接口可以重试，哪些接口必须谨慎；能说明为什么支付、退款、售后这类高风险链路不能只靠“HTTP 调通”。

> 跨服务调用不是把接口调通就结束了；真正可靠的 InternalService 必须同时考虑“谁能调用、等多久、失败怎么办、能不能重试、会不会重复执行、如何保护下游”。

## 1. 学习内容

### 1.1 为什么服务间调用也需要“安全 + 稳定性”？

小白容易把跨服务调用理解成：

```text
A 服务发 HTTP 请求给 B 服务，B 返回 JSON。
```

这只是最表层。真实项目要问更多问题：

| 问题 | 为什么重要 |
|---|---|
| 谁能调用这个接口？ | 防止未授权服务调用支付/退款 |
| 请求等多久？ | 防止下游慢拖垮上游 |
| 失败是否重试？ | 防止网络抖动导致业务失败 |
| 重试会不会重复扣款？ | 防止资金事故 |
| 下游一直挂怎么办？ | 防止雪崩 |
| 错误怎么给用户？ | 防止暴露内部细节 |

---

### 1.2 服务间鉴权：确认“调用者是谁”

内网调用也要鉴权，尤其是支付和售后。

常见方案：

| 方案 | 适合场景 | 小白理解 |
|---|---|---|
| 固定 internal token | 简单内部系统 | 服务间约定暗号 |
| HMAC 签名 | 重要业务接口 | 用密钥证明请求没被改 |
| mTLS | 高安全基础设施 | 双方都有证书 |
| IP 白名单 | 辅助限制 | 只允许特定来源机器 |
| timestamp + nonce | 防重放 | 旧请求不能反复用 |

HMAC 简化思路：

```text
调用方：用 secret 对 body + timestamp 做签名
被调用方：用同一个 secret 重新算签名
两边结果一致：说明请求可信
```

伪代码：

```php
<?php

$signature = hash_hmac('sha256', json_encode($payload) . $timestamp, $secret);
```

注意：签名不是为了隐藏数据，而是为了验证“数据没被篡改 + 调用方知道密钥”。

---

### 1.3 timeout：每个跨服务请求都应该有等待上限

错误示例：

```php
<?php

HttpClient::post($url, $params); // 没有 timeout，可能一直卡
```

更好的写法：

```php
<?php

HttpClient::post($url, $params, [
    'timeout' => 2,
]);
```

不同场景 timeout 可以不同：

| 场景 | 建议思路 |
|---|---|
| 查询配置 | 短超时，失败可用默认值 |
| 查询支付状态 | 短超时，可提示稍后刷新 |
| 创建支付 | 谨慎设置，避免用户长时间等待 |
| 后台批处理 | 可以稍长，但要有上限 |

小白重点：

```text
没有 timeout 的跨服务调用，是线上事故的常见来源。
```

---

### 1.4 retry：不是所有失败都能重试

重试能解决临时网络问题，但也可能制造重复执行。

可以重试的常见条件：

1. 请求是查询类，没有副作用。
2. 请求有幂等键，重复调用不会重复执行。
3. 只对网络超时、连接失败等临时错误重试。
4. 重试次数很少，比如 1-2 次。
5. 有退避等待，比如 100ms、300ms。

不建议盲目重试：

| 场景 | 原因 |
|---|---|
| 扣款 | 可能重复扣款 |
| 退款 | 可能重复退款 |
| 创建订单 | 可能重复创建 |
| 发券 | 可能重复发放权益 |

如果必须重试高风险操作，前提是有稳定幂等键：

```text
idempotency_key = order_no + action + version
```

---

### 1.5 幂等：重试前必须理解的概念

幂等是指：同一个请求执行一次和执行多次，最终结果一致。

例如：

```text
查询订单状态：执行多次都只是查询 → 通常幂等
支付扣款：执行多次可能扣多次 → 默认不幂等
带 idempotency_key 的扣款：同一个 key 只扣一次 → 可设计成幂等
```

支付请求中常见字段：

```php
<?php

[
    'order_no' => 'O1001',
    'amount' => 10000,
    'idempotency_key' => 'pay_capture_O1001_v1',
]
```

下游支付服务要保证：同一个 `idempotency_key` 只处理一次。

---

### 1.6 熔断：下游故障时保护上游

如果支付服务已经大量超时，订单服务继续疯狂请求，只会让情况更糟。

熔断流程：

```text
连续失败达到阈值
  ↓
打开熔断：暂时不再请求支付服务
  ↓
快速返回“支付服务繁忙”
  ↓
等待一段时间后半开试探
  ↓
恢复成功则关闭熔断
```

小白类比：

```text
电路过载时保险丝断开，避免烧坏整个系统。
```

---

### 1.7 错误降级：失败时给用户什么？

不同场景的失败提示不同。

| 场景 | 下游失败时建议 |
|---|---|
| 支付状态查询失败 | “暂时无法获取支付状态，请稍后刷新” |
| capture 超时 | “支付处理中，请稍后查看订单状态” |
| 退款申请失败 | “退款申请提交失败，请稍后重试或联系客服” |
| 售后服务不可用 | “售后服务暂时不可用，请稍后再试” |

不要直接返回：

```text
curl error 28
SQLSTATE...
provider timeout stack trace
```

这些是内部细节，应该进日志，不应该给用户。

---

### 1.8 策略表：不同接口怎么配置？

| 接口类型 | timeout | retry | 是否需要幂等 | 熔断 | 错误提示 |
|---|---:|---|---|---|---|
| 查询支付状态 | 短 | 可少量重试 | 不强制 | 可 | 稍后刷新 |
| capture 扣款 | 中短 | 谨慎，必须幂等 | 必须 | 可 | 支付处理中 |
| 发起退款 | 中 | 谨慎，必须幂等 | 必须 | 可 | 退款处理中/失败 |
| 获取配置 | 短 | 可 | 不强制 | 可 | 使用默认值 |
| 创建售后单 | 中 | 谨慎 | 建议 | 可 | 提交失败或处理中 |

### 1.9 Laravel / Node.js 类比

Laravel HTTP Client：

```php
<?php

Http::timeout(2)
    ->retry(2, 100)
    ->withHeaders($headers)
    ->post($url, $payload);
```

Node axios：

```js
await axios.post(url, payload, {
  timeout: 2000,
  headers,
});
```

但要记住：语法不是重点，策略才是重点。

## 8. 今日自测题

### 8.1 内网服务之间调用为什么也要鉴权？

参考答案：

> ✅ 因为内网接口不是“裸奔接口”。内网可能被误调用，某个服务被攻破后可能横向调用支付、退款等高风险接口，而且被调用方需要知道“是谁调用了我”。所以支付、售后这类接口即使在内网也要用 token 或签名鉴权。

---

### 8.2 HMAC 签名鉴权的核心思路是什么？

参考答案：

> ✅ 调用方用双方约定的 secret 对 body + timestamp 做签名，被调用方用同一个 secret 重新算一遍，两边结果一致就说明请求可信。签名不是为了隐藏数据，而是证明“数据没被篡改 + 调用方知道密钥”。

---

### 8.3 timeout、retry、circuit breaker 分别解决什么问题？

参考答案：

> ✅ timeout 解决“下游太慢不能无限等”，超过时间就放弃；retry 解决“网络抖动等临时失败”，但必须结合幂等；circuit breaker 熔断解决“下游持续故障”，像电闸跳闸一样先断开一段时间，避免请求继续压垮下游、引发雪崩。

---

### 8.4 为什么 retry 一定要结合幂等？支付扣款能随便重试吗？

参考答案：

> ✅ 因为重试意味着同一个请求可能被执行多次。如果扣款、退款这类接口没有幂等设计，重试就可能导致重复扣款或重复退款，造成资金事故。所以扣款不能盲目重试，只有幂等设计完善（带 idempotency_key）或查询类接口才适合自动重试。

---

### 8.5 什么是错误降级？下游支付服务挂了应该怎么给用户提示？

参考答案：

> ✅ 错误降级是指下游失败时给用户一个可理解、可接受的结果，而不是把 curl 错误或渠道错误码原样抛出去。比如支付服务超时，应提示“支付处理中，请稍后查看订单状态”，同时在日志里保留下游原始错误和 trace_id 用于排障。
