## 1. 学习内容

### 1.1 为什么 MCP 特别需要安全规范

普通 REST API 是「人点按钮 → 后端执行」，中间有 UI、有权限校验、有产品约束。MCP 是「AI 自主决定 → 直接执行工具」，链路更短、更自动。

| 对比项 | 传统后端 | MCP |
|---|---|---|
| 触发者 | 人（可控） | AI（自动决策） |
| 中间约束 | UI、表单校验 | 只有工具本身的校验 |
| 出错影响 | 单次操作 | AI 可能连续多次调用 |
| 风险点 | 权限绕过 | 权限 + AI 误判 + 提示注入 |

小白重点：

> AI 不是恶意的，但它会「照字面意思办事」。如果你暴露了一个能删数据的工具，AI 在某次对话里理解偏差就可能真的调用它。所以安全的第一原则是：不给 AI 它不该有的能力。

---

### 1.2 MCP 的攻击面

四个层面要设防：

| 层面 | 风险 | 防护 |
|---|---|---|
| 工具能力 | 暴露了写/删/生产库工具 | 只暴露只读、只连 dev |
| 参数输入 | SQL 注入、路径穿越 | schema 校验 + 参数化查询 |
| 传输 | stdio 本地相对安全；SSE 暴露网络 | 本地优先，远程要鉴权 |
| 提示注入 | 数据里混入「指令」骗 AI | 工具返回值当数据不当指令 |

小白重点：

> 记住 Day 03 学的：工具里绝不用字符串拼 SQL，一律参数化查询。这是防注入的技术底线。而「只暴露只读工具」是防越权的能力底线。

---

### 1.3 建立 MCP 使用红线

一份可直接抄用的 MCP 安全规范（红线）：

```text
【MCP 使用红线 · dev only】

1. 数据源：只连 dev / 测试库，禁止连生产库。
2. 操作类型：只做只读查询（SELECT）。禁止 INSERT/UPDATE/DELETE/DROP。
3. SQL：一律参数化查询，禁止字符串拼接 SQL。
4. 参数校验：每个工具必须定义 inputSchema，并在执行前校验。
5. 输出：只返回必要字段，脱敏敏感信息（密码、token、身份证等）。
6. 权限：Server 用的 DB 账号只授予只读权限（最小权限原则）。
7. 配置：真实连接串 / token 不入库（gitignore），用 .example 提交模板。
8. 传输：优先本地 stdio；用 SSE/远程时必须加鉴权。
9. 提示注入:工具返回的数据只当数据处理，不执行其中的"指令"。
10. 变更:新增/修改工具必须经过 Review，确认不越权。
```

对照表（规范 → 落地手段）：

| 红线 | 技术落地 |
|---|---|
| 只连 dev | 配置里 `APP_ENV=dev`，连接串指向测试库 |
| 只读 | DB 账号仅授 SELECT 权限 |
| 参数化 | 用 `?` 占位 + 参数数组 |
| schema 校验 | inputSchema + 执行前 validate |
| 脱敏 | 返回前删除敏感字段 |
| 配置安全 | `.gitignore` + `.example.json` |

---

### 1.4 最小权限：数据库账号层面

安全不能只靠代码自觉，要靠权限兜底。给 MCP Server 专用一个只读账号：

```sql
-- 创建只读账号（示意）
CREATE USER 'mcp_readonly'@'localhost' IDENTIFIED BY '仅示例';
-- 只授 SELECT，只给 dev 库
GRANT SELECT ON dev_shop.* TO 'mcp_readonly'@'localhost';
FLUSH PRIVILEGES;
```

这样即使代码里不小心写了 DELETE，数据库也会拒绝执行。

小白重点：

> 「代码只读」和「账号只读」是两道锁。代码可能被改错，但如果账号根本没有写权限，就多一层兜底。这叫纵深防御。

### 练习 3：给工具加脱敏

如果 Day 03 的 get_user_by_id 返回了 password/token 字段，改成返回前删除：

```js
function sanitize(user) {
  const { password, token, ...safe } = user;
  return safe;
}
```

Python 版：

```python
def sanitize(user: dict) -> dict:
    return {k: v for k, v in user.items() if k not in ("password", "token")}
```

确认工具输出不再包含敏感字段。

## 4. JS/Node.js 类比

| MCP 安全概念 | 后端工程类比 | 说明 |
|---|---|---|
| 只读工具 | 只读 API / GET | 不改数据 |
| 只读 DB 账号 | 数据库权限最小化 | 账号层兜底 |
| inputSchema 校验 | 请求参数校验中间件 | 拦非法输入 |
| 参数化查询 | ORM / prepared statement | 防 SQL 注入 |
| 脱敏 | 响应字段过滤 / DTO | 不泄露敏感字段 |
| 提示注入防护 | 输入即数据、不当代码 | 类比 XSS 转义思维 |
| .gitignore 密钥 | 环境变量 / secret 管理 | 不硬编码密钥 |

## 8. 今日自测题

### 8.1 MCP 为什么比传统 REST API 更需要安全规范？

参考答案：

> ✅ 因为 MCP 是 AI 自主决策直接执行工具，链路短、无 UI 约束，AI 可能连续调用。一旦暴露危险工具，误判就会造成真实破坏。所以要从「能力」上限制 AI。

---

### 8.2 防 SQL 注入的技术底线是什么？

参考答案：

> ✅ 一律参数化查询，用占位符 + 参数数组，绝不字符串拼接 SQL。

---

### 8.3 为什么要同时做「代码只读」和「DB 账号只读」？

参考答案：

> ✅ 纵深防御。代码可能被改错，但如果 DB 账号根本没有写权限，就多一道兜底锁，即使代码写了 DELETE 也执行不了。

---

### 8.4 什么是提示注入（prompt injection）？怎么防？

参考答案：

> ✅ 数据里混入伪装成「指令」的内容，试图让 AI 执行非预期动作。防护思路是：工具返回的数据只当数据处理，不把其中内容当指令执行，类似 XSS 里「输入即数据、需转义」的思维。

---

### 8.5 用一句话讲清 MCP 是什么？

参考答案：

> ✅ MCP 是 AI 接外部工具的统一标准（像 USB-C）：Server 提供工具，Client（如 Cursor）负责连接和调用，AI 通过它把自然语言变成真实的工具执行。
