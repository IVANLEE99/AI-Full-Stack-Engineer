## 1. 学习内容

### 1.1 什么叫"需求阶段收口"

收口 = 需求不再模糊、不再有开放问题，可以放心进入开发。

```text
没收口：  文档写了，但"这个字段哪来的""这个边界没定"还在问
收口了：  拿着 PRD + API 契约，开发能直接动手，不用回头问需求
```

小白重点：判断收口的唯一标准是"可开工性"。文档再漂亮，只要还留着"待定/TODO/看情况"，就没收口。

### 1.3 文档自洽性检查（最容易漏）

三份核心文档必须互相对得上：

```text
用户故事(US)  ←→  API 契约(接口)  ←→  PRD(整体)

检查：
- 每条 US 是否都能找到对应接口？（无孤儿故事）
- 每个接口是否都服务于某条 US？（无多余接口）
- PRD 里的功能清单是否和 US 一一对应？
- 字段命名在三份文档里是否一致？
```

举例自洽检查表：

| 用户故事 | 对应接口 | PRD 功能点 | 一致？ |
|---|---|---|---|
| US-001 智能问答 | POST /api/chat | 对话功能 | ✅ |
| US-002 订单查询 | GET /api/orders/{order_no} | 订单查询 | ✅ |
| US-003 政策问答 | POST /api/knowledge/search | 知识库检索 | ✅ |

小白重点：自洽性检查是需求收口的"最后一关"。三份文档对不上，开发时一定会踩坑。宁可今天多花半小时对表，也别留到编码周返工。

### 1.5 预习 Vue3 对话 UI 的核心概念

下周第一个要做的是对话界面，先认识几个关键点：

```text
对话 UI 三要素：
1. 消息列表：用数组渲染，每条 { role: 'user'|'assistant', content: '...' }
2. 输入框：绑定输入，回车/点击发送
3. 状态：加载中(loading)、出错(error)、空态(empty)
```

Vue3 伪代码（预习，不用现在跑）：

```vue
<script setup>
import { ref } from 'vue'

const messages = ref([])      // 消息列表
const input = ref('')         // 输入内容
const loading = ref(false)    // 加载态

async function send() {
  messages.value.push({ role: 'user', content: input.value })
  loading.value = true
  // 调用 Day 04 定义的 POST /api/chat
  // const res = await fetch('/api/chat', {...})
  loading.value = false
}
</script>
```

对应的 PHP 薄 API（预习）：

```php
<?php

declare(strict_types=1);

// 薄 API：只做接收 → 转交 Agent → 返回统一结构
// POST /api/chat
$input = json_decode(file_get_contents('php://input'), true);

$reply = handleChat($input['message'] ?? '');  // 交给 Agent 层

echo json_encode([
    'code' => 0,
    'message' => 'ok',
    'data' => ['reply' => $reply],
], JSON_UNESCAPED_UNICODE);
```

小白重点："薄 API"指后端接口只做很少的事：接收请求、调用 Agent/工具、按统一结构返回。业务逻辑在 Agent 和工具层，不堆在接口里。这正是 Day 03 架构图里"PHP 薄 API 层"的含义。

## 4. JS/Node.js 类比

| 本周概念 | Node/前端类比 | 说明 |
|---|---|---|
| 需求收口 | 需求冻结 / feature freeze | 不再改需求，进入实现 |
| 整体验收清单 | 发布前 checklist | 逐项确认可发 |
| 文档自洽性 | 类型对齐 / 接口联调 | 前后端约定一致 |
| 薄 API | Express 里薄薄的 route handler | 路由只转发，逻辑在 service |
| 对话 UI 消息列表 | React/Vue 列表渲染 | 数组 map 成 DOM |
| 从 PRD 到代码 | 从设计稿到组件 | 文档驱动实现 |

小白重点：需求周结束 ≈ "feature freeze（需求冻结）"。之后进开发就尽量不改需求。这和你写前端时"设计稿定了再动手"是同一个道理——先冻结，再实现，减少来回返工。
