## 1. 学习内容

### 1.1 为什么先画架构图

架构图就是软件的"施工图纸"。没有图纸就开工，就像不画图直接盖楼——盖到一半发现承重墙位置错了，返工代价极高。

架构图帮你在写第一行代码前回答三个问题：

```text
1. 系统由哪几个部分（模块/服务）组成？
2. 每个部分负责什么、不负责什么？
3. 请求/数据在它们之间怎么流动？
```

小白重点：架构图不追求画得漂亮，追求"边界清楚、数据流清楚"。哪怕是手画的框图，只要能说清"谁调谁、谁存什么"，就是好图。

---

### 1.2 四层总览

运营知识助手采用四层结构：

```text
┌──────────────────────────────────────────────┐
│  ① Vue3 前端（对话 UI）                          │
│     聊天窗口、输入框、历史记录、来源展示             │
└───────────────┬──────────────────────────────┘
                │ HTTP / JSON (RESTful)
                ▼
┌──────────────────────────────────────────────┐
│  ② PHP 薄 API 层（网关 / BFF）                   │
│     接收请求、鉴权、参数校验、编排、返回统一结构       │
└───────┬───────────────────┬──────────────────┘
        │                   │
        ▼                   ▼
┌───────────────┐   ┌───────────────────────────┐
│ ③ MCP 工具      │   │ ④ RAG 检索                  │
│  查库存/查订单   │   │  向量库检索政策文档 → 找依据    │
│（确定性动作）    │   │（模糊问答的知识来源）           │
└───────┬───────┘   └───────────┬───────────────┘
        │                       │
        └──────────┬────────────┘
                   ▼
        ┌────────────────────────┐
        │  大模型 LLM              │
        │  把工具结果/检索片段       │
        │  组织成自然语言答复         │
        └────────────────────────┘
```

各层一句话职责：

| 层 | 技术 | 负责 | 不负责 |
|---|---|---|---|
| ① 前端 | Vue3 | 界面、交互、展示来源 | 业务逻辑、数据存储 |
| ② API | PHP（薄） | 鉴权、校验、编排、统一返回 | 复杂算法、直接跑模型 |
| ③ MCP 工具 | PHP/脚本 | 确定性动作（查库存/订单） | 模糊问答 |
| ④ RAG | 向量库 + 检索 | 从文档找有依据的片段 | 精确数据（那是 MCP 的活） |

---

### 1.3 一次对话的数据流

以"US-007：客服问一条运营政策"为例，走一遍全流程：

```text
1. 客服在 Vue3 输入："满 99 包邮的活动什么时候结束？"
2. 前端 POST /api/chat  { message: "...", session_id: "..." }
3. PHP 薄 API：
   - 校验参数、鉴权
   - 判断这是"政策问答"类问题 → 走 RAG
4. RAG：把问题转成向量，在向量库检索最相关的政策文档片段
   → 找到《包邮活动细则 v3》第 2 条
5. PHP 把"问题 + 检索到的片段"打包给大模型
6. 大模型基于片段生成答复，并标注来源
7. PHP 组装统一响应返回前端
8. 前端展示答复 + 来源"《包邮活动细则 v3》"
```

再看"US-001：查库存"（走 MCP 而不是 RAG）：

```text
1. 客服问："蓝牙耳机A1 还有货吗？"
2. PHP 判断这是"确定性查询" → 调用 MCP 工具 queryStock
3. MCP 工具查商品系统只读接口 → 返回 { stock: 42, status: "有货" }
4. 大模型把结构化结果转成人话："蓝牙耳机A1 有货，库存 42 件。"
5. 返回前端
```

小白重点：**MCP 走"确定性数据"，RAG 走"模糊知识"**。库存数字必须精确（用 MCP 查真实系统），政策解读是模糊语义（用 RAG 找文档）。分清这两条路，是整个架构的核心。

---

### 1.4 RAG 是什么，为什么能"防止胡说"

RAG = Retrieval-Augmented Generation（检索增强生成）。

一句话：**先检索，再生成**。不让大模型凭"记忆"回答，而是先从你的知识库里捞出相关文档片段，把片段塞给模型，让它"照着材料回答"。

对比：

| 方式 | 过程 | 问题 |
|---|---|---|
| 纯大模型 | 直接问模型 | 会编造（幻觉），且不知道你公司的政策 |
| RAG | 先检索文档 → 带着片段让模型答 | 有据可查，能标来源，答不出就说不知道 |

RAG 的关键步骤：

```text
① 离线（建库）：把政策文档切成小段 → 转成向量 → 存进向量库
② 在线（问答）：
   问题 → 转成向量 → 在向量库找最相似的几段
        → 把这几段 + 问题一起给模型 → 生成带来源的答复
```

小白重点：这正好实现了 Day 02 的 US-008（答案附带来源）和 US-009（没有就说不知道）。因为答复是"照着检索到的片段"写的，所以能标来源；如果检索不到相关片段，就让模型回答"知识库中未找到相关内容"。

"向量"通俗解释：把一句话变成一串数字坐标，语义相近的句子坐标也相近。检索就是"找坐标最近的邻居"。这一步你不用手写算法，用现成的向量库（如轻量的本地方案）即可。

---

### 1.5 MCP 工具是什么

MCP（Model Context Protocol）你可以先理解为"给大模型用的标准化工具箱"。模型自己不会查数据库，但你可以给它注册一批"工具"，模型需要时就调用。

对助手来说，MCP 工具就是几个确定性动作：

```text
- queryStock(productName)   查库存
- queryOrder(orderNo)        查订单状态
- queryLogistics(orderNo)    查物流节点
```

每个工具有清晰的"输入 → 输出"，像一个普通函数：

```text
工具名：queryStock
输入：{ productName: string }
输出：{ found: bool, stock: int, status: "有货|缺货|预售" }
```

MCP 与 RAG 的分工再强调一次：

| 需求 | 用谁 | 为什么 |
|---|---|---|
| 查库存数量 | MCP 工具 | 要精确、实时、结构化 |
| 查订单状态 | MCP 工具 | 同上 |
| 解读运营政策 | RAG | 是文档里的模糊语义 |
| "退款要几天" | RAG（政策文档） | 属于规则解读 |

小白重点：新手容易想"全都塞给大模型让它自己搞"。但库存这种数据必须来自真实系统的确定性查询，绝不能让模型"猜一个数字"。工具（MCP）和知识（RAG）各司其职，才靠谱。

---

### 1.6 为什么 PHP 是"薄" API

"薄 API"（也叫 BFF，Backend For Frontend）的意思是：PHP 层尽量轻，只做这几件事：

```text
1. 接收前端请求、鉴权（谁在用）
2. 校验参数（消息不能为空等）
3. 判断走 MCP 还是 RAG（编排）
4. 调用工具/检索，把结果交给模型
5. 组装成统一格式返回前端
```

它**不做**：复杂业务算法、不把大模型跑在 PHP 进程里、不直接写业务库存等数据。

小白重点：薄 API 的好处是职责单一、好维护、好测试。前面阶段你学的 PHP 分层（Controller/Service）在这里依然适用——Controller 收请求，Service 负责编排 MCP/RAG。

### 1.7 技术选型与理由

今天还要产出一张"技术选型表"，每一项都要写"为什么选它"。示例：

| 层 | 选型 | 理由 |
|---|---|---|
| 前端 | Vue3 + Vite | 组件化、上手快、生态成熟，适合做对话 UI |
| 状态管理 | Pinia | Vue3 官方推荐，比 Vuex 简洁 |
| 后端 API | PHP 8 + 轻框架 | 延续本课程主线，薄网关够用 |
| 通信 | RESTful JSON | 简单直观，先不上流式（后续可加 SSE） |
| 检索 | 本地向量库方案 | 毕业项目规模小，本地即可，免运维 |
| 工具协议 | MCP | 标准化，模型调用工具的通用方式 |
| 大模型 | 可切换的模型接口 | 用抽象接口封装，方便替换 |

小白重点：选型没有"最好"，只有"最合适"。评价标准是：满足需求、团队熟悉、维护成本可控。毕业项目要避免为了炫技上一堆重型组件。

## 4. JS/Node.js 类比

| 架构概念 | 前端/Node 世界类比 | 说明 |
|---|---|---|
| 架构图 | 组件树 / 系统设计文档 | 先规划再动手 |
| Vue3 前端层 | React/Vue 单页应用 | 只管视图与交互 |
| PHP 薄 API（BFF） | Node.js 的 Express/Koa 网关 | 一层薄薄的接口编排 |
| RESTful JSON | fetch / axios 调接口 | 前后端约定的数据格式 |
| RAG 检索 | 先查数据库再渲染，而不是硬编码 | "先取数据再生成" |
| 向量库 | 全文搜索（如 Elasticsearch）的语义版 | 按相似度找内容 |
| MCP 工具 | 一组可被调用的 API/SDK 方法 | 确定性的函数调用 |
| 大模型 LLM | 一个外部 AI 服务 API | 调用它生成文本 |

小白重点：如果你写过 Node 的 BFF 层（前端专用的中间层，聚合多个后端接口再喂给前端），那 PHP 薄 API 就是同一个角色。BFF 不写核心业务，只做聚合和适配——薄 API 也是。

```js
// Node BFF 的味道，和 PHP 薄 API 是一回事
app.post('/api/chat', async (req, res) => {
  const { message } = req.body;
  const route = classify(message);        // 判断走工具还是检索
  const context = route === 'tool'
    ? await callMcpTool(message)           // MCP：确定性查询
    : await ragRetrieve(message);          // RAG：检索文档
  const answer = await llm(message, context);
  res.json({ answer, source: context.source });
});
```

## 8. 今日自测题

### 8.1 运营知识助手分哪四层？各管什么？

参考答案：

> ✅ ① Vue3 前端（界面/交互）② PHP 薄 API（鉴权/校验/编排/统一返回）③ MCP 工具（确定性动作，如查库存/订单）④ RAG 检索（从知识库找有依据的片段）。最后由大模型把工具结果或检索片段组织成人话。

---

### 8.2 RAG 是什么？为什么能防止胡说？

参考答案：

> ✅ RAG=检索增强生成，即"先检索再生成"。先从知识库捞出相关文档片段，让模型照着片段回答。因此答案有据可查、能标来源，检索不到就回答"不知道"，避免了模型凭记忆编造（幻觉）。

---

### 8.3 查库存该走 MCP 还是 RAG？为什么？

参考答案：

> ✅ 走 MCP 工具。库存数字必须精确、实时、结构化，只能来自真实系统的确定性查询，绝不能让模型"猜"。RAG 用于政策解读这类模糊语义问答。

---

### 8.4 "PHP 薄 API"的"薄"体现在哪？

参考答案：

> ✅ 薄 API（BFF）只做鉴权、参数校验、判断走 MCP 还是 RAG 的编排、以及组装统一返回。它不写复杂业务算法、不在 PHP 进程里跑模型、不直接写业务数据。职责单一、好维护好测试。

---

### 8.5 技术选型的评价标准是什么？

参考答案：

> ✅ 没有"最好"只有"最合适"：能满足需求、团队熟悉、维护成本可控。毕业项目应避免为炫技上重型组件，每项选型都要能说出理由。
