## 1. 学习内容

### 1.1 先认识毕业项目：运营知识助手

从这周开始，你要做一个贯穿全栈（Vue3 前端 + PHP 薄 API + MCP 工具 + RAG 检索）的毕业项目，代号 `graduation-project`。

它叫"运营知识助手"，一句话描述：

> 一个能回答运营同学日常问题的智能问答工具。比如"这个商品还有货吗""这个订单为什么退款失败""大促期间优惠券叠加规则是什么"，它能从公司内部的知识库里找到答案，并用自然语言回复。

小白重点：不要一上来就想"怎么写代码"。毕业项目和平时练习最大的区别是——它要求你**先想清楚做什么，再动手做**。本周整整 7 天都不写业务代码，全部用来把需求和边界想明白。

你可以把它类比成盖房子：

```text
Week 21（本周）= 画图纸、定预算、和业主确认需求
Week 22 之后    = 真正砌墙、通水电（写代码）
```

图纸没画好就砌墙，最后一定返工。

### 1.2 需求到底从哪里来

新手常见误区：需求 = 老板说的一句话。

实际上，一句"给运营做个问答机器人"里，藏着大量没说清楚的东西：

| 老板说的 | 真正要问清楚的 |
|---|---|
| 给运营做个问答机器人 | 哪些运营？（客服运营 / 商品运营 / 活动运营） |
| 能回答问题 | 回答哪些问题？答不上来怎么办？ |
| 从知识库找答案 | 知识库是什么？谁维护？多久更新？ |
| 智能一点 | 要不要联网？要不要查数据库？能不能瞎编？ |

所以需求的来源是**调研**，不是**猜测**。调研的对象是：

1. 真实用户（运营同学）
2. 用户现在的工作流程
3. 用户现在遇到的具体麻烦

小白重点：需求分析的第一性原理是"**替用户省事**"。如果你的产品没有让某个具体的人、在某个具体的场景下更省事，那它大概率没用。

---

### 1.3 什么是"场景"，它和"功能"的区别

这是今天最重要的概念。

- **功能（Feature）**：产品能做什么。例如"支持关键词搜索"。
- **场景（Scenario）**：用户在什么情况下、为了什么、会用到这个功能。例如"客服小李在接待客户时，客户问某商品有没有现货，小李需要 5 秒内查到库存"。

对比一下：

| 维度 | 功能视角（错误起点） | 场景视角（正确起点） |
|---|---|---|
| 出发点 | 我能做什么 | 用户要什么 |
| 描述 | 支持库存查询 | 客服接待时要 5 秒查到库存 |
| 是否可验证 | 模糊 | 可以：真去问客服"你现在怎么查的" |
| 容易犯的错 | 做一堆没人用的功能 | 只做用户真正需要的 |

小白重点：**先有场景，后有功能**。功能是场景的解决方案，不是起点。产品经理和普通开发最大的差距，就在于是否习惯从场景出发思考。

---

### 1.4 场景四要素：把痛点说清楚

一个合格的场景描述，必须包含四个要素。记住这个口诀：**谁、何时、想干啥、现状痛**。

```text
【角色】谁在用？（越具体越好，别写"用户"）
【触发】在什么情况下会用到？
【目标】他想达成什么结果？
【现状痛点】现在没有这个产品时，他怎么做？为什么难受？
```

我们先看一个反面例子（太模糊）：

```text
运营想快速查东西。
```

问题：谁？查什么？现在怎么查？完全没说清楚，无法验证，也无法设计功能。

再看一个正面例子：

```text
【角色】客服运营小李（每天接待 80+ 个客户咨询）
【触发】客户在聊天窗口问"XX 商品现在有货吗？"
【目标】5 秒内给出准确的"有货 / 缺货 / 预售"答复
【现状痛点】现在要切到后台系统，搜商品名，等页面加载，
            再看库存字段；一次要 30 秒以上，客户等得不耐烦，
            高峰期一天要查上百次，非常耗神。
```

对比一下就能感受到差距：

| 对比项 | 模糊场景 | 清晰场景 |
|---|---|---|
| 能不能画出用户画像 | 不能 | 能：高频接待的客服 |
| 能不能量化痛点 | 不能 | 能：30 秒 → 目标 5 秒 |
| 能不能推导出功能 | 不能 | 能：需要快速库存查询 |
| 能不能验证真假 | 不能 | 能：去问真客服核对 |

---

### 1.5 如何区分"真痛点"和"伪需求"

毕业项目最容易翻车的地方，就是把自己想象的需求当成真需求。这里给你一张自查表：

| 判断维度 | 真痛点的特征 | 伪需求的特征 |
|---|---|---|
| 频率 | 每天/每周高频发生 | 一年就一两次 |
| 人群 | 有明确的、真实存在的人 | "大家""所有人" |
| 现状 | 现在有个难受的替代方案 | 现在根本没人这么干 |
| 代价 | 不解决会浪费时间/出错 | 不解决也没啥损失 |
| 可验证 | 能找到真人核对 | 只能靠想象 |

小白重点：一个简单的验证方法叫"**假装它已经上线**"。问自己：如果这个功能明天就能用，那个具体的人（比如客服小李）会不会真的天天点开用？如果答案模糊，多半是伪需求。

### 1.6 为运营知识助手锁定 3 类核心场景

结合"运营知识助手"，我们把调研范围锁定在运营同学最高频的三类问题上。这也是后面写用户故事、定 API 的基础。

#### 场景类型 1：商品查询

```text
【角色】客服运营（高频接待客户）
【触发】客户询问某商品的库存 / 价格 / 规格 / 是否上架
【目标】快速拿到准确的商品信息，直接回复客户
【现状痛点】要登录多个后台系统交叉查询，慢且容易看错字段
```

#### 场景类型 2：订单查询

```text
【角色】售后运营
【触发】客户投诉"我的订单为什么还没发货 / 退款失败"
【目标】快速定位订单状态、物流节点、退款卡在哪一步
【现状痛点】订单系统、物流系统、支付系统是分开的，
            要跳三四个系统拼出全貌，新人根本记不住
```

#### 场景类型 3：政策问答

```text
【角色】活动运营 / 新入职运营
【触发】需要确认"优惠券能不能叠加""大促规则""退换货政策"
【目标】拿到权威、最新的规则说明，避免答错被投诉
【现状痛点】规则散落在几十个文档、群聊、邮件里，
            版本混乱，经常拿旧规则答客户，出了问题要背锅
```

小白重点：这三类场景刚好对应后面要设计的三个能力——**知识检索（RAG）**、**结构化查询（工具调用/MCP）**、**对话式回答（前端 + API）**。所以场景不是白列的，它直接决定了系统架构。

### 1.7 场景清单模板（今天要产出的东西）

把调研结果整理成一张清单。建议用下面的 Markdown 表格模板，存成 `graduation-project/docs/SCENARIOS.md`：

```markdown
# 运营知识助手 - 场景调研清单 v0.1

| 编号 | 场景名称 | 角色 | 触发时机 | 目标 | 现状痛点 | 频率 | 优先级 |
|------|----------|------|----------|------|----------|------|--------|
| SC-01 | 商品库存查询 | 客服运营 | 客户问是否有货 | 5秒内给出库存 | 切多系统查30秒+ | 高 | P0 |
| SC-02 | 订单状态查询 | 售后运营 | 客户催发货/退款 | 定位订单卡点 | 跨3系统拼信息 | 高 | P0 |
| SC-03 | 运营政策问答 | 活动运营 | 确认规则 | 拿到最新权威规则 | 规则散乱版本乱 | 中 | P1 |
```

小白重点：

- **优先级** P0/P1 是行业常用写法。P0 = 不做就没法上线的核心；P1 = 重要但可以稍后；P2 = 锦上添花。先做 P0。
- 频率和优先级往往正相关：越高频的痛点，越值得优先解决。

## 2. 源码阅读

本日无指定源码阅读，重点完成场景调研练习与复盘。

不过你可以做一件"类源码阅读"的事：找一份你熟悉的开源项目的 README 或 issue 模板（比如 GitHub 上任意热门项目的 `ISSUE_TEMPLATE`），观察它是怎么引导用户描述"问题场景"的。

阅读时重点找这些内容：

1. 它有没有让用户写"复现步骤"（对应场景的"触发时机"）
2. 它有没有让用户写"期望结果"（对应场景的"目标"）
3. 它有没有让用户写"实际结果"（对应场景的"现状痛点"）

建议你在笔记里写出类似表格：

| Issue 模板字段 | 对应场景要素 |
|---|---|
| 复现步骤 | 触发时机 |
| 期望行为 | 目标 |
| 实际行为 | 现状痛点 |
| 环境信息 | 角色/上下文 |

你会发现：好的 issue 模板，本质上就是在逼用户把"场景"说清楚。

## 4. JS/Node.js 类比

今天没有代码，但产品思维在前端/Node 世界同样通用。

| 产品/需求概念 | 前端/Node 世界的类比 | 说明 |
|---|---|---|
| 场景（Scenario） | 用户故事的来源、E2E 测试用例 | 都是"用户在什么情况下做什么" |
| 场景四要素 | 一条 Cypress/Playwright 测试的 Given-When-Then | 前置条件、操作、期望结果 |
| 真痛点 vs 伪需求 | 该不该造这个 npm 包 | 有没有人真的会 `npm install` |
| P0/P1 优先级 | GitHub issue 的 label（`priority:high`） | 排期依据 |
| 场景清单 | 需求 backlog | 待办事项池 |
| 替用户省事 | 好的 API 设计（少写样板代码） | 核心都是降低使用成本 |

小白重点：写过 E2E 测试的话，你会发现"场景"和"测试用例"几乎是一回事——都要写清楚"在什么前提下、做了什么操作、期望得到什么"。产品的场景，就是还没写成代码的测试用例。

## 8. 今日自测题

### 8.1 "场景"和"功能"有什么区别？

参考答案：

> ✅ 功能是"产品能做什么"，场景是"用户在什么情况下、为了什么而使用"。正确的做法是先有场景，后有功能——功能是场景的解决方案。

---

### 8.2 场景四要素是哪四个？

参考答案：

> ✅ 角色（谁）、触发时机（何时/什么情况）、目标（想达成什么）、现状痛点（现在怎么凑合、为什么难受）。口诀：谁、何时、想干啥、现状痛。

---

### 8.3 怎么判断一个需求是不是伪需求？

参考答案：

> ✅ 看五个维度：频率是否高、人群是否具体真实、现状有没有难受的替代方案、不解决有没有代价、能不能找真人核对。同时可以用"假装它已上线，那个人会不会天天用"来快速验证。

---

### 8.4 运营知识助手的 3 类核心场景是什么？分别对应什么技术能力？

参考答案：

> ✅ 商品查询（对应结构化查询/工具调用 MCP）、订单查询（对应跨系统结构化查询）、政策问答（对应知识检索 RAG）。场景直接决定了系统架构。

---

### 8.5 为什么本周不写业务代码，全用来做需求分析？

参考答案：

> ✅ 因为需求和边界没想清楚就写代码，一定返工。就像盖房子要先画图纸。毕业项目考察的正是"先想清楚做什么，再动手做"的工程能力。
