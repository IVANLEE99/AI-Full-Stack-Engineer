## 1. 学习内容

### 1.1 从场景到用户故事

昨天你写了 3 个场景（商品查询、订单查询、政策问答）。今天要把这些场景拆成更小、更可执行的"用户故事"。

关系是这样的：

```text
场景（大） → 拆成多个用户故事（小） → 每个故事再拆成开发任务（更小）
```

举例：

```text
场景 SC-01：商品库存查询
   ├── US-001：作为客服，我想用商品名查库存，以便快速回复客户
   ├── US-002：作为客服，我想看到库存状态标签（有货/缺货/预售）
   └── US-003：作为客服，商品不存在时我想收到明确提示
```

小白重点：场景是"一大块需求"，用户故事是"一小口能吃下的需求"。故事要小到一个开发者能在 1-2 天内做完，这样才好排期、好验收。

---

### 1.2 用户故事的标准句型

行业通用的用户故事模板（英文叫 User Story，源自敏捷开发）：

```text
作为【角色】，
我想要【某个功能/能力】，
以便【获得某种价值/达成某个目标】。
```

英文原版是：

```text
As a [role],
I want [feature],
so that [benefit].
```

三段式各自的作用：

| 段落 | 回答的问题 | 常见错误 |
|---|---|---|
| 作为【角色】 | 谁要用？ | 写"用户""系统"这种没脸的角色 |
| 我想要【功能】 | 要什么能力？ | 直接写技术实现（如"调 MySQL"） |
| 以便【价值】 | 为什么要？图啥？ | 省略，导致做完没人知道有啥用 |

小白重点：最容易被省略的是第三段"以便"。但它恰恰最重要——它是这个故事存在的理由。如果一个故事写不出"以便"，很可能是伪需求。

对比正误：

```text
❌ 作为用户，我想要一个搜索框。
   （角色太泛、没说图啥、更像在描述 UI）

✅ 作为客服运营，我想要用商品名搜索库存，
   以便在接待客户时 5 秒内答复是否有货。
   （角色具体、能力清晰、价值明确）
```

---

### 1.3 好故事的标准：INVEST 原则

判断一个用户故事写得好不好，业界用 INVEST 这个口诀（六个单词首字母）：

| 字母 | 含义 | 通俗解释 |
|---|---|---|
| **I**ndependent | 独立 | 尽量不依赖其他故事，能单独开发验收 |
| **N**egotiable | 可协商 | 是需求不是死命令，细节可以讨论 |
| **V**aluable | 有价值 | 对用户有实实在在的价值 |
| **E**stimable | 可估算 | 大小能被估出来（不是"改善体验"这种没边的） |
| **S**mall | 小 | 小到 1-2 天能做完 |
| **T**estable | 可测试 | 有明确的验收标准，能判断做没做完 |

小白重点：新手最常违反的是 **S（小）** 和 **T（可测试）**。像"作为用户，我想要一个好用的助手"——太大、没法测，直接作废。

---

### 1.4 验收标准（AC）：Given-When-Then

用户故事只说了"要什么"，但"做到什么程度算完成"要靠**验收标准（Acceptance Criteria，简称 AC）**。

最通用的 AC 写法是 **Given-When-Then**（前置条件 - 触发操作 - 期望结果）：

```text
Given（假设）某个前置条件成立，
When（当）用户执行了某个操作，
Then（那么）系统应该给出某个明确结果。
```

一个故事通常配 2-5 条 AC，覆盖正常情况和异常情况。

示例（对应"商品名查库存"这个故事）：

```text
AC1（正常）
Given 商品"蓝牙耳机A1"在库存系统中存在且有货
When 客服输入"蓝牙耳机A1"并查询
Then 助手返回该商品，并显示状态"有货"和剩余数量

AC2（缺货）
Given 商品存在但库存为 0
When 客服查询该商品
Then 助手显示状态"缺货"，且不显示负数

AC3（不存在）
Given 商品名在系统中查不到
When 客服查询
Then 助手明确提示"未找到该商品，请核对名称"，而不是报错或瞎编
```

小白重点：好的 AC 一定要覆盖"异常路径"。只写正常情况的 AC，等于没考虑"商品不存在""网络超时""用户输入乱码"这些真实会发生的情况。这也是资深工程师和新手的分水岭。

对比：

| 维度 | 弱 AC | 强 AC |
|---|---|---|
| 覆盖面 | 只有正常路径 | 正常 + 异常 + 边界 |
| 可测性 | "查询要准确" | Given-When-Then 三段明确 |
| 歧义 | 多 | 少，能直接写成测试用例 |

---

### 1.5 完整示例：US-001

把故事和 AC 组合起来，一个完整的用户故事条目长这样。这就是今天要产出的标准格式，建议存到 `graduation-project/docs/USER_STORIES.md`：

```markdown
### US-001 商品库存查询

- **角色**：客服运营
- **故事**：作为客服运营，我想要用商品名查询库存状态，
  以便在接待客户时快速答复"是否有货"。
- **优先级**：P0
- **对应场景**：SC-01
- **验收标准**：
  - AC1：Given 商品存在且有货，When 输入商品名查询，
    Then 返回商品并显示"有货"及剩余数量。
  - AC2：Given 商品存在但库存为 0，When 查询，
    Then 显示"缺货"，不显示负数。
  - AC3：Given 商品不存在，When 查询，
    Then 明确提示"未找到该商品"，不报错、不编造。
- **备注**：库存数据来自商品系统的只读接口，助手不写库存。
```

小白重点：注意最后的"备注"里写了一句边界——"助手不写库存"。在需求阶段就明确"我不做什么"，和明确"我做什么"一样重要。这能防止后面无限扩张。

### 1.6 为运营知识助手规划 US-001 到 US-010

围绕昨天的 3 个场景，把 10 条故事分配好。下面是建议的骨架（今天你要把每条补全 AC）：

| 编号 | 场景 | 故事一句话 | 优先级 |
|---|---|---|---|
| US-001 | 商品查询 | 用商品名查库存状态 | P0 |
| US-002 | 商品查询 | 看到有货/缺货/预售状态标签 | P0 |
| US-003 | 商品查询 | 商品不存在时收到明确提示 | P0 |
| US-004 | 订单查询 | 用订单号查订单当前状态 | P0 |
| US-005 | 订单查询 | 看到物流节点（已发货/在途/签收） | P1 |
| US-006 | 订单查询 | 退款失败时看到失败原因 | P1 |
| US-007 | 政策问答 | 用自然语言问运营政策，得到答案 | P0 |
| US-008 | 政策问答 | 答案附带来源文档（可追溯） | P0 |
| US-009 | 政策问答 | 知识库没有时，助手明确说"不知道" | P0 |
| US-010 | 通用 | 对话有历史记录，可回看上一条 | P2 |

小白重点：

- US-008（答案附带来源）和 US-009（不知道就说不知道）是 AI 类产品的**安全底线**。没有它们，助手会"一本正经地胡说八道"（业界叫"幻觉/hallucination"），这在企业里是不可接受的。
- 这两条也直接决定了 Day 3 的架构要用 RAG（检索增强），因为答案必须"有据可查"。

---

### 1.7 用户故事常见反模式

| 反模式 | 例子 | 为什么不好 | 怎么改 |
|---|---|---|---|
| 角色泛化 | 作为用户… | 不知道给谁做 | 写成"客服运营" |
| 写成技术任务 | 我想要建一张 orders 表 | 那是实现不是需求 | 写"我想查订单状态" |
| 缺"以便" | …我想要搜索框 | 不知道价值 | 补上"以便快速答复" |
| 太大 | 我想要一个智能助手 | 没法估、没法测 | 拆成 10 个小故事 |
| 无 AC | 只有故事没验收 | 做没做完说不清 | 补 Given-When-Then |

替代练习：找一个你用过的开源项目，去 GitHub 的 Issues / Discussions 里翻几条被标记为 `feature request`（功能请求）的帖子。

阅读时重点找这些内容：

1. 提需求的人有没有说清"我是谁、想干啥、图啥"
2. 维护者有没有追问验收标准（"做成什么样算满足你？"）
3. 一个大需求是怎么被拆成多个小 issue 的

建议在笔记里整理成表格：

| 观察点 | 该 issue 的情况 | 对应用户故事概念 |
|---|---|---|
| 提需求者角色 |  | 作为【角色】 |
| 想要的能力 |  | 我想要【功能】 |
| 期望价值 |  | 以便【价值】 |
| 有没有验收标准 |  | AC |
| 有没有被拆分 |  | 故事拆分 |

你会发现：写得清楚的 issue 天生就接近一条合格的用户故事，而含糊的 issue 往往推进不下去。

## 4. JS/Node.js 类比

| 用户故事概念 | 前端/Node 世界类比 | 说明 |
|---|---|---|
| 用户故事（User Story） | Jira / GitHub 上的一张 Story 卡片 | 敏捷开发的最小需求单元 |
| 三段式句型 | issue 模板里的"背景 / 期望 / 价值" | 结构化描述需求 |
| 验收标准 AC | 一条自动化测试用例 | 都在定义"什么算做完了" |
| Given-When-Then | BDD 测试（Cucumber / Jest 的 describe-it） | 前置-操作-断言 |
| INVEST 原则 | 好的 PR 应该小而聚焦 | 小步提交、易 review |
| 故事拆分 | 把大 feature 拆成多个小 PR | 便于并行开发和回滚 |
| P0/P1 优先级 | issue 上的 `priority` label | 排期依据 |

小白重点：如果你写过 Jest 测试，`describe('商品查询', () => { it('商品存在时返回有货', ...) })` 里的 `it('...')` 描述，本质就是一条 AC。用户故事的 AC 和你的测试用例名，是同一件事的两种语言。

```js
// 一条 AC 天然就能翻译成一条测试
describe('US-001 商品库存查询', () => {
  it('AC1: 商品有货时返回"有货"和数量', () => { /* ... */ });
  it('AC2: 库存为0时返回"缺货"且不为负', () => { /* ... */ });
  it('AC3: 商品不存在时提示"未找到"而非报错', () => { /* ... */ });
});
```

## 8. 今日自测题

### 8.1 用户故事的标准句型是什么？

参考答案：

> ✅ "作为【角色】，我想要【功能/能力】，以便【价值/目标】。"三段分别回答：谁用、要什么、图什么。其中"以便"最容易被漏掉，但它是故事存在的理由。

---

### 8.2 INVEST 原则指的是什么？

参考答案：

> ✅ Independent（独立）、Negotiable（可协商）、Valuable（有价值）、Estimable（可估算）、Small（小）、Testable（可测试）。用来判断一个用户故事写得好不好，新手最常违反 Small 和 Testable。

---

### 8.3 验收标准（AC）为什么要写异常路径？

参考答案：

> ✅ 因为真实世界一定会出现商品不存在、网络超时、输入非法等情况。只写正常路径的 AC，等于没考虑这些，做出来的东西一遇到异常就崩或瞎编。覆盖异常路径是资深工程师和新手的分水岭。

---

### 8.4 Given-When-Then 三段分别是什么意思？

参考答案：

> ✅ Given=前置条件（假设什么成立），When=触发操作（用户做了什么），Then=期望结果（系统应该给出什么）。它能直接翻译成一条自动化测试用例。

---

### 8.5 为什么 US-008、US-009 对 AI 助手特别重要？

参考答案：

> ✅ US-008 要求答案附带来源文档（可追溯），US-009 要求知识库没有时明确说"不知道"。这两条是 AI 产品的安全底线，用来防止"一本正经地胡说八道"（幻觉）。它们也决定了架构必须用 RAG（检索增强），保证答案有据可查。
