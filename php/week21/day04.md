## 1. 学习内容

### 1.1 什么是 API 契约，为什么重要

API 契约（API Contract）是前后端之间关于"接口长什么样"的书面约定。它规定：

```text
- 方法（GET / POST / ...）
- 路径（/api/chat）
- 请求参数（字段名、类型、是否必填）
- 响应结构（成功返回什么、失败返回什么）
- 错误码（每种错误怎么表示）
```

为什么重要？三个理由：

| 好处 | 说明 |
|---|---|
| 并行开发 | 契约定了，前端按契约造假数据先做界面，后端按契约实现，互不阻塞 |
| 减少扯皮 | "你返回的字段和说好的不一样"——有契约就有依据 |
| 可测试 | 契约是测试用例的来源，能直接对照验收 |

小白重点：本周前面（Day 02）你写了用户故事的验收标准（AC），今天的 API 契约就是把这些 AC 落到"接口字段"级别。AC 说"返回来源"，契约就要有一个 `sources` 字段。

---

### 1.2 一个接口定义要包含什么

定义一个接口，至少要说清这几项。以"发送对话消息"为例：

```text
接口名称：发送对话消息
方法 + 路径：POST /api/chat
简介：用户发一条消息，返回助手答复
请求头：Authorization: Bearer <token>
请求体（JSON）：见下
响应体（JSON）：见下
错误码：400 参数错误 / 401 未鉴权 / 500 服务异常
关联用户故事：US-001、US-007、US-008
```

小白重点：好接口定义要能让"没和你说过话的前端同学"照着就能调通，不需要再问你。字段要写清类型、是否必填、示例值。

---

### 1.3 统一响应结构

所有接口返回统一的外层结构，前端只写一套解析逻辑即可。

成功：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "answer": "蓝牙耳机A1 有货，库存 42 件。"
  }
}
```

失败：

```json
{
  "code": 40001,
  "message": "message 不能为空",
  "data": null
}
```

约定：

| 字段 | 类型 | 说明 |
|---|---|---|
| `code` | int | 0 表示成功，非 0 表示业务错误码 |
| `message` | string | 给人看的提示信息 |
| `data` | object\|null | 成功时的数据；失败时为 null |

小白重点：注意区分"HTTP 状态码"和"业务 code"。HTTP 状态码是传输层（200/400/500），业务 code 在响应体里表示业务结果。很多团队成功时 HTTP 用 200，业务结果看 `code`。保持全项目一致即可。

---

### 1.4 对话接口 `/api/chat`

这是核心接口，对应 US-001~US-009 的入口。

请求：

```text
POST /api/chat
Authorization: Bearer <token>
```

```json
{
  "session_id": "sess_20260707_001",
  "message": "满 99 包邮的活动什么时候结束？"
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `session_id` | string | 是 | 会话 ID，用于上下文关联 |
| `message` | string | 是 | 用户输入，非空，建议 ≤ 500 字 |

响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "reply_id": "reply_abc123",
    "answer": "满 99 包邮活动将于 6 月 30 日 24:00 结束。",
    "route": "rag",
    "sources": [
      {
        "title": "《包邮活动细则 v3》",
        "snippet": "满 99 元包邮，活动截止 6 月 30 日 24:00。",
        "doc_id": "doc_007"
      }
    ]
  }
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `reply_id` | string | 本次回复 ID，用于反馈/追溯 |
| `answer` | string | 助手答复文本 |
| `route` | string | 走的路径：`rag` / `tool` / `none` |
| `sources` | array | 来源列表（对应 US-008）；无来源时为 `[]` |

小白重点：`sources` 字段直接兑现了 US-008"答案要能追溯来源"。当走 MCP 工具查库存时，`route` 是 `tool`，`sources` 可为空但可以带上"数据来自库存系统"这类说明。

---

### 1.5 查询接口 `/api/query/*`

这类接口对应 MCP 的确定性查询（也可由前端直接调、或由对话接口内部调用）。定义两个示例。

查库存：

```text
GET /api/query/stock?product=蓝牙耳机A1
```

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "product": "蓝牙耳机A1",
    "found": true,
    "stock": 42,
    "status": "有货"
  }
}
```

查订单：

```text
GET /api/query/order?order_no=NO20260707001
```

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "order_no": "NO20260707001",
    "found": true,
    "status": "已发货",
    "logistics": "已到达 XX 分拨中心"
  }
}
```

| 说明 | 内容 |
|---|---|
| 只读 | 查询接口不改数据，只读（对应 Agent 边界：不下单不改价） |
| 找不到 | `found: false`，`data` 其余字段可为 null，`code` 仍为 0（查询成功但无结果） |

小白重点：查询接口是"只读"的。这呼应 Day 05 会强调的 Agent 边界——助手能查不能改。任何"下单/改价/退款"都不在本项目范围。

### 1.6 知识库接口 `/api/knowledge/*`

RAG 的知识来自文档，需要接口管理这些文档。定义两个基础接口。

上传/新增文档：

```text
POST /api/knowledge/docs
```

```json
{
  "title": "包邮活动细则 v3",
  "content": "满 99 元包邮，活动截止 6 月 30 日 24:00 ...",
  "category": "运营政策"
}
```

响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "doc_id": "doc_007",
    "chunks": 5,
    "status": "indexed"
  }
}
```

| 字段 | 说明 |
|---|---|
| `doc_id` | 文档 ID |
| `chunks` | 切成了几段（用于向量化） |
| `status` | `indexed` 表示已入向量库可被检索 |

文档列表：

```text
GET /api/knowledge/docs?category=运营政策&page=1&page_size=20
```

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "total": 12,
    "items": [
      { "doc_id": "doc_007", "title": "包邮活动细则 v3", "category": "运营政策" }
    ]
  }
}
```

小白重点：知识库接口体现了 RAG 的"离线建库"环节（Day 03 的 1.4）。有了这些接口，运营才能自己维护知识来源，助手才能答得上新政策。

---

### 1.7 统一错误码

集中定义一张错误码表，避免各写各的。

| code | HTTP | 含义 | 场景 |
|---|---|---|---|
| 0 | 200 | 成功 | 一切正常 |
| 40001 | 400 | 参数错误 | message 为空、超长 |
| 40100 | 401 | 未鉴权 | 缺 token 或 token 失效 |
| 40400 | 404 | 资源不存在 | doc_id 不存在 |
| 42900 | 429 | 请求过于频繁 | 限流 |
| 50000 | 500 | 服务内部错误 | 未预期异常 |
| 50010 | 502 | 上游模型/检索异常 | 大模型/向量库调用失败 |

小白重点：`50010` 单独列出，是因为 AI 类项目的上游（模型、向量库）不稳定很常见，前端需要能区分"是我参数错了"还是"上游挂了"，给用户不同提示。

## 4. JS/Node.js 类比

| API 契约概念 | 前端/Node 世界类比 | 说明 |
|---|---|---|
| API 契约 | OpenAPI / Swagger 文档 | 接口的书面规范 |
| 统一响应结构 | axios 拦截器里统一解析的 `{code,data}` | 前端一套逻辑处理 |
| 请求/响应字段类型 | TypeScript interface | 约定字段形状 |
| 契约先行 + mock | msw / json-server 造假数据 | 前端不等后端就能开发 |
| 错误码表 | HTTP 状态 + 业务错误枚举 | 统一的错误语义 |
| 只读查询接口 | GET 幂等接口 | 不改数据 |

小白重点：如果你在前端项目里写过 TypeScript 的 `interface`，那你已经在写"契约"了。API 契约就是把这份 interface 写成前后端都认的文档，而不是只存在于前端代码里。

```ts
// 前端用 TS interface 描述契约（和 API_CONTRACT.md 对齐）
interface ChatResponse {
  code: number;
  message: string;
  data: {
    reply_id: string;
    answer: string;
    route: 'rag' | 'tool' | 'none';
    sources: { title: string; snippet: string; doc_id: string }[];
  } | null;
}
```

## 8. 今日自测题

### 8.1 API 契约为什么能让前后端并行开发？

参考答案：

> ✅ 契约把接口的方法、路径、请求/响应字段都约定死了。前端可以按契约造假数据（mock）先做界面，后端按契约实现，两边互不阻塞，最后对接就能通。

---

### 8.2 一个接口定义至少要包含哪些要素？

参考答案：

> ✅ 方法+路径、请求头（鉴权）、请求字段（名/类型/是否必填）、响应结构（成功和失败）、错误码，以及关联的用户故事。目标是让没沟通过的前端照着就能调通。

---

### 8.3 HTTP 状态码和业务 code 有什么区别？

参考答案：

> ✅ HTTP 状态码是传输层结果（200/400/500）；业务 code 在响应体里表示业务结果（如 0 成功、40001 参数错误）。二者可以并存，关键是全项目保持一致。

---

### 8.4 对话接口里 `sources` 字段有什么作用？

参考答案：

> ✅ 它兑现 US-008"答案可追溯来源"，列出 RAG 检索到的文档片段来源（标题、片段、doc_id）。前端据此展示"依据来自《XX 细则》"，无来源时为空数组。

---

### 8.5 为什么错误码里要单独列"上游模型/检索异常"？

参考答案：

> ✅ 因为 AI 项目依赖大模型和向量库这些不稳定的上游，它们挂掉很常见。单列一个码（如 50010）能让前端区分"是用户参数错了"还是"上游挂了"，从而给用户不同的提示。
