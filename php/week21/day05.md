## 1. 学习内容

### 1.1 PRD 是什么

PRD（Product Requirements Document，产品需求文档）是"要做一个什么产品"的权威说明书。它回答四个问题：

```text
1. 为什么做？   —— 背景与痛点
2. 给谁做？     —— 目标用户
3. 做什么？     —— 功能范围（含明确不做的）
4. 做成什么样算成功？—— 验收标准与成功指标
```

小白重点：PRD 是"一个团队的共识锚点"。开发、测试、产品出现分歧时，翻 PRD。它不是给用户看的说明书，是给做产品的人看的。

---

### 1.2 PRD 标准骨架

一份中小型项目 PRD 的常见骨架：

| 章节 | 内容 | 来自本周哪天 |
|---|---|---|
| 1. 背景与目标 | 为什么做、要解决什么痛点 | Day 01 场景调研 |
| 2. 目标用户 | 谁用、什么角色 | Day 01 |
| 3. 范围与非目标 | 做什么、明确不做什么 | Day 01 + Agent 边界 |
| 4. 用户故事清单 | US-001~US-010 及 AC | Day 02 |
| 5. 系统架构 | 架构图与技术选型 | Day 03 |
| 6. 接口清单 | 对话/查询/知识库接口 | Day 04 |
| 7. 验收标准 | 整体验收条目 | Day 02 AC 汇总 |
| 8. 成功指标 | 上线后怎么衡量 | 今日新增 |
| 9. 风险与依赖 | 已知风险、外部依赖 | 今日新增 |
| 10. 里程碑 | 大致排期 | 今日新增 |

小白重点：注意第 4~6 章几乎是把前几天的产出直接搬进来。所以"写 PRD"的大半工作，其实你这周前四天已经做完了。

---

### 1.3 背景与目标怎么写

用一段话说清痛点和目标，别空谈。

```text
【背景】
运营同学每天被大量重复问题打断：商品是否有货、订单到哪了、
某活动政策怎么算。这些答案分散在库存系统、订单系统和一堆
文档里，人工查询慢且容易答错。

【目标】
做一个"运营知识助手"：用户用自然语言提问，助手自动判断该查
数据（库存/订单）还是查文档（政策），给出带来源的答复，把
重复问答的人力成本降下来。
```

小白重点：背景里的痛点要能对上 Day 01 的三个场景。目标要"可被验证"，别写"提升运营效率"这种没法测的话。

---

### 1.4 范围与非目标（最重要）

明确"做什么"容易，明确"不做什么"才见功力。非目标能防止范围蔓延。

```text
【本期做（In Scope）】
- 商品库存查询（自然语言）
- 订单状态查询（自然语言）
- 运营政策问答（RAG，带来源）
- 知识库文档管理（新增/列表）

【本期不做（Out of Scope / 非目标）】
- 不做下单、改价、退款等写操作
- 不做多轮复杂业务办理（如售后工单流转）
- 不做语音输入
- 不做多语言
```

Agent 边界（本项目的红线）：

| 能力 | 允许 | 说明 |
|---|---|---|
| 查库存/订单 | ✅ 只读 | 通过 MCP 工具查询 |
| 查政策文档 | ✅ | RAG 检索 |
| 下单/改价/退款 | ❌ | 严禁写操作 |
| 无依据编造答案 | ❌ | 查不到就说查不到（对应 US-009） |

小白重点：这张边界表是整个项目的"安全护栏"。它同时约束了 Day 04 的接口（查询接口只读）和 Day 03 的架构（MCP 工具不暴露写能力）。定稿时一定要写清。

---

### 1.5 成功指标

上线后拿什么判断"做成了"。指标要可量化。

| 指标 | 目标值 | 怎么测 |
|---|---|---|
| 意图路由准确率 | ≥ 90% | 抽样人工核对 route 是否选对 |
| 答案带来源率 | 100%（RAG 路径） | 检查 sources 非空 |
| 平均响应时间 | ≤ 3s | 接口埋点统计 |
| 无依据编造率 | 0（硬指标） | 抽查有无瞎编 |

小白重点：区分"成功指标"和"验收标准"。验收是"功能有没有做出来"（做没做），指标是"做得好不好"（好不好）。定稿要两者都有。

---

### 1.6 风险与依赖

提前把已知的坑写出来。

| 风险/依赖 | 影响 | 应对 |
|---|---|---|
| 大模型服务不稳定 | 答不出/超时 | 错误码 50010 + 前端降级提示 |
| 知识库文档过期 | 答案错误 | 提供知识库管理接口，运营定期更新 |
| 意图判断出错 | 路由到错误路径 | 保留 route 字段便于排查，可回退到 RAG |
| 向量检索召回差 | 找不到相关文档 | 优化切分与 embedding，可人工补文档 |

小白重点：把风险写进 PRD，不是为了显得严谨，是为了让团队提前有心理预期和应对方案。AI 项目的头号风险永远是"上游模型不稳 + 检索不准"。

---

### 1.7 定稿的判定标准

什么样的 PRD 才算能"定稿 v1.0"？满足下面全部：

```text
✅ 背景痛点能对上真实场景（Day 01）
✅ In Scope / Out of Scope 都写清了
✅ Agent 边界（红线）明确
✅ 每条用户故事都有可测的 AC（Day 02）
✅ 架构图与接口清单齐全（Day 03/04）
✅ 有可量化的成功指标
✅ 列出主要风险与应对
✅ 经过至少一轮 AI Review 并修订
```

小白重点：定稿不是"写完就定"，而是"过了检查清单 + 改过一轮"。定稿后要打版本号 v1.0，之后的改动走版本记录。

## 4. JS/Node.js 类比

| PRD 概念 | 前端/工程世界类比 | 说明 |
|---|---|---|
| PRD | RFC / 设计文档 | 动工前的权威说明 |
| 范围与非目标 | MVP 定义 / 需求冻结 | 明确边界防蔓延 |
| Agent 边界 | 权限白名单 | 只允许清单内的能力 |
| 用户故事 + AC | 需求 + 测试用例 | 可验收 |
| 接口清单 | OpenAPI 引用 | 复用 Day 04 契约 |
| 版本号 v1.0 | 语义化版本 SemVer | 变更可追溯 |
| 成功指标 | 埋点 / 监控指标 | 上线后看数据 |

小白重点：PRD ≈ RFC + OpenAPI 引用 + Agent 能力说明。你前面几天的产出（用户故事、架构、契约）都是 PRD 的"引用物料"，PRD 是把它们串成一个能被评审、能被开工的整体。

## 5. AI Review 提问

把 `PRD.md` 全文贴给 AI，问：

```text
我在做毕业项目"运营知识助手"，这是我的 PRD 初稿（Week 21 Day 05）：
（粘贴 PRD.md 全文）

请扮演资深产品经理 + 后端架构师，帮我评审能否定稿：

1. 背景痛点是否具体、能对上真实场景？
2. 范围与非目标是否清晰？Agent 边界（只读、不下单不改价、不编造）是否写死？
3. 每条用户故事的验收标准是否可测？
4. 架构与接口清单是否和用户故事对得上，有没有对不上的字段？
5. 成功指标是否可量化？风险是否覆盖了模型/检索不稳？
6. 如果我把这份 PRD 交给开发团队，他们能直接开工吗？还缺什么？

请用中文输出：可定稿的部分、必须补的部分、逐条修改建议、定稿前 checklist。
```

## 8. 今日自测题

### 8.1 PRD 主要回答哪四个问题？

参考答案：

> ✅ 为什么做（背景痛点）、给谁做（目标用户）、做什么（功能范围含非目标）、做成什么样算成功（验收标准与成功指标）。

---

### 8.2 为什么"非目标（Out of Scope）"很重要？

参考答案：

> ✅ 它防止范围蔓延，让团队清楚"这期不做什么"，避免开发中不断加需求。本项目明确不做下单/改价/退款、不做语音、不做多语言。

---

### 8.3 本项目的 Agent 边界是什么？

参考答案：

> ✅ 只读查询（库存/订单）+ RAG 政策问答（带来源），严禁下单/改价/退款等写操作，查不到就说查不到、不编造答案。

---

### 8.4 "验收标准"和"成功指标"有什么区别？

参考答案：

> ✅ 验收标准看"功能做没做出来"（做/没做），成功指标看"做得好不好"（可量化，如路由准确率≥90%、响应≤3s）。定稿两者都要有。

---

### 8.5 PRD 什么时候才能定稿 v1.0？

参考答案：

> ✅ 满足检查清单：章节齐全、范围与边界清晰、AC 可测、架构接口一致、指标可量化、风险已列，且经过至少一轮 AI Review 并修订，最后打版本号 v1.0。
