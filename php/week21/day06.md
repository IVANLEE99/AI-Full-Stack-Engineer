## 今日目标

用一份评审 checklist，对昨天定稿的 PRD 做一次"自问自答的评审会议"，找出漏洞、补齐风险与边界，得出"能开发/需返工"的结论。

今天你要真正掌握这一句话：

> 评审会议不是走过场，而是"需求门禁"：一份 PRD 只有过了门禁才允许进入开发。评审的核心动作是"扮演不同角色，用挑刺的眼光对每条需求提问"，把问题在写代码之前发现。

## 1. 学习内容

### 1.1 评审会议是什么

评审会议（Review）是开发前的"质量门禁"：把 PRD 拿出来，让相关角色一起挑毛病，确认"这份需求能不能开工"。

```text
没有评审：  写 PRD → 直接开发 → 开发中发现需求有洞 → 返工、扯皮
有评审：    写 PRD → 评审挑刺 → 补洞 → 开发（少返工）
```

小白重点：评审的价值在于"提前发现问题"。需求阶段改一句话，开发阶段可能要改一整个模块。越早发现，成本越低。

---

### 1.2 自问自答评审：扮演多个角色

一个人也能开评审会——方法是轮流"扮演"不同角色，用各自的视角挑刺。

| 角色 | 关注点 | 典型提问 |
|---|---|---|
| 产品经理 | 需求是否解决真实痛点 | 这个功能用户真的要吗？ |
| 后端开发 | 接口是否可实现、字段是否清晰 | 这个字段哪来的？怎么算？ |
| 前端开发 | 交互是否明确、状态是否齐全 | 加载中/出错/空态怎么显示？ |
| 测试 | 每条需求能不能测 | 这条 AC 怎么写测试用例？ |
| 安全 | 有没有越权、写操作风险 | Agent 会不会误下单/改价？ |

小白重点：轮流戴不同"帽子"提问，是自测需求最有效的方法。尤其"测试"和"安全"两顶帽子，最容易挖出隐藏的洞。

---

### 1.3 评审 Checklist

对 PRD 逐项检查，每项打 ✅ 通过 / ⚠️ 有问题 / ❌ 缺失。

| # | 检查项 | 判定 |
|---|---|---|
| 1 | 背景痛点具体、能对上真实场景 |  |
| 2 | 目标用户明确 |  |
| 3 | In Scope 清晰 |  |
| 4 | Out of Scope（非目标）明确 |  |
| 5 | Agent 边界（只读、不写、不编造）写死 |  |
| 6 | 用户故事符合"作为…我想…以便…"格式 |  |
| 7 | 每条用户故事都有可测 AC |  |
| 8 | 架构图技术选型合理、职责清晰 |  |
| 9 | 接口清单覆盖所有用户故事 |  |
| 10 | 接口字段命名统一（如都用 snake_case） |  |
| 11 | 统一响应结构 + 错误码表 |  |
| 12 | 成功指标可量化 |  |
| 13 | 主要风险已列 + 有应对 |  |
| 14 | 有版本号与修订记录 |  |

小白重点：checklist 就是"需求门禁的关卡"。任何一项是 ❌，原则上都不能进开发。⚠️ 要评估严重程度。

---

### 1.4 问题严重级别

评审发现的问题要分级，决定"是否阻塞开发"。

| 级别 | 含义 | 处理 |
|---|---|---|
| P0 阻塞 | 缺了没法开发（如接口不覆盖需求） | 必须改完才能开工 |
| P1 严重 | 会导致返工（如字段命名不统一） | 开工前改 |
| P2 一般 | 影响体验，可并行改 | 记录，开发中处理 |
| P3 建议 | 优化项 | 可延后 |

小白重点：不是所有问题都阻塞开发。学会分级，才不会因为一个 P3 小建议卡住整个项目，也不会放过一个 P0 大洞。

---

### 1.5 评审记录模板

评审要留痕，方便追踪问题是否修复。

```text
# PRD 评审记录 v1.0

日期：2026-07-12
评审人：本人（自问自答，多角色）

## 发现问题
| 编号 | 角色视角 | 问题描述 | 级别 | 状态 |
|------|---------|---------|------|------|
| R-01 | 测试 | US-005 的 AC 无法测（"响应快"没定义） | P1 | 已修：改为≤3s |
| R-02 | 后端 | 查询接口缺少订单不存在的错误码 | P1 | 已修：补 40402 |
| R-03 | 安全 | 未写死"禁止写操作"边界 | P0 | 已修：补 Agent 边界表 |
| R-04 | 前端 | 未定义流式返回的加载态 | P2 | 记录，开发中处理 |

## 结论
- P0/P1 全部修复
- 评审结论：✅ 允许进入开发
```

小白重点：每个问题都要有"编号 + 级别 + 状态"。评审的闭环是"发现 → 修订 → 复评 → 关闭"，不是"提了就完"。

---

### 1.6 一个完整的自问自答样例

对 US-002（订单状态查询）走一遍多角色提问：

```text
【产品】运营真的常查订单状态吗？—— 是，Day 01 场景二已确认。
【后端】order_no 从哪来？用户直接输入还是从对话里抽取？
        —— 需补充：由模型从用户消息中抽取，抽不到则反问用户。
【前端】订单不存在时前端显示什么？—— 需定义空态文案。
【测试】AC"返回订单状态"能测吗？—— 能：给已知 order_no，断言 status 字段。
【安全】这个接口只读吗？会不会能改订单？—— 只读，已在 Agent 边界写死。
```

从这一轮就挖出 2 个需要补充的点（order_no 来源、订单不存在的处理）。

小白重点：这就是评审的真实产出——把"想当然"的地方问出来，逼自己写清楚。

## 4. JS/Node.js 类比

| 评审概念 | 工程世界类比 | 说明 |
|---|---|---|
| 需求评审 | Code Review | 都是"合入前挑刺" |
| 评审 checklist | PR 模板 / CI 检查项 | 逐项过关 |
| P0/P1/P2/P3 | Issue 优先级标签 | 分级处理 |
| 门禁（不过不开发） | CI 红灯不允许 merge | 质量门 |
| 评审记录 | PR 评论区讨论记录 | 留痕可追踪 |
| 五角色提问 | 多 reviewer 会审 | 不同视角挑刺 |

小白重点：需求评审 ≈ 代码评审，只是对象从代码变成了 PRD。"CI 不过不许合并"对应"评审不过不许开发"。你在前面 PHP 阶段习惯的 PR Review 流程，思路完全可以迁移到需求评审。

## 8. 今日自测题

### 8.1 需求评审的核心作用是什么？

参考答案：

> ✅ 作为"需求门禁"，在写代码之前发现问题。需求阶段改一句话的成本，远低于开发阶段改一个模块。

---

### 8.2 一个人怎么开评审会？

参考答案：

> ✅ 自问自答：轮流扮演产品、后端、前端、测试、安全等角色，用各自视角对每条需求挑刺提问。测试和安全视角最容易挖出隐藏问题。

---

### 8.3 评审发现的问题为什么要分级？

参考答案：

> ✅ 不是所有问题都阻塞开发。P0（缺了没法开发）、P1（会返工）必须开工前改；P2/P3 可并行或延后。分级避免被小问题卡住，也不放过大洞。

---

### 8.4 评审记录怎样才算闭环？

参考答案：

> ✅ 每个问题有"编号 + 级别 + 状态"，走完"发现 → 修订 → 复评 → 关闭"。不是提了就完，要跟踪到关闭。

---

### 8.5 评审通过的判定条件是什么？

参考答案：

> ✅ checklist 关键项全部 ✅，且 P0、P1 问题全部修复关闭，才允许进入开发。P2/P3 可记录后并行处理。
