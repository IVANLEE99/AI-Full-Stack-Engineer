## 1. 学习内容

### 1.1 为什么订单需要状态？

订单不是一条静态数据。它会经历多个阶段：

```text
创建订单 → 待支付 → 已支付 → 待发货 → 已发货 → 已完成
```

也可能中途取消或售后：

```text
待支付 → 已取消
已支付 → 退款中 → 已退款
```

状态的作用：

| 作用 | 示例 |
|---|---|
| 告诉用户订单进度 | 待支付、待发货、已完成 |
| 控制可执行操作 | 待支付可支付，已完成可评价 |
| 控制后端业务规则 | 已取消订单不能再支付 |
| 控制前端展示 | 不同状态展示不同 badge 和按钮 |
| 支撑运营和售后 | 退款、发货、取消都依赖状态 |

---

### 1.2 什么是状态机？

状态机可以理解为：

```text
一个对象有哪些状态，以及状态之间允许如何变化。
```

订单状态机就是：

```text
订单有哪些状态，以及订单可以从哪个状态流转到哪个状态。
```

例如：

```text
待支付 → 已支付：允许
待支付 → 已取消：允许
已取消 → 已支付：不允许
已完成 → 待支付：不允许
```

小白重点：状态机的重点不只是“有哪些状态”，还有“哪些状态流转是合法的”。

---

### 1.3 `STATUS_MAPPING` 怎么理解？

项目中可能有类似：

```php
<?php

public const STATUS_PENDING_PAY = 10;
public const STATUS_PAID = 20;
public const STATUS_SHIPPED = 30;
public const STATUS_FINISHED = 40;
public const STATUS_CANCELED = 90;

public const STATUS_MAPPING = [
    self::STATUS_PENDING_PAY => '待支付',
    self::STATUS_PAID => '已支付',
    self::STATUS_SHIPPED => '已发货',
    self::STATUS_FINISHED => '已完成',
    self::STATUS_CANCELED => '已取消',
];
```

它解决的是：

```text
数据库里存数字状态，代码和前端展示需要读懂这些数字代表什么。
```

不要只背数字，要理解含义。

---

### 1.4 后端状态 vs 前端展示状态

后端状态通常更细，前端展示可能更聚合。

| 后端状态 | 前端文案 | 前端按钮 |
|---|---|---|
| 待支付 | 待支付 | 去支付、取消订单 |
| 已支付/待发货 | 待发货 | 查看详情 |
| 已发货 | 待收货 | 确认收货、查看物流 |
| 已完成 | 已完成 | 评价、再次购买 |
| 已取消 | 已取消 | 删除订单、再次购买 |
| 退款中 | 退款中 | 查看售后 |

前端 badge 可能来自后端返回的状态文案，也可能前端自己用 `statusMap` 映射。

---

### 1.5 订单状态流转图

你可以先画基础版：

```text
创建订单
  ↓
待支付
  ├─ 支付成功 → 已支付/待发货 → 已发货 → 已完成
  └─ 超时/用户取消 → 已取消

已支付/待发货
  ├─ 商家发货 → 已发货
  └─ 申请退款 → 退款中 → 已退款
```

真实项目可能更复杂，但小白阶段先掌握主线。

---

### 1.6 每个状态下允许什么操作？

| 状态 | 允许操作 | 不允许操作 |
|---|---|---|
| 待支付 | 支付、取消 | 发货、评价 |
| 已支付/待发货 | 申请退款、等待发货 | 再次支付 |
| 已发货 | 确认收货、查看物流 | 取消订单 |
| 已完成 | 评价、再次购买 | 支付、发货 |
| 已取消 | 再次购买 | 支付、发货 |

后端必须校验操作是否合法，不能只靠前端隐藏按钮。

---

### 1.7 非法状态流转有什么风险？

| 非法流转 | 风险 |
|---|---|
| 已取消 → 已支付 | 用户可能支付不存在/失效订单 |
| 已完成 → 待支付 | 数据混乱 |
| 未支付 → 发货 | 商家损失 |
| 已退款 → 发货 | 资金和物流混乱 |
| 重复支付成功回调 | 订单状态重复更新 |

小白重点：订单状态是资金、库存、发货、售后的共同基础，不能随便改。

---

### 1.8 JS/Node.js 类比

前端可能有：

```js
const statusMap = {
  10: '待支付',
  20: '待发货',
  30: '待收货',
  40: '已完成',
  90: '已取消',
};
```

后端 PHP 里的 `STATUS_MAPPING` 类似：

```php
<?php

public const STATUS_MAPPING = [
    10 => '待支付',
    20 => '待发货',
    30 => '待收货',
    40 => '已完成',
    90 => '已取消',
];
```

但后端比前端多一个责任：校验状态流转是否合法。

## 8. 今日自测题

### 8.1 什么是订单状态机？

参考答案：

> ✅ 状态机描述一个对象有哪些状态，以及状态之间允许如何变化。订单状态机就是订单有哪些状态（如待支付、已支付、已发货、已完成、已取消），以及订单可以从哪个状态合法地流转到哪个状态。重点不只是“有哪些状态”，还有“哪些流转是合法的”。

---

### 8.2 `STATUS_MAPPING` 解决的是什么问题？

参考答案：

> ✅ 数据库里通常只存数字状态（如 10、20、30），代码和前端需要读懂这些数字代表什么。`STATUS_MAPPING` 就是把状态数值映射成可读文案，例如 `10 => '待支付'`。

---

### 8.3 后端状态和前端展示状态一定一一对应吗？

参考答案：

> ✅ 不一定。后端状态通常更细，前端展示可能更聚合。例如后端“已支付/待发货”在前端可能统一显示为“待发货”。前端 badge 可以来自后端返回的文案，也可以前端用 `statusMap` 自己映射。

---

### 8.4 举一个非法状态流转的例子，并说明风险。

参考答案：

> ✅ 例如“已取消 → 已支付”是非法流转，如果允许，用户可能对一个已失效的订单完成支付，导致资金和订单数据混乱。其它如“已完成 → 待支付”“未支付 → 发货”都属于必须禁止的非法流转。

---

### 8.5 状态流转校验为什么不能只靠前端隐藏按钮？

参考答案：

> ✅ 前端隐藏按钮只是界面控制，用户可以绕过前端直接调接口。订单状态关系到资金、库存、发货、售后，后端必须校验每个操作在当前状态下是否合法，不能随便更改。
