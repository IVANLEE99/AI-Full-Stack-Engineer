## 1. 学习内容

### 1.1 为什么下单需要防重复提交？

用户点击“提交订单”时，可能发生：

- 用户手速很快，连续点击两次
- 网络慢，用户以为没反应又点一次
- 前端重复发送请求
- App 重试请求
- 恶意用户直接用脚本重复请求

如果后端没有防护，可能出现：

| 问题 | 后果 |
|---|---|
| 创建重复订单 | 用户看到多笔待支付订单 |
| 重复扣库存 | 库存不准确 |
| 重复使用优惠券 | 优惠券被多次核销 |
| 重复创建支付单 | 支付状态混乱 |
| 重复写入订单明细 | 数据不一致 |

小白重点：订单接口必须假设请求会重复到达。

---

### 1.2 前端防重复点击够不够？

前端常见做法：

```js
submitButton.disabled = true;
await createOrder();
submitButton.disabled = false;
```

这能减少用户误点，但不能保证安全。

原因：

- 用户可以绕过前端直接调接口
- 前端 JS 可能被篡改
- 网络重试仍可能发生
- 多设备同时提交无法靠一个按钮控制

所以正确思路是：

```text
前端 disabled 提升体验
后端锁/幂等保证正确性
```

---

### 1.3 Redis 分布式锁怎么理解？

最简单的锁思路：

```text
同一个用户同一个下单动作，在一段时间内只允许一个请求进入核心逻辑。
```

Redis 常见命令思路：

```text
SET lock_key random_value NX EX 10
```

含义：

| 部分 | 含义 |
|---|---|
| `lock_key` | 锁的 key |
| `random_value` | 锁的持有者标识 |
| `NX` | 只有 key 不存在时才设置成功 |
| `EX 10` | 10 秒后自动过期 |

如果设置成功，说明拿到锁；如果失败，说明已有请求正在处理。

---

### 1.4 `LockHandleRedis` 可能做什么？

项目里的 `LockHandleRedis` 这类工具通常封装：

- 生成锁 key
- 尝试加锁
- 设置过期时间
- 执行业务逻辑
- finally 中释放锁
- 加锁失败时返回错误

伪代码：

```php
<?php

$lockKey = 'order:create:' . $userId;

if (!$lock->acquire($lockKey, 10)) {
    return $this->endFail('请勿重复提交');
}

try {
    $result = $this->orderService->create($params);
} finally {
    $lock->release($lockKey);
}
```

小白重点：锁必须有过期时间，否则服务异常后可能永远锁住。

---

### 1.5 锁粒度怎么设计？

锁粒度决定“锁住多大范围”。

常见选择：

| 锁 key | 粒度 | 优缺点 |
|---|---|---|
| `order:create:{userId}` | 用户级 | 简单，防止同一用户重复下单 |
| `order:create:{userId}:{cartId}` | 用户 + 购物车 | 更精确 |
| `order:create:{userId}:{goodsId}` | 用户 + 商品 | 适合单商品立即购买 |
| `stock:{skuId}` | SKU 库存级 | 控制库存并发，但压力大 |

原则：

```text
锁太粗：影响正常并发
锁太细：挡不住重复提交
```

例如同一个用户同时买两个完全不同商品，如果用 `order:create:{userId}`，可能会被误拦截；但对小白阶段，先理解用户级锁最容易。

---

### 1.6 锁超时怎么设计？

锁必须设置过期时间。

如果超时时间太短：

```text
订单还没创建完，锁已经过期，第二个请求又进来了。
```

如果超时时间太长：

```text
请求失败后用户很久不能再次提交。
```

设计时要考虑：

| 因素 | 说明 |
|---|---|
| 平均下单耗时 | 例如 500ms-2s |
| 最慢情况 | 第三方服务、库存、优惠券 |
| 用户体验 | 失败后多久允许重试 |
| 是否有自动释放 | `finally` 释放 + 过期兜底 |

常见做法：

```text
业务正常结束主动释放；异常时依赖过期时间兜底。
```

---

### 1.7 锁和幂等有什么区别？

锁和幂等都能处理重复请求，但重点不同。

| 机制 | 重点 | 示例 |
|---|---|---|
| 锁 | 同一时刻只让一个请求执行 | 用户下单时加锁 |
| 幂等 | 同一请求重复执行也只产生一次结果 | 使用 request_id / order_token |

更完善的下单设计通常是：

```text
前端防重复点击 + 后端锁 + 幂等 token + 数据库唯一约束
```

小白阶段先掌握锁，后续再深入幂等。

---

### 1.8 Node.js 类比

Node 中 Redis 锁伪代码：

```js
const lockKey = `order:create:${userId}`;
const ok = await redis.set(lockKey, requestId, 'NX', 'EX', 10);

if (!ok) {
  throw new Error('请勿重复提交');
}

try {
  return await orderService.create(params);
} finally {
  await releaseLock(lockKey, requestId);
}
```

PHP 的 `LockHandleRedis` 类似，只是封装形式不同。
