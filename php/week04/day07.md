### 1.6 预习 BFF：为什么下一阶段要学网关？

前 4 周主要关注单体/基础项目阅读。

下一阶段会进入：

```text
BFF 网关 + 微服务
```

你要先理解一个问题：如果前端页面需要多个后端服务的数据，应该怎么办？

可能的结构：

```text
前端
  ↓
BFF / 网关
  ↓
用户服务 / 商品服务 / 订单服务 / 支付服务
```

BFF 会负责：

- 接收前端请求
- 做登录校验
- 聚合多个服务数据
- 统一响应格式
- 适配前端页面需要的数据结构

但核心业务规则仍然应该放在具体服务里。

### 1.14 第六步：预习 BFF 的核心动机

下一阶段会学 BFF/网关。先理解为什么需要它。

假设一个首页需要：

- 用户信息
- 商品列表
- 优惠券
- 订单状态
- 站点配置

如果前端直接调用多个内网服务，会有问题：

1. 前端要知道太多后端细节。
2. 多个请求会增加页面加载复杂度。
3. 登录鉴权、错误处理、响应格式会分散。
4. 内网服务不应该全部暴露给前端。
5. 前端页面需要的数据结构和后端服务结构不一定一致。

BFF 的作用：

```text
专门为前端页面聚合和适配数据的后端层。
```

简单图：

```text
前端页面
  ↓
BFF / 网关
  ├─ 用户服务
  ├─ 商品服务
  ├─ 订单服务
  └─ 配置服务
```

---

### 1.15 第七步：BFF 应该做什么，不应该做什么

| BFF 应该做 | BFF 不应该做 |
|---|---|
| 接收前端请求 | 替代所有后端服务 |
| 做登录鉴权 | 写大量核心业务规则 |
| 聚合多个服务数据 | 直接承担复杂订单/支付逻辑 |
| 适配前端字段 | 绕过服务边界直接乱查库 |
| 统一响应格式 | 把所有逻辑堆成新单体 |

小白理解：

```text
BFF 像前端的专属服务员，负责把多个后厨的菜组合成前端需要的一份套餐。
但真正做菜的核心逻辑仍然在各个后端服务里。
```
