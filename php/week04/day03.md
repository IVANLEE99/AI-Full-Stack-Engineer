## 1. 学习内容

### 1.1 什么是配置中心？

配置中心可以理解成一个“集中管理配置的地方”。

以前配置可能写在代码里：

```php
<?php

$enableCoupon = true;
```

后来可能写在本地配置文件：

```php
<?php

return [
    'enable_coupon' => true,
];
```

再进一步，企业项目会把配置放到远程配置中心：

```text
配置中心
  - site.show_coupon = true
  - pay.enable_stripe = false
  - order.max_amount = 10000
```

业务代码通过统一函数读取：

```php
<?php

$enableCoupon = g_config('site', 'show_coupon', false);
```

小白重点：配置中心不是 PHP 独有概念，而是后端工程里常见的配置管理方案。

---

### 1.2 远程配置、本地配置、环境变量有什么区别？

| 类型 | 常见位置 | 适合存什么 | 是否适合频繁改 |
|---|---|---|---|
| 环境变量 | `.env` / 服务器环境 | DB 密码、Redis 地址、运行环境 | 不适合 |
| 本地配置文件 | `config/*.php` | 框架配置、服务配置 | 不适合频繁改 |
| 远程配置中心 | Nacos/Consul/etcd/DB | 业务开关、运营配置、阈值 | 适合 |
| 本地缓存 | 文件/内存/Redis | 远程配置副本 | 自动同步 |

举例：

```text
数据库密码：适合环境变量
是否展示优惠券入口：适合远程配置
路由规则：适合代码或框架配置
首页 banner：适合配置中心或后台配置
```

不要把所有东西都放配置中心。配置中心是工具，不是垃圾桶。

---

### 1.3 什么是动态配置？

动态配置是指：系统运行过程中，配置可以被调整，并影响业务行为。

例如：

```php
<?php

$showCoupon = g_config('site', 'show_coupon', false);
```

运营在后台把 `show_coupon` 从 `false` 改成 `true` 后，前端优惠券入口开始显示。

适合动态配置的场景：

| 场景 | 为什么适合动态配置 |
|---|---|
| 功能开关 | 可以快速开启/关闭 |
| 活动开关 | 活动开始结束需要调整 |
| 首页展示 | banner、导航经常变 |
| 风控阈值 | 需要按风险动态调整 |
| 支付渠道 | 某渠道异常时可临时关闭 |

---

### 1.4 什么是静态配置？

静态配置是指：不希望运行时随意变化，通常跟部署、框架、基础设施有关。

例如：

```env
APP_ENV=production
DB_HOST=127.0.0.1
REDIS_HOST=127.0.0.1
```

这些配置通常在部署时确定，不应该被运营后台随意修改。

适合静态配置的场景：

| 场景 | 为什么不适合动态配置 |
|---|---|
| 数据库连接 | 改错会导致系统不可用 |
| Redis 地址 | 属于基础设施配置 |
| APP_ENV | 决定运行环境 |
| 日志目录 | 和服务器部署相关 |
| Composer autoload | 属于代码结构 |

小白重点：动态配置偏业务，静态配置偏运行环境和基础设施。

---

### 1.5 为什么需要本地缓存？

如果每次调用 `g_config()` 都访问远程配置中心，会有问题：

- 请求慢
- 配置中心压力大
- 配置中心故障时业务也跟着故障
- 网络抖动会影响接口响应

所以常见做法是：

```text
远程配置中心
  ↓ 同步/拉取
本地缓存 / Redis / 文件缓存
  ↓ 快速读取
g_config()
  ↓
业务代码
```

本地缓存的作用：

| 作用 | 说明 |
|---|---|
| 提升性能 | 不必每次远程请求 |
| 降低依赖 | 配置中心短暂异常时仍可使用旧值 |
| 稳定接口 | 避免配置读取拖慢业务接口 |
| 支持兜底 | 没有缓存时再使用 default |

---

### 1.6 配置读取完整流程

你可以把完整流程画成：

```text
运营/管理员
  ↓ 修改配置
远程配置中心（如 Nacos/Consul/etcd/DB）
  ↓ 推送或拉取
本地缓存 / Redis / 文件缓存
  ↓
g_config(module, key, default)
  ↓
业务代码判断
  ↓
前端展示或业务行为变化
```

如果配置读取失败，兜底流程可能是：

```text
读本地缓存
  ↓ 如果没有
读远程配置
  ↓ 如果失败
返回 default
```

注意：真实项目里的顺序可能不同，你要以项目源码为准。

---

### 1.7 Nacos、Consul、etcd 怎么理解？

你今天不需要安装它们，只需要知道它们大概解决什么问题。

| 工具 | 可以类比成什么 | 常见用途 |
|---|---|---|
| Nacos | 配置中心 + 服务发现平台 | Java/微服务体系常见 |
| Consul | 服务发现 + KV 配置 | 微服务基础设施 |
| etcd | 分布式 KV 存储 | Kubernetes 等系统常见 |

对 PHP 小白来说，先记住：

```text
它们都可以作为“远程配置存储/分发”的基础设施。
```

不要一开始就陷入安装和集群原理，先理解“为什么需要远程配置”。

---

### 1.8 什么时候不要用动态配置？

动态配置很方便，但不是越多越好。

不适合动态配置的场景：

| 场景 | 风险 |
|---|---|
| 数据库表结构 | 改错会导致代码和 DB 不匹配 |
| 核心金额公式 | 可能引发财务风险 |
| 权限规则核心逻辑 | 改错会越权 |
| 密钥明文配置 | 有泄露风险 |
| 复杂业务流程 | 配置过多会变成不可维护的规则引擎 |

判断原则：

```text
如果改错会造成资金、安全、数据一致性严重问题，就不要轻易做成普通动态配置。
```

---

### 1.9 小白实操：把配置中心想成“远程开关面板”

如果你刚接触后端，可以先把配置中心想成一个后台开关面板：

```text
开关面板上有很多按钮：
- 是否展示优惠券
- 是否开启 Stripe 支付
- 最大下单金额是多少
- 首页 banner 是哪几张图
```

业务代码不把这些值写死，而是每次通过 `g_config()` 去读取：

```php
<?php

$showCoupon = g_config('site', 'show_coupon', false);
```

这句话背后的含义是：

```text
去配置系统里找 site 模块的 show_coupon 配置。
如果找不到，就用 false。
```

配置中心的价值就是：不用改代码，也能调整一部分业务行为。

---

### 1.10 第一步：先区分三类配置

学习配置中心最容易混淆的是：环境变量、本地配置、远程配置。

你可以按“谁改、什么时候改、改了影响什么”来区分。

| 配置类型 | 谁通常修改 | 修改频率 | 示例 | 小白理解 |
|---|---|---:|---|---|
| 环境变量 | 运维/开发 | 很低 | `DB_HOST`、`APP_ENV` | 程序运行环境 |
| 本地配置文件 | 开发 | 低 | `config/app.php` | 框架启动设置 |
| 远程配置中心 | 开发/运营/管理员 | 中到高 | `show_coupon`、`enable_pay` | 业务开关面板 |

判断口诀：

```text
决定系统怎么运行的，多半是环境/框架配置。
决定业务怎么展示或开放的，多半是业务动态配置。
```

---

### 1.11 第二步：理解“动态”不是“随便改”

动态配置的意思是运行时可以调整，但这不代表任何东西都应该动态化。

适合动态配置：

```text
改错了影响可控，或者需要快速开关。
```

例如：

```php
<?php

$showBanner = g_config('site', 'show_banner', true);
$enableCoupon = g_config('site', 'show_coupon', false);
```

不适合普通动态配置：

```text
改错了会导致资金、安全、数据一致性事故。
```

例如：

- 数据库连接地址
- 用户权限核心规则
- 订单金额核心公式
- 明文密钥
- 表结构定义

小白重点：动态配置是为了提高业务灵活性，不是为了把系统变成“所有规则都能在线乱改”。

---

### 1.12 第三步：理解为什么需要缓存

假设每次执行：

```php
<?php

g_config('site', 'show_coupon', false);
```

都去远程配置中心请求一次，会发生什么？

1. 接口变慢。
2. 配置中心压力很大。
3. 网络抖动会影响业务接口。
4. 配置中心挂了，业务也可能挂。

所以企业项目通常会有缓存：

```text
远程配置中心保存最新配置
  ↓
同步到本地缓存 / Redis / 文件缓存
  ↓
g_config() 优先读取缓存
  ↓
业务代码快速拿到配置
```

缓存不是为了“偷懒”，而是为了性能和稳定性。

---

### 1.13 第四步：理解 default 的兜底顺序

一个简化的读取流程可以这样理解：

```text
先读本地缓存
  ↓ 找到了：返回缓存值
没找到：读远程配置
  ↓ 找到了：返回远程值，并可能写入缓存
还没找到或失败：返回 default
```

伪代码：

```php
<?php

function g_config(string $module, string $key, mixed $default = null): mixed
{
    $cacheValue = readFromCache($module, $key);

    if ($cacheValue !== null) {
        return $cacheValue;
    }

    $remoteValue = readFromRemoteConfig($module, $key);

    if ($remoteValue !== null) {
        saveToCache($module, $key, $remoteValue);
        return $remoteValue;
    }

    return $default;
}
```

注意：真实项目不一定完全这样写，但你先掌握思路。

### 1.14 第五步：用场景判断配置类型

请按下面步骤判断一个配置该放哪里。

#### 场景 1：数据库地址

```text
DB_HOST=127.0.0.1
```

判断：静态配置 / 环境变量。

原因：数据库地址属于基础设施，改错会导致系统连不上数据库，不应该让运营后台随便改。

#### 场景 2：首页 banner

```text
site.banner = [...]
```

判断：动态配置。

原因：首页 banner 经常由运营调整，改错通常只是展示问题，适合配置化。

#### 场景 3：支付渠道开关

```text
pay.enable_stripe = false
```

判断：动态配置，但属于高风险配置。

原因：支付渠道异常时需要快速关闭，但默认值和权限控制要谨慎。

#### 场景 4：APP_ENV

```text
APP_ENV=production
```

判断：环境变量。

原因：它决定应用运行环境，不应该运行时随便变。

---

### 1.15 配置中心故障时怎么办？

系统设计时必须考虑：配置中心可能不可用。

常见兜底策略：

| 场景 | 推荐兜底 |
|---|---|
| 首页 banner 读取失败 | 返回空数组或默认 banner |
| 支付渠道读取失败 | 默认关闭相关支付渠道 |
| 客服电话读取失败 | 使用固定默认客服电话或提示文案 |
| 风控阈值读取失败 | 使用保守阈值，避免放大风险 |

小白重点：兜底不是“随便给一个值”，而是“配置失败时系统进入更安全的状态”。

---

### 1.16 今日易错点

| 易错点 | 正确理解 |
|---|---|
| 以为配置中心可以放所有配置 | 只适合需要动态调整的业务配置 |
| 以为动态配置一定更高级 | 过度动态化会让系统不可控 |
| 忽略缓存 | 没有缓存会影响性能和稳定性 |
| default 随便写 | 默认值要按业务风险设计 |
| 把环境变量和业务配置混在一起 | 环境变量偏基础设施，业务配置偏运营/业务行为 |

## 4. JS/Node.js 类比
