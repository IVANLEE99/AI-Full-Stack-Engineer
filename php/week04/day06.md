## 1. 学习内容

### 1.1 为什么需要配置项清单？

如果项目里到处都有：

```php
<?php

g_config('site', 'show_coupon', false);
g_config('pay', 'enable_stripe', false);
g_config('order', 'max_amount', 10000);
```

但没有统一清单，团队会遇到问题：

- 不知道项目到底有哪些配置
- 不知道配置默认值是什么
- 不知道改了配置会影响哪个页面
- 不知道哪些配置有资金或安全风险
- 新同事读项目很困难
- 线上排查问题时不知道该查哪个配置

配置项清单就是为了解决这些问题。

---

### 1.2 配置项清单应该包含哪些字段？

最小清单建议包含：

| 字段 | 含义 |
|---|---|
| module | 配置所属模块 |
| key | 配置项名称 |
| default | 默认值 |
| 类型 | bool/string/int/array 等 |
| 含义 | 这个配置控制什么 |
| 影响范围 | 影响哪个页面/接口/业务 |
| 风险等级 | 改错后影响多大 |
| 负责人 | 谁最了解这个配置 |

示例：

| module | key | default | 类型 | 含义 | 影响范围 | 风险 |
|---|---|---|---|---|---|---|
| `site` | `show_coupon` | `false` | bool | 是否展示优惠券入口 | 首页/商品详情 | 中 |
| `pay` | `enable_stripe` | `false` | bool | 是否开启 Stripe 支付 | 支付页 | 高 |
| `order` | `max_amount` | `10000` | int | 最大下单金额 | 下单接口 | 高 |

---

### 1.3 如何从代码里找配置项？

最直接的方法是搜索：

```text
g_config(
```

每找到一处，就记录：

```php
<?php

$enableStripe = g_config('pay', 'enable_stripe', false);
```

拆解成：

| 字段 | 值 |
|---|---|
| module | `pay` |
| key | `enable_stripe` |
| default | `false` |
| 影响 | 是否展示/允许 Stripe 支付 |
| 风险 | 高 |

如果项目用常量：

```php
<?php

g_config(ConfigHelper::MODULE_PAY, 'enable_stripe', false);
```

你还要跳到 `ConfigHelper` 看：

```php
<?php

public const MODULE_PAY = 'pay';
```

小白重点：不要只复制代码，要翻译成业务含义。

---

### 1.4 如何判断风险等级？

可以先用三级风险：

| 风险 | 判断标准 | 示例 |
|---|---|---|
| 高 | 影响资金、安全、订单、权限 | 支付开关、退款开关、最大金额 |
| 中 | 影响核心体验或重要展示 | 优惠券入口、活动开关、库存展示 |
| 低 | 影响文案、非核心展示 | 客服电话、提示语、banner 文案 |

示例判断：

```php
<?php

$allowRefund = g_config('order', 'allow_refund', false);
```

这属于高风险，因为退款涉及资金。

```php
<?php

$noticeText = g_config('site', 'notice_text', '欢迎光临');
```

这通常是低风险，因为只是展示文案。

---

### 1.5 如何判断默认值是否安全？

默认值不是随便填的。

判断原则：

```text
配置读取失败时，系统应该进入更安全、更可控的状态。
```

例如：

| 配置 | 不推荐默认值 | 推荐默认值 | 原因 |
|---|---|---|---|
| 支付开关 | `true` | `false` | 读取失败时不要误开放支付 |
| 退款开关 | `true` | `false` | 避免资金风险 |
| 首页 banner | 报错 | `[]` | 展示为空比页面崩溃好 |
| 客服电话 | 空字符串 | 默认客服电话/提示文案 | 保证用户有入口 |

你要在清单里写出默认值是否合理。

### 1.6 配置项清单模板

今天最终建议输出这个表：

| module | key | default | 类型 | 含义 | 影响范围 | 风险 | 默认值是否安全 | 备注 |
|---|---|---|---|---|---|---|---|---|
| `site` | `show_coupon` | `false` | bool | 是否展示优惠券入口 | 首页/商品详情 | 中 | 是 | 影响运营入口 |
| `pay` | `enable_stripe` | `false` | bool | 是否开启 Stripe | 支付页 | 高 | 是 | 支付渠道异常时关闭 |
| `order` | `max_amount` | `10000` | int | 最大下单金额 | 下单接口 | 高 | 需确认 | 要看业务客单价 |

如果配置太多，可以先做 10 个，确保质量。

### 1.7 Node.js 类比：feature flags 文档

Node 项目里可能会有 feature flags：

```js
const enabled = await flags.get('checkout.enable_new_pay', false);
```

团队通常也需要文档记录：

| flag | default | owner | risk | description |
|---|---|---|---|---|
| `checkout.enable_new_pay` | `false` | payment team | high | 新支付流程开关 |

PHP 的配置项清单本质类似。

---

### 1.8 小白实操：配置项清单就是“开关说明书”

如果项目里有很多配置，但没有清单，就像家里有一排电闸却没有标签：

```text
你不知道哪个开关控制客厅，哪个控制厨房，哪个一关会让冰箱断电。
```

配置项清单就是给每个开关贴标签：

```text
这个配置叫什么？
默认值是什么？
影响哪个页面？
改错会不会影响支付/订单/权限？
谁负责？
```

所以今天不是简单填表，而是在训练工程管理意识。

### 1.9 第一步：从代码里拆配置

看到一行：

```php
<?php

$enableStripe = g_config('pay', 'enable_stripe', false);
```

你要拆成：

| 字段 | 值 |
|---|---|
| module | `pay` |
| key | `enable_stripe` |
| default | `false` |
| 类型 | bool |
| 含义 | 是否开启 Stripe 支付 |
| 影响范围 | 支付页/下单支付流程 |
| 风险 | 高 |

小白重点：不要只抄 `pay`、`enable_stripe`、`false`，一定要翻译成业务语言。

---

### 1.10 第二步：判断类型

配置值常见类型：

| 类型 | 示例 | 说明 |
|---|---|---|
| bool | `true` / `false` | 开关类配置 |
| string | `'默认商城'` | 文案、名称、URL |
| int | `10000` | 数量、金额、阈值 |
| array | `[]` | banner 列表、导航列表 |
| float | `0.15` | 比例、费率，需谨慎 |

判断类型时，优先看默认值：

```php
<?php

g_config('site', 'show_coupon', false);   // bool
g_config('site', 'name', '默认商城');      // string
g_config('order', 'max_amount', 10000);   // int
g_config('site', 'banners', []);          // array
```

但真实项目里配置中心可能把值存成字符串，所以读取后是否需要类型转换，也要留意。

---

### 1.11 第三步：判断影响范围

影响范围要写得具体，不要只写“影响业务”。

不推荐：

```text
影响业务
影响页面
影响系统
```

推荐：

```text
首页优惠券入口是否展示
支付页是否展示 Stripe 支付方式
下单接口是否允许超过指定金额
售后页客服电话展示
```

示例：

| 配置 | 不清楚的影响写法 | 更好的影响写法 |
|---|---|---|
| `site.show_coupon` | 影响前端 | 首页/商品详情是否展示优惠券入口 |
| `pay.enable_stripe` | 影响支付 | 支付页是否展示并允许 Stripe 支付 |
| `order.max_amount` | 影响订单 | 下单接口是否拒绝超额订单 |

---

### 1.12 第四步：判断风险等级

先用三档风险即可。

#### 高风险

改错会影响资金、安全、订单、权限、数据一致性。

示例：

```php
<?php

g_config('pay', 'enable_stripe', false);
g_config('order', 'allow_refund', false);
g_config('order', 'max_amount', 10000);
```

#### 中风险

改错会影响核心体验、转化率、重要运营入口。

示例：

```php
<?php

g_config('site', 'show_coupon', false);
g_config('site', 'activity_enabled', false);
```

#### 低风险

改错主要影响文案或非核心展示。

示例：

```php
<?php

g_config('site', 'notice_text', '欢迎光临');
g_config('site', 'support_phone', '400-000-0000');
```

注意：低风险不等于完全没风险。例如客服电话错了，也会影响用户体验。

---

### 1.13 第五步：判断默认值是否安全

默认值检查要问：

```text
如果配置读取失败，使用这个 default 会不会让系统进入危险状态？
```

示例：

| 配置 | default | 是否安全 | 原因 |
|---|---|---|---|
| `pay.enable_stripe` | `false` | 是 | 读取失败时不开放支付渠道 |
| `order.allow_refund` | `true` | 否 | 读取失败时可能错误开放退款 |
| `site.banners` | `[]` | 是 | banner 为空比页面崩溃好 |
| `order.max_amount` | `999999999` | 需谨慎 | 过大可能失去风控意义 |

小白重点：默认值不是为了“看起来有值”，而是为了“失败时更安全”。

示例清单：

|---|---|---|---|---|---|---|---|---|
| `site` | `name` | `默认商城` | string | 站点名称 | 页面标题/分享标题 | 低 | 是 | 文案类配置 |

### 1.16 第八步：真实项目中怎么维护清单？

配置项清单不是写一次就结束。

建议规则：

1. 新增 `g_config()` 时，同步更新配置清单。
2. 高风险配置必须写明负责人和回滚方式。
3. 默认值变更要经过 code review。
4. 删除配置前先确认没有代码引用。
5. 线上问题复盘时补充配置风险说明。

如果团队成熟，可以把配置清单放到：

- 项目文档
- 后台配置管理说明
- Wiki
- 接口文档旁边
- 配置中心元数据

### 1.17 今日易错点

| 易错点 | 正确理解 |
|---|---|
| 只记录 module/key/default | 还要记录含义、影响、风险 |
| 风险等级全写低 | 涉及支付、订单、权限通常是高风险 |
| 默认值只看能不能运行 | 还要看失败时是否安全 |
| 清单写完不维护 | 配置新增/修改时要同步更新 |
| 影响范围写太泛 | 要写具体页面、接口或业务流程 |

## 8. 今日自测题

### 8.1 配置项清单最少应该包含哪些字段？

参考答案：至少包含 module、key、default、类型、含义、影响范围、风险等级；更完整时还可加上默认值是否安全和负责人。

### 8.2 为什么支付开关通常被判为高风险配置？

参考答案：因为它直接影响资金链路和用户支付行为，改错可能造成交易问题或资金风险，所以属于高风险。

### 8.3 为什么首页 banner 的默认值可以设为空数组 `[]`？

参考答案：因为配置读取失败时，返回空数组只是不展示 banner，比让页面报错崩溃更安全、更可控。

### 8.4 如何判断一个默认值是否安全？

参考答案：看配置读取失败、使用这个默认值时，系统是否会进入更安全、更可控的状态。会造成资金/安全/数据风险的默认值就不安全。

### 8.5 新增一个 `g_config()` 时，除了写代码还应该做什么？

参考答案：同步更新配置项清单，记录 module/key/default、含义、影响范围和风险等级，并评估默认值是否安全；高风险配置还要写明负责人和回滚方式。
