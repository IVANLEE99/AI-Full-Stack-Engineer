## 1. 学习内容

### 1.1 为什么需要配置项清单？

如果项目里到处都有：

```php
<?php

g_config('site', 'show_coupon', false);
g_config('pay', 'enable_stripe', false);
g_config('order', 'max_amount', 10000);
```

但没有统一清单，团队会遇到问题：

- 不知道项目到底有哪些配置
- 不知道配置默认值是什么
- 不知道改了配置会影响哪个页面
- 不知道哪些配置有资金或安全风险
- 新同事读项目很困难
- 线上排查问题时不知道该查哪个配置

配置项清单就是为了解决这些问题。

---

### 1.2 配置项清单应该包含哪些字段？

最小清单建议包含：

| 字段 | 含义 |
|---|---|
| module | 配置所属模块 |
| key | 配置项名称 |
| default | 默认值 |
| 类型 | bool/string/int/array 等 |
| 含义 | 这个配置控制什么 |
| 影响范围 | 影响哪个页面/接口/业务 |
| 风险等级 | 改错后影响多大 |
| 负责人 | 谁最了解这个配置 |

示例：

| module | key | default | 类型 | 含义 | 影响范围 | 风险 |
|---|---|---|---|---|---|---|
| `site` | `show_coupon` | `false` | bool | 是否展示优惠券入口 | 首页/商品详情 | 中 |
| `pay` | `enable_stripe` | `false` | bool | 是否开启 Stripe 支付 | 支付页 | 高 |
| `order` | `max_amount` | `10000` | int | 最大下单金额 | 下单接口 | 高 |

---

### 1.3 如何从代码里找配置项？

最直接的方法是搜索：

```text
g_config(
```

每找到一处，就记录：

```php
<?php

$enableStripe = g_config('pay', 'enable_stripe', false);
```

拆解成：

| 字段 | 值 |
|---|---|
| module | `pay` |
| key | `enable_stripe` |
| default | `false` |
| 影响 | 是否展示/允许 Stripe 支付 |
| 风险 | 高 |

如果项目用常量：

```php
<?php

g_config(ConfigHelper::MODULE_PAY, 'enable_stripe', false);
```

你还要跳到 `ConfigHelper` 看：

```php
<?php

public const MODULE_PAY = 'pay';
```

小白重点：不要只复制代码，要翻译成业务含义。

---

### 1.4 如何判断风险等级？

可以先用三级风险：

| 风险 | 判断标准 | 示例 |
|---|---|---|
| 高 | 影响资金、安全、订单、权限 | 支付开关、退款开关、最大金额 |
| 中 | 影响核心体验或重要展示 | 优惠券入口、活动开关、库存展示 |
| 低 | 影响文案、非核心展示 | 客服电话、提示语、banner 文案 |

示例判断：

```php
<?php

$allowRefund = g_config('order', 'allow_refund', false);
```

这属于高风险，因为退款涉及资金。

```php
<?php

$noticeText = g_config('site', 'notice_text', '欢迎光临');
```

这通常是低风险，因为只是展示文案。

---

### 1.5 如何判断默认值是否安全？

默认值不是随便填的。

判断原则：

```text
配置读取失败时，系统应该进入更安全、更可控的状态。
```

例如：

| 配置 | 不推荐默认值 | 推荐默认值 | 原因 |
|---|---|---|---|
| 支付开关 | `true` | `false` | 读取失败时不要误开放支付 |
| 退款开关 | `true` | `false` | 避免资金风险 |
| 首页 banner | 报错 | `[]` | 展示为空比页面崩溃好 |
| 客服电话 | 空字符串 | 默认客服电话/提示文案 | 保证用户有入口 |

你要在清单里写出默认值是否合理。

---

### 1.6 配置项清单模板

今天最终建议输出这个表：

| module | key | default | 类型 | 含义 | 影响范围 | 风险 | 默认值是否安全 | 备注 |
|---|---|---|---|---|---|---|---|---|
| `site` | `show_coupon` | `false` | bool | 是否展示优惠券入口 | 首页/商品详情 | 中 | 是 | 影响运营入口 |
| `pay` | `enable_stripe` | `false` | bool | 是否开启 Stripe | 支付页 | 高 | 是 | 支付渠道异常时关闭 |
| `order` | `max_amount` | `10000` | int | 最大下单金额 | 下单接口 | 高 | 需确认 | 要看业务客单价 |

如果配置太多，可以先做 10 个，确保质量。

---

### 1.7 Node.js 类比：feature flags 文档

Node 项目里可能会有 feature flags：

```js
const enabled = await flags.get('checkout.enable_new_pay', false);
```

团队通常也需要文档记录：

| flag | default | owner | risk | description |
|---|---|---|---|---|
| `checkout.enable_new_pay` | `false` | payment team | high | 新支付流程开关 |

PHP 的配置项清单本质类似。
