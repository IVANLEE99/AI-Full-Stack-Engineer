### 1.3 必须掌握的异步处理风险

| 风险 | 你要能解释什么 |
|---|---|
| Webhook 伪造 | 必须验签，不能只靠 URL 隐藏 |
| Webhook 超时 | 处理太慢会导致支付平台重试 |
| 消息丢失 | 生产者发送失败需要日志、重试、确认机制 |
| 重复消息 | MQ 可能重投，消费者必须幂等 |
| 消费失败 | 需要重试、失败记录、死信队列 |
| 死信堆积 | 需要监控和人工处理机制 |
| 事件字段过大或敏感 | 消息不应携带密钥、卡信息或完整敏感响应 |
| 状态重复更新 | 已支付订单不能被重复消息重复处理或状态回退 |

### 1.4 Webhook → MQ → Consumer 主线

请尝试口述：

```text
支付平台向 outer Webhook 发起回调。
后端读取 raw body 和签名头，用 webhook secret 验签。
验签成功后解析 event type 和 event_id。
后端用 event_id 做幂等检查，并校验订单号、支付单、金额和币种。
确认支付成功后更新支付单状态，投递 payment.succeeded MQ 事件。
Webhook 快速返回 2xx 给支付平台。
RabbitMQ 根据 exchange 和 route_key 把消息投递到队列。
订单消费者、通知消费者、权益消费者异步处理消息。
消费者处理前检查 event_id 或业务唯一键，保证重复消息不会重复更新订单或重复发权益。
消费失败时进入重试，超过次数进入死信队列，等待排查或补偿。
```

如果你能完整口述，说明 Week 08 主线已经建立。

### 1.5 事件驱动架构入门

事件驱动不是简单把函数改成 MQ。

事件驱动的关键是：

```text
一个服务发生业务事实后，发布事件；其他服务订阅事件并做自己的事情。
```

例如：

```text
payment.succeeded
```

代表一个已经发生的事实：

```text
支付已经成功。
```

消费者可以有多个：

| 消费者 | 处理内容 |
|---|---|
| 订单消费者 | 更新订单为已支付/待发货 |
| 通知消费者 | 发送短信/邮件/App push |
| 权益消费者 | 发放积分、会员权益、课程权限 |
| 数据消费者 | 写统计、埋点、报表 |
| 审计消费者 | 写审计日志 |

发布者不需要知道所有消费者的细节，这就是解耦。
