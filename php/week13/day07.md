### 1.2 端到端回归测试

一次性跑完所有关键路径：

```bash
# 1. 健康检查
curl -s http://127.0.0.1:8000/health

# 2. 缺 Key → 期望 401
curl -s -X POST http://127.0.0.1:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o-mini","messages":[{"role":"user","content":"hi"}]}'

# 3. 非法请求（空 messages）→ 期望 422
curl -s -X POST http://127.0.0.1:8000/chat \
  -H "Authorization: Bearer $GATEWAY_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o-mini","messages":[]}'

# 4. 正常 OpenAI 调用
curl -s -X POST http://127.0.0.1:8000/chat \
  -H "Authorization: Bearer $GATEWAY_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o-mini","messages":[{"role":"user","content":"用一句话介绍你自己"}]}'

# 5. 正常 Claude 调用
curl -s -X POST http://127.0.0.1:8000/chat \
  -H "Authorization: Bearer $GATEWAY_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"claude-3-5-sonnet","messages":[{"role":"user","content":"用一句话介绍你自己"}]}'
```

小白重点：回归测试就是「把之前分散验证过的点，一次性再跑一遍」，确保后期的整理没把之前跑通的东西改坏。

### 1.4 MCP 是什么

MCP（Model Context Protocol，模型上下文协议）是一套标准协议，规定「AI 模型/客户端如何统一地发现并调用外部工具、读取外部数据源」。

一句话对比：

| | 本周的 LLM Gateway | 下周的 MCP |
|---|---|---|
| 方向 | 应用 → 统一接口 → 多个模型 | 模型/客户端 → 统一协议 → 多个工具/数据 |
| 统一的是 | 「不同厂商模型」的调用 | 「模型如何用工具」的方式 |
| 类比 | 多支付渠道的统一收银台 | 多设备的统一 USB 接口 |

小白重点：网关解决「我怎么统一地调不同模型」，MCP 解决「模型怎么统一地调我的工具」。两者互补，都是「加一层标准中间协议来消除碎片化」，这跟你在 PHP 阶段学的「接口/抽象层」是同一种工程思想。

---

### 1.5 MCP 预习要点

下周会展开，今天先建立概念：

```text
- MCP 有三种核心角色：
  Host（宿主，如 AI 客户端）
  Client（连接器）
  Server（提供工具/资源的一方，通常是你要写的）
- MCP Server 对外暴露三类能力：
  Tools（可被模型调用的函数）
  Resources（可读取的数据）
  Prompts（预置提示模板）
- 通信基于 JSON-RPC 消息
```

小白重点：不用今天就搞懂细节。记住「MCP Server 是你写的、对外提供 Tools/Resources/Prompts」这一句，下周上手就快。

## 4. JS/Node.js 类比

| 概念 | Node 类比 | 说明 |
|---|---|---|
| LLM Gateway | Express 里的统一 API 层 | 对外一个接口，对内多实现 |
| MCP Server | 实现某个标准接口的插件 | 遵守协议就能被任意客户端接入 |
| MCP Tools | 注册的可调用函数 | 类似暴露给外部的 RPC 方法 |
| JSON-RPC | 约定的消息格式 | 类似统一的请求/响应约定 |
| 加中间协议 | 中间件/适配器模式 | 用一层标准消除碎片化 |
