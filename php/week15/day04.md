## 今日目标

让客服 Agent 支持三种场景：查订单、查商品、常见问题（FAQ）兜底。核心是「意图识别 + 路由」——判断用户想干什么，然后走对应的处理逻辑。

今天你要真正掌握这一句话：

> 多场景路由 = 先做意图分类（用户想查订单？查商品？还是问常识？），再把请求分发到对应的工具或回答逻辑；识别不出来的，统一走 FAQ / 兜底，绝不让请求「掉地上」。

## 1. 学习内容

### 1.1 什么是意图（Intent）

意图就是「用户这句话到底想干嘛」。同一件事，用户有一百种说法：

```text
"我的快递到哪了"        → 意图：查订单
"订单 A123 发货没"      → 意图：查订单
"iPhone 还有货吗"       → 意图：查商品
"这个多少钱"            → 意图：查商品
"你们支持七天无理由吗"   → 意图：FAQ（退换货政策）
```

小白重点：

> 意图识别就是把「五花八门的自然语言」归类到「有限的几个处理分支」。分类对了，才能路由对。

---

### 1.2 两种意图识别方式

方式一：关键词规则（简单但脆弱）

```js
function detectIntentByKeyword(text) {
  if (/订单|物流|快递|发货/.test(text)) return "query_order";
  if (/多少钱|价格|库存|有货|详情/.test(text)) return "query_product";
  return "faq"; // 兜底
}
```

方式二：让模型判断（更聪明，能理解语义）

```js
// 让模型输出一个意图标签
const messages = [
  { role: "system", content: "你是意图分类器。只输出以下之一：query_order / query_product / faq" },
  { role: "user", content: "我买的东西怎么还没到" },
];
// 模型输出：query_order
```

对比表：

| 方式 | 优点 | 缺点 | 适用 |
|---|---|---|---|
| 关键词规则 | 快、免费、可控 | 换个说法就失效，维护累 | 简单场景/兜底 |
| 模型判断 | 懂语义、抗改写 | 慢一点、有成本 | 复杂/真实场景 |

小白重点：

> 别以为规则法很土——它常被当作「快速兜底」和「模型判断的补充」，两者可以配合用。

---

### 1.3 工业界更爱：让模型通过 Tool Calling 自己选

其实前两天学的 Tool Calling 本身就是一种意图路由。你把工具都注册好，模型会根据用户的话自己决定「该调哪个工具」——这就等于它做了意图识别。

```js
const tools = [
  { type: "function", function: { name: "get_order",   /* 查订单 */ } },
  { type: "function", function: { name: "get_product", /* 查商品 */ } },
];
// 用户说"我的快递到哪了" → 模型自己选 get_order
// 用户说"这手机多少钱"   → 模型自己选 get_product
// 用户说"能七天无理由退吗" → 模型不调工具，直接用 FAQ 知识回答
```

小白重点：

> 有了 Tool Calling，「意图识别」和「路由」很大程度上被模型接管了。你要做的是：把工具描述写清楚（模型才选得准），以及处理「模型没选任何工具」的情况（走 FAQ / 直接回答）。

---

### 1.4 设计三种意图

我们的客服 Agent 支持三种：

| 意图 | 触发 | 处理方式 |
|---|---|---|
| query_order | 问订单、物流、发货 | 调 get_order 工具 |
| query_product | 问商品价格、库存、详情 | 调 get_product 工具 |
| faq | 问政策、时效等常识 | 用 System Prompt 里的知识直接回答 |

新增一个查商品的工具（延续昨天的写法）：

```js
const getProductTool = {
  type: "function",
  function: {
    name: "get_product",
    description: "根据商品名称或 ID 查询价格、库存和简介",
    parameters: {
      type: "object",
      properties: {
        keyword: { type: "string", description: "商品名称或 ID，例如 'iPhone' 或 'P1001'" },
      },
      required: ["keyword"],
    },
  },
};
```

---

### 1.5 用路由函数分发

真正执行工具时，用一个路由把工具名映射到函数（策略模式）：

```js
// 工具名 → 实际执行函数
const toolRouter = {
  get_order: (args) => queryOrder(args.order_no),
  get_product: (args) => queryProduct(args.keyword),
};

function executeTool(name, args) {
  const handler = toolRouter[name];
  if (!handler) {
    return { error: `未知工具：${name}` }; // 防止路由到不存在的分支
  }
  return handler(args);
}
```

小白重点：

> `toolRouter` 就是一张「意图 → 处理逻辑」的分发表。加新场景时，只需往表里加一行，不用改一大堆 if/else。这就是策略模式的好处。

---

### 1.6 FAQ 兜底：接住没匹配上的请求

当模型没调任何工具（说明它判断这是常识问题），就走 FAQ。而 FAQ 知识可以直接写进 System Prompt：

```js
const systemPrompt = `
你是优选商城客服小优。
（角色、能力、边界... 见 Day 03）

# 常见问题（FAQ 知识库）
- 退换货：支持签收后 7 天无理由退换（生鲜、定制商品除外）。
- 配送时效：一般 48 小时内发货，偏远地区 3~5 天。
- 发票：下单后可在订单详情页申请电子发票。

当用户问到以上问题时，直接用这些知识回答，不需要调用工具。
当用户问题既不属于订单/商品，FAQ 里也没有时，礼貌说明并建议联系人工客服。
`.trim();
```

小白重点：

> 「兜底」的意思是：无论用户说什么，都有一个分支接住，不会出现「Agent 卡住不回话」。查不到、没匹配上、超范围——统统有对应的话术。

---

### 1.7 完整路由流程图（文字版）

```text
用户消息
   │
   ▼
模型（带 tools + FAQ 的 system prompt）
   │
   ├── 选择了 get_order   → 执行查订单 → 回喂 → 模型总结
   ├── 选择了 get_product → 执行查商品 → 回喂 → 模型总结
   └── 没选工具           → 用 FAQ 知识直接回答
                              └── FAQ 也答不了 → 引导人工客服（最终兜底）
```

## 3. 练习任务

### 练习 1：加上查商品工具

在昨天的 Agent 基础上，新增 `get_product` 工具和 `queryProduct` 模拟函数：

```js
// 模拟商品数据库
const productDB = {
  P1001: { name: "优选蓝牙耳机", price: 199, stock: 42 },
  P1002: { name: "优选保温杯", price: 89, stock: 0 },
};

function queryProduct(keyword) {
  // 简单按 ID 或名称模糊匹配
  const found = Object.entries(productDB).find(
    ([id, p]) => id === keyword || p.name.includes(keyword)
  );
  if (!found) return { error: "未找到该商品" };
  const [id, p] = found;
  return { id, ...p, in_stock: p.stock > 0 };
}
```

## 4. JS/Node.js 类比

| 路由概念 | Node.js / JS 类比 | 说明 |
|---|---|---|
| 意图识别 | 请求解析 / 路由匹配 | 判断这个请求该走哪 |
| toolRouter 分发表 | Express 路由表 / 策略模式 map | name → handler |
| 加新意图 | 注册新路由 | 往表里加一行 |
| FAQ 兜底 | 404 / default 路由 | 没匹配上的统一处理 |
| 反问澄清 | 参数校验失败要求补参 | 信息不够先要参数 |

一句话类比：

> 多场景路由 ≈ Express 里的路由表 + 一个 default（404）兜底：每种意图对应一个 handler，没匹配上的都交给兜底，绝不让请求无人处理。

## 8. 今日自测题

### 8.1 什么是意图路由？

参考答案：

> ✅ 先判断用户这句话想干什么（意图分类），再把请求分发到对应的工具或回答逻辑（路由）。核心是「五花八门的说法 → 有限的处理分支」。

---

### 8.2 关键词规则和让模型判断，各自的优缺点？

参考答案：

> ✅ 关键词规则快、免费、可控，但换个说法就失效、维护累；模型判断懂语义、抗改写，但慢一点、有成本。真实项目常把两者配合用，规则做兜底。

---

### 8.3 有了 Tool Calling，还需要单独写意图识别吗？

参考答案：

> ✅ 很大程度上不需要——模型根据工具描述自己就会选工具，相当于做了意图识别。你要做的是把工具描述写清楚，并处理「模型没选任何工具」（走 FAQ）和模糊输入（反问澄清）的情况。

---

### 8.4 toolRouter 分发表的好处是什么？

参考答案：

> ✅ 它是「工具名 → 处理函数」的映射表（策略模式）。加新场景只需加一行，不用改一堆 if/else，可读性和可扩展性都好。

---

### 8.5 为什么一定要有 FAQ 兜底和最终兜底？

参考答案：

> ✅ 保证任何用户输入都有分支接住，不会「掉地上」导致 Agent 卡住不回话。FAQ 接住常识问题，最终兜底（转人工）接住谁都答不了的问题。
