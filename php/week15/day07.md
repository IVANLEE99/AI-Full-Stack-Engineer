### 1.4 while 循环写 Agent 的痛点（引出 LangGraph）

本周的 Agent 是一根 while 循环 + if 判断。简单场景没问题，但业务一复杂就开始难受：

```text
想象客服 Agent 要处理这样的流程：
  用户投诉 → 先判断情绪 → 如果很生气，走"安抚+优先转人工"分支
                        → 如果一般，走"查订单+尝试自助解决"分支
                            → 解决了 → 结束
                            → 没解决 → 转人工，并带上已查到的信息
```

如果用 while + if/else 硬写，代码会变成这样：

```js
// 伪代码：流程一复杂，if 就开始嵌套地狱
if (mood === "angry") {
  escalate();
} else {
  const order = await getOrder();
  if (order.canSelfServe) {
    const result = await selfServe();
    if (!result.solved) {
      escalate(withContext);
    }
  } else {
    escalate(withContext);
  }
}
```

痛点很明显：

| 痛点 | 表现 |
|---|---|
| 流程看不清 | 分支藏在层层 if 里，读代码猜不出整体流程 |
| 状态难管理 | "已查到的订单""用户情绪"等状态在函数间乱传 |
| 难加节点 | 想在中间插一步"记录工单"，要改好几处 |
| 无法可视化 | 没法一眼看出"从哪到哪、有几条路径" |

小白重点：

> while + if 写单个 Agent 够用，但当流程有多分支、多回退、多角色协作时，就需要一种能「把流程画出来」的工具——这就是下周要学的 LangGraph 要解决的问题。

---

### 1.5 预习 LangGraph：把流程画成一张图

LangGraph 是一个把 Agent/工作流建模成**状态图（State Graph）**的框架。核心就三个概念：

```text
节点（Node）：一个处理步骤，比如"查订单""判断情绪""转人工"
边（Edge）  ：节点之间的连线，决定下一步走哪个节点
状态（State）：在整张图里流动、被各节点读写的共享数据
```

用一张图对比本周的 while Agent 和 LangGraph：

```text
本周（while 循环）：              LangGraph（状态图）：

  ┌──────────────┐               ┌────────┐
  │  while 循环   │               │ 判断情绪 │
  │  if / else   │               └───┬────┘
  │  藏着所有分支 │            生气 ↙    ↘ 一般
  └──────────────┘         ┌──────┐   ┌────────┐
   流程要读代码才懂          │转人工 │   │ 查订单  │
                            └──────┘   └───┬────┘
                                    解决↙    ↘没解决
                                 ┌────┐   ┌──────┐
                                 │结束 │   │转人工 │
                                 └────┘   └──────┘
                              流程一眼看清、可视化
```

对比表：

| 对比项 | 本周 while Agent | LangGraph |
|---|---|---|
| 流程表达 | 藏在 if/else 里 | 显式的节点 + 边 |
| 状态管理 | 手动传参 | 统一的 State 对象 |
| 分支/回退 | 嵌套 if | 条件边（conditional edge） |
| 可视化 | 无 | 能画出流程图 |
| 适合场景 | 单 Agent、流程简单 | 复杂编排、多分支、多角色 |

小白重点：

> 不要觉得 LangGraph 否定了本周所学。恰恰相反——LangGraph 的每个「节点」内部，往往就是一个本周这样的「调模型/调工具」逻辑。本周学的是「一个 Agent 怎么工作」，下周学的是「多个步骤怎么编排成流程」。

## 4. JS/Node.js 类比

| 概念 | Node.js / JS 类比 | 说明 |
|---|---|---|
| 本周 while Agent | 一个大 handler 里的 while + if | 简单直接，但复杂了难维护 |
| LangGraph 节点 | 一个个独立的中间件函数 | 每个只干一件事 |
| LangGraph 边 | 中间件的 next() / 路由跳转 | 决定下一步去哪 |
| 条件边 | if 决定调哪个 next | 按状态选分支 |
| State | req 对象 / 全局 context | 在整条链路里流动的数据 |
| 可视化流程图 | 架构图 / 时序图 | 一眼看清整体走向 |

一句话类比：

> 从本周到下周，就像从「把所有逻辑写在一个 Express handler 里」升级到「用中间件链 + 路由把流程拆成清晰的节点」：功能没变，但流程从「藏在代码里」变成「画得出来、改得动」。这和你在 PHP 阶段从「一个大方法」重构成「Controller → Service → 多个清晰步骤」是同一种进步。
