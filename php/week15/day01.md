## 1. 学习内容

### 1.1 先分清楚：「调用 LLM」和「Agent」不是一回事

很多小白会把这两件事混在一起。我们先用一张对比表分清楚。

普通「调用 LLM」：

```text
用户问题 → 发给大模型 → 大模型直接回一句话 → 结束
```

Agent：

```text
用户问题 → 发给大模型 → 大模型说「我要先查订单」
        → 后端真的去查订单 → 把订单结果再发给大模型
        → 大模型基于订单结果回答 → 结束
```

对比表：

| 对比项 | 普通调用 LLM | Agent |
|---|---|---|
| 能不能查真实数据 | 不能，只能靠模型记忆 | 能，靠调用工具查数据库/API |
| 交互次数 | 1 次请求 1 次回答 | 可能来回多次 |
| 谁决定要不要查数据 | 你在代码里写死 | 大模型自己决定 |
| 典型场景 | 翻译、润色、写文案 | 客服、查订单、执行操作 |

小白重点：

> Agent 的核心不是「更聪明的模型」，而是「让模型能调用你后端的函数」。模型本身不能连数据库，是你的后端在执行工具，模型只是「下指令」。

---

### 1.2 Agent 的三个核心角色

一个最小可用的 Agent，只需要三个角色：

```text
1. LLM（大脑）        ：负责理解意图、决定调用哪个工具
2. Tool（工具/手脚）  ：真正干活的函数，比如查订单、查商品
3. Orchestrator（编排器）：你的后端代码，负责把 1 和 2 串起来的那个 while 循环
```

用一张图先建立整体印象：

```text
        ┌─────────────┐
用户 ──▶ │ Orchestrator │ ──▶ 把「用户消息 + 工具清单」发给 LLM
        │  (你的后端)   │
        └──────┬──────┘
               │  LLM 返回：请调用 getOrder(order_id="A123")
               ▼
        ┌─────────────┐
        │    Tool      │  真正去数据库查订单
        │  getOrder()  │
        └──────┬──────┘
               │  返回：{status:"已发货", ...}
               ▼
        Orchestrator 把结果再发回 LLM ──▶ LLM 生成最终回答 ──▶ 返回给用户
```

小白重点：

> 你写的后端代码（Orchestrator）才是 Agent 的「主程序」。LLM 只是被你反复调用的一个「决策 API」。不要以为模型会自己连你的数据库——是你的代码在中间搬运。

---

### 1.3 Tool Calling / Function Calling 到底是什么

Tool Calling（OpenAI 早期叫 Function Calling，Anthropic 叫 Tool Use）是本周最核心的概念。

它的本质是：

> 你在调用大模型时，除了发消息，还额外附带一份「工具说明书」（用 JSON 描述每个工具叫什么、干什么、需要什么参数）。模型如果觉得需要用工具，就不会直接回答，而是返回一段结构化的「调用请求」，告诉你「调用哪个工具、参数填什么」。

一份工具定义（tools 参数）长这样，用 Node/JS 示例：

```js
// 这是你告诉大模型「你有哪些工具可以用」的清单
const tools = [
  {
    type: "function",
    function: {
      name: "get_order",              // 工具名字，模型会用这个名字点名
      description: "根据订单号查询订单状态、物流和金额", // 描述越清楚，模型越会用对
      parameters: {                   // 参数用 JSON Schema 描述
        type: "object",
        properties: {
          order_id: {
            type: "string",
            description: "订单号，例如 A123456",
          },
        },
        required: ["order_id"],
      },
    },
  },
];
```

当你带着这份 `tools` 去请求模型，模型可能不直接回答，而是返回：

```json
{
  "role": "assistant",
  "tool_calls": [
    {
      "id": "call_abc123",
      "type": "function",
      "function": {
        "name": "get_order",
        "arguments": "{\"order_id\": \"A123456\"}"
      }
    }
  ]
}
```

注意 `arguments` 是一个「字符串形式的 JSON」，你需要自己 `JSON.parse` 出来。

小白重点：

> 模型不会真的执行工具，它只会告诉你「我想调用 get_order，参数是 A123456」。真正执行 `get_order` 的是你的后端代码。执行完，你要把结果再发回模型。

---

### 1.4 一次完整的 Tool Calling 走 4 步

我们把上面拆成 4 步，这是理解 Agent 的关键：

```text
第 1 步：你 → 模型
  发送：[用户消息] + [工具清单 tools]

第 2 步：模型 → 你
  返回：tool_calls（我要调用 get_order，参数 order_id=A123456）

第 3 步：你（后端）执行工具
  真正跑 get_order("A123456") → 得到 {status:"已发货"}
  把这个结果作为一条「role: tool」的消息，追加到对话里

第 4 步：你 → 模型（再来一次）
  发送：[用户消息, 模型的 tool_calls, 工具执行结果]
  模型这次看到了订单数据，生成最终自然语言回答
```

关键点：**第 4 步是「再请求一次模型」**。这就是为什么 Agent 是个循环——你可能要来回好几次。

用一张时序表看：

| 轮次 | 谁在说话 | 内容 |
|---|---|---|
| 1 | 用户 | 我的订单 A123456 到哪了？ |
| 2 | 模型 | （tool_calls）请调用 get_order(order_id=A123456) |
| 3 | 后端 | （执行工具）返回 {status:"已发货", eta:"明天"} |
| 4 | 模型 | 您的订单 A123456 已发货，预计明天送达 |

---

### 1.5 Agent 循环（Agent Loop）为什么是 while

有时候一个问题需要调用多个工具。例如：

> 用户：帮我看看订单 A123 收货地址对不对，如果是北京的就查一下北京仓有没有货。

模型可能：

```text
第 1 轮：调用 get_order 拿收货地址
第 2 轮：发现是北京，调用 get_stock(city="北京") 查库存
第 3 轮：综合两个结果，给出最终回答
```

所以 Orchestrator 必须写成一个 while 循环：**只要模型还在要求调用工具，就继续执行工具并回喂；直到模型不再要求工具、直接给答案，才跳出循环。**

用 Node/JS 写出这个循环的核心结构：

```js
async function runAgent(userMessage) {
  // messages 是整个对话历史，会不断追加
  const messages = [
    { role: "system", content: "你是电商客服助手" },
    { role: "user", content: userMessage },
  ];

  // 安全上限：最多循环 5 次，防止模型无限调用工具
  for (let step = 0; step < 5; step++) {
    // 1. 带着工具清单请求模型
    const response = await callLLM({ messages, tools });
    const msg = response.choices[0].message;
    messages.push(msg); // 把模型这轮的回复也存进历史

    // 2. 模型没有要求调用工具 → 说明它直接回答了，结束循环
    if (!msg.tool_calls) {
      return msg.content; // 这就是最终答案
    }

    // 3. 模型要求调用工具 → 逐个执行，把结果追加回 messages
    for (const call of msg.tool_calls) {
      const args = JSON.parse(call.function.arguments);
      const result = await executeTool(call.function.name, args);
      messages.push({
        role: "tool",
        tool_call_id: call.id,
        content: JSON.stringify(result),
      });
    }
    // 循环继续：带着工具结果再问一次模型
  }

  return "抱歉，处理超时了，请稍后再试。"; // 兜底
}
```

小白重点：

> 记住这个循环的三个出口：
> 1. 模型直接回答（没有 tool_calls）→ 返回答案，结束
> 2. 模型要求调工具 → 执行、回喂、继续循环
> 3. 循环次数用完 → 兜底返回，防止死循环

### 1.6 客服 Agent 的整体架构图

现在把今天学的东西拼成一张完整的客服 Agent 架构图。这也是你今天要产出的图。

```text
┌──────────────────────────────────────────────────────────┐
│                         用户 (User)                        │
│                    "订单 A123 到哪了？"                     │
└───────────────────────────┬──────────────────────────────┘
                            │ HTTP 请求
                            ▼
┌──────────────────────────────────────────────────────────┐
│                  API 入口 (Controller)                     │
│              接收消息、鉴权、记录日志                        │
└───────────────────────────┬──────────────────────────────┘
                            ▼
┌──────────────────────────────────────────────────────────┐
│              Agent Orchestrator (核心 while 循环)          │
│                                                            │
│   ┌────────────┐   带 tools 请求    ┌──────────────────┐  │
│   │ 对话历史    │ ─────────────────▶ │      LLM         │  │
│   │ messages[] │ ◀───────────────── │ (决策：调哪个工具)│  │
│   └────────────┘   返回 tool_calls  └──────────────────┘  │
│         │                                                  │
│         │ 执行工具                                          │
│         ▼                                                  │
│   ┌──────────────────────────────────────────────┐        │
│   │              Tool 层 (真正干活)                 │        │
│   │  get_order()   get_product()   search_faq()    │        │
│   └──────┬──────────────┬───────────────┬─────────┘        │
│          ▼              ▼               ▼                  │
│      订单服务/DB     商品服务/DB      FAQ 知识库             │
└──────────────────────────────────────────────────────────┘
                            │ 最终自然语言回答
                            ▼
                          用户
```

架构图里必须体现的 5 个要素（画图时自检）：

| 要素 | 说明 |
|---|---|
| 用户入口 | 消息从哪进来 |
| Orchestrator 循环 | 那个反复调 LLM 的 while |
| LLM 决策 | 模型返回 tool_calls |
| Tool 层 | 每个工具背后连的是什么（DB/API/知识库） |
| 回喂闭环 | 工具结果要再发回 LLM |

## 4. JS/Node.js 类比

| Agent 概念 | Node.js / JS 类比 | 说明 |
|---|---|---|
| Agent Loop | `while` 循环里反复 `await fetch` | 只是每轮调的是 LLM API |
| Tool 定义 | 一份「路由表 / 接口文档」 | 告诉模型有哪些函数可调 |
| `tool_calls` | 前端发来的「请求体」 | 模型告诉你调哪个函数、传什么参数 |
| 执行 Tool | 后端真正跑 `getOrder()` | 模型不执行，你的代码执行 |
| 回喂结果 | 把函数返回值 `push` 回数组 | 让模型看到执行结果 |
| Orchestrator | Express 里的一个 handler | 串联全流程的主控代码 |
| maxSteps 保护 | 请求重试上限 / 超时 | 防止无限循环 |

一句话类比：

> Agent Loop ≈ 一个 `while` 循环，每轮 `await` 一次大模型；模型像个「只会下命令但不动手的项目经理」，你的后端代码是「真正干活的工程师」。

## 8. 今日自测题

### 8.1 Agent 和普通调用 LLM 最大的区别是什么？

参考答案：

> ✅ 普通调用是「问一句答一句」，模型只能靠自身记忆回答；Agent 让模型能「下指令调用后端工具」，从而查到真实数据（订单、库存等），并且可能来回多轮。

---

### 8.2 模型会自己执行 `get_order` 吗？

参考答案：

> ✅ 不会。模型只会返回 `tool_calls`，告诉你「我想调用 get_order，参数是什么」。真正执行的是你的后端代码，执行完还要把结果回喂给模型。

---

### 8.3 Agent Loop 为什么要写成循环，而不是一次请求？

参考答案：

> ✅ 因为一个问题可能需要调用多个工具，模型会分多轮决策（先查订单、再查库存）。所以要用 while/for 循环：只要模型还要求调工具就继续，直到它直接给出答案。

---

### 8.4 Agent 循环有哪三个出口？

参考答案：

> ✅ 1）模型没有 tool_calls，直接返回答案；2）模型要求调工具，执行后回喂并继续循环；3）循环次数用完，兜底返回防止死循环。

---

### 8.5 工具执行结果要怎么处理才能让模型看到？

参考答案：

> ✅ 把结果作为一条 `role: "tool"` 的消息（带上对应的 `tool_call_id`），追加进 messages 数组，然后再次请求模型。模型看到工具结果后才能生成最终回答。
