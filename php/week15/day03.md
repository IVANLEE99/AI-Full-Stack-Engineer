## 今日目标

给客服 Agent 编写一份高质量的 System Prompt：定好角色、能力边界、语气和「不知道就说不知道」的规则，并用 3 轮对话验证它真的按 Prompt 办事。

今天你要真正掌握这一句话：

> System Prompt 是 Agent 的「岗前培训手册」——它在每次对话里都以最高优先级注入，决定 Agent 是谁、能做什么、不能做什么、遇到边界怎么办；写得好，模型少幻觉、少越权、语气稳。

## 1. 学习内容

### 1.1 四种消息角色的分工

Agent 对话里有四种角色，先分清：

```text
system    ：岗前培训手册。定义 Agent 是谁、规则是什么。优先级最高
user      ：用户说的话
assistant ：模型的回复（可能是文字，也可能是 tool_calls）
tool      ：工具执行结果（昨天学的回喂）
```

对比表：

| 角色 | 谁写的 | 作用 | 类比 |
|---|---|---|---|
| system | 你（开发者） | 定规则、定人设 | 员工手册 |
| user | 用户 | 提问 | 客户来电 |
| assistant | 模型 | 回答 | 客服说话 |
| tool | 后端 | 工具结果 | 后台查到的数据 |

小白重点：

> `system` 通常只在对话最开头出现一次，但它对整段对话都生效。它不是「第一句话」，而是「贯穿始终的规则」。

---

### 1.2 为什么 System Prompt 优先级最高

模型在生成回答时，会把 System Prompt 当成最高优先级的约束。这带来两个实际好处：

1. 防跑题：用户想让客服 Agent 写代码、聊八卦，System Prompt 可以约束「只聊订单/商品/售后」。
2. 防越权：System Prompt 可以规定「不能编造订单信息、不能承诺退款金额」。

用 Node 中间件类比：

```js
// System Prompt ≈ 全局中间件，每个请求都先过它
app.use((req, res, next) => {
  req.context = {
    role: "电商客服助手",
    canDo: ["查订单", "查商品", "答常见问题"],
    cannotDo: ["编造信息", "承诺退款", "闲聊无关话题"],
  };
  next();
});
```

小白重点：

> System Prompt 像全局中间件：它不是某一次请求的逻辑，而是每一次请求都要先经过的「上下文和规则」。

---

### 1.3 一个好用的 System Prompt 六段式结构

写 System Prompt 别靠灵感，用固定结构：

```text
1. 角色（你是谁）
2. 能力（你能做什么，有哪些工具）
3. 边界（你不能做什么）
4. 工具使用规则（什么时候调工具）
5. 语气风格（怎么说话）
6. 兜底规则（不知道/查不到怎么办）
```

对比表：

| 段落 | 作用 | 不写会怎样 |
|---|---|---|
| 角色 | 定人设 | 回答飘忽，一会儿客服一会儿助手 |
| 能力 | 让模型知道自己有工具 | 该调工具时不调 |
| 边界 | 防越权/防跑题 | 乱承诺、乱聊 |
| 工具规则 | 定何时调工具 | 该查订单时瞎编一个状态 |
| 语气 | 统一体验 | 一会儿客气一会儿生硬 |
| 兜底 | 处理未知 | 查不到就胡编（幻觉） |

---

### 1.4 亲手写客服 Agent 的 System Prompt

按六段式写一版完整的：

```js
const systemPrompt = `
你是「优选商城」的在线客服助手，名字叫小优。

# 你能做什么
- 帮用户查询订单状态和物流（使用 get_order 工具）
- 帮用户查询商品信息和库存（使用 get_product 工具）
- 回答常见问题（退换货政策、配送时效等）

# 你不能做什么
- 不能编造订单号、物流状态、价格等任何信息。没查到就如实说没查到。
- 不能承诺具体的退款金额或退款时间，这类问题请引导用户联系人工客服。
- 不回答与购物无关的话题（如写代码、聊天、时政），礼貌地把话题引回购物。

# 什么时候调用工具
- 用户问「我的订单/物流/发货」→ 调用 get_order（需要订单号，没有就先问用户要）
- 用户问「某商品价格/库存/详情」→ 调用 get_product
- 不要在没有工具结果的情况下，自己编造订单或商品数据。

# 说话风格
- 简洁、亲切、有礼貌，每次回复控制在 2~3 句话。
- 用「您」称呼用户。

# 遇到不确定时
- 查不到数据 → 如实告知，并建议用户核对信息或联系人工客服。
- 超出你能力范围 → 礼貌说明，并引导到人工客服。
`.trim();
```

小白重点：

> 用 Markdown 小标题（`#`）分段能显著提升模型的遵守度。把 Prompt 当成写给新员工的手册来写，而不是一句模糊的「你是客服」。

---

### 1.5 重点写好「边界」和「兜底」

这两段是防幻觉、防越权的核心。反面例子对比：

| 写法 | 效果 |
|---|---|
| ❌「你是客服，帮用户解决问题」 | 用户问退款金额，模型可能瞎编一个数字 |
| ✅「不能承诺退款金额，引导到人工客服」 | 模型会说「退款金额需人工核实，为您转接」 |
| ❌ 不写兜底 | 查不到订单时，模型可能编一个「已发货」 |
| ✅「查不到就如实说，别编造」 | 模型会说「没查到这个订单，请核对订单号」 |

小白重点：

> 幻觉（模型一本正经地编造事实）在客服场景是重大事故。System Prompt 里写清「不知道就说不知道、不许编造」，是把幻觉风险降到最低的第一道防线。

---

### 1.6 把 System Prompt 接进 Agent Loop

用昨天的 Loop，只需把 messages 的第一条换成今天写的 systemPrompt：

```js
async function runAgent(userMessage, history = []) {
  const messages = [
    { role: "system", content: systemPrompt }, // 今天写的手册
    ...history,                                 // 之前的多轮对话
    { role: "user", content: userMessage },
  ];
  // ... 后面的 Agent Loop 和昨天一样（调模型、执行工具、回喂）
}
```

小白重点：

> `system` 永远放在 messages 数组最前面，且只放一条。多轮对话时，把历史对话接在 system 之后、当前 user 之前。

## 4. JS/Node.js 类比

| Prompt 概念 | Node.js / JS 类比 | 说明 |
|---|---|---|
| System Prompt | 全局中间件 / 全局配置 | 每次请求都先经过 |
| 角色定义 | 服务的职责说明 | 这个服务是干嘛的 |
| 能力边界 | 权限/白名单 | 能调哪些接口、不能干什么 |
| 工具使用规则 | 路由到 handler 的条件 | 什么请求走哪个逻辑 |
| 兜底规则 | 默认分支 / catch | 没匹配上/出错怎么办 |
| 多轮 history | session / 会话上下文 | 记住之前说过什么 |

一句话类比：

> System Prompt ≈ 给这个「客服服务」写的全局配置 + 员工手册，它决定服务的人格和规则，而 user 消息只是一次次进来的请求。

## 8. 今日自测题

### 8.1 System Prompt 在对话里放在哪、放几条？

参考答案：

> ✅ 放在 messages 数组最前面，通常只放一条。它对整段对话都生效，是最高优先级的规则，不是「第一句话」而是「贯穿始终的手册」。

---

### 8.2 好的 System Prompt 六段式结构是哪六段？

参考答案：

> ✅ 角色、能力、边界、工具使用规则、语气风格、兜底规则。

---

### 8.3 为什么必须在 Prompt 里写「不知道就说不知道」？

参考答案：

> ✅ 防幻觉。模型倾向于「一本正经地编造」，在客服场景编造订单状态或退款金额是重大事故。明确写「不许编造、查不到就如实说」是降低幻觉的第一道防线。

---

### 8.4 用户让客服 Agent 写代码，应该怎么处理？

参考答案：

> ✅ Prompt 里的「边界」段应约束只服务购物相关话题，遇到无关请求礼貌拒绝并把话题引回购物。这靠 System Prompt 的边界规则实现，而不是靠代码 if 判断。

---

### 8.5 System Prompt 和 Node 中间件像在哪？

参考答案：

> ✅ 都是「每次请求都先经过的全局逻辑/上下文」。中间件为每个请求注入统一处理，System Prompt 为每轮对话注入统一的人设和规则。
