### 1.2 纯向量检索的短板

本周你用的都是**语义检索**（向量相似度）。它很强，但有明显短板：

```text
场景：知识库里有一条 "错误码 PXG-2048 表示网关超时"
用户问：      "PXG-2048 是什么意思？"

语义检索：   "PXG-2048" 这种型号/编号，Embedding 模型没见过，
             向量化后语义很弱，可能召回不到那条精确记录。

关键词检索： 直接匹配 "PXG-2048" 这个词，秒中。
```

| 检索方式 | 擅长 | 短板 |
|---|---|---|
| 向量（语义） | 同义、改写、"意思相近" | 精确关键词、型号、缩写、专名 |
| 关键词（BM25） | 精确词、专有名词、代码 | 同义词、换个说法就搜不到 |

小白重点：两种方式的强项和弱项**正好互补**。这就是要做 Hybrid（混合）的根本原因。

---

### 1.3 认识 BM25（关键词检索）

BM25 是经典的关键词打分算法，你可以先这样理解：

```text
BM25 打分 ≈ 看查询词在文档里出现得多不多（词频）
           × 这个词是不是"稀有词"（越稀有越值钱，IDF）
           ÷ 文档长度归一化（防止长文档靠堆词占便宜）
```

一句话：

> BM25 是搜索引擎（Elasticsearch、Lucene）几十年验证过的关键词相关性算法，本质是"精确匹配 + 词的稀有度加权"。

小白重点：你不用手写 BM25，Elasticsearch / OpenSearch / 部分向量库都内置了。今天只要理解它"擅长精确关键词"这个定位即可。

---

### 1.4 Hybrid Search 怎么融合

Hybrid Search = 同时跑向量检索和关键词检索，再把两份结果**融合排序**：

```text
用户查询
   │
   ├──►  向量检索  ──► 结果A（按相似度排序）
   │
   └──►  BM25 检索 ──► 结果B（按关键词得分排序）
                          │
                          ▼
              融合排序（如 RRF）→ 最终 Top-K
```

最常用的融合方法是 **RRF（Reciprocal Rank Fusion，倒数排名融合）**，思路极简：

```text
一个文档的最终得分 = Σ  1 / (k + 该文档在各路结果里的排名)

k 通常取 60。排名越靠前（rank 越小），1/(k+rank) 越大，得分越高。
一个文档如果在向量结果和关键词结果里都靠前，融合后自然排最前。
```

小白重点：RRF 只看"排名"，不看两路分数的绝对大小，所以不用纠结"相似度 0.8 和 BM25 得分 12 怎么比"。这让融合变得非常好实现。

---

### 1.5 一个极简 RRF 示例

```python
def rrf_fuse(vector_ids, keyword_ids, k=60, top_k=5):
    """vector_ids / keyword_ids 都是按排名排好的 id 列表"""
    scores = {}
    for rank, doc_id in enumerate(vector_ids):
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank)
    for rank, doc_id in enumerate(keyword_ids):
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank)
    # 按融合得分从高到低
    ranked = sorted(scores.items(), key=lambda x: x[1], reverse=True)
    return [doc_id for doc_id, _ in ranked[:top_k]]


vec = ["d3", "d1", "d7"]      # 向量检索排名
kw  = ["d1", "d9", "d3"]      # 关键词检索排名
print(rrf_fuse(vec, kw))      # d1、d3 在两路都靠前，融合后领先
```

小白重点：这段代码是下周的核心。今天读懂"两路排名 → 倒数相加 → 重新排序"就够了。

## 4. JS/Node.js 类比

| RAG / 检索概念 | Node.js / 前端类比 | 说明 |
|---|---|---|
| 向量（语义）检索 | 模糊搜索 / "猜你想搜" | 按意思找 |
| BM25（关键词）检索 | `Array.includes` / 全文精确匹配 | 按词找 |
| Hybrid Search | 前端把两个数据源结果合并去重排序 | 融合多路结果 |
| RRF 融合 | 多个排行榜按名次加权合并 | 只看排名，不看绝对分 |
| 验收清单 | 上线前 checklist | 逐项确认 |

一句话类比：

> Hybrid Search ≈ 你同时问了"懂语义的搜索"和"认识精确词的搜索"两位专家，再按他们各自的排名投票合并，得到既懂意思又不漏关键词的结果。

## 8. 今日自测题

### 8.1 纯向量（语义）检索的最大短板是什么？

参考答案：

> ✅ 对精确关键词、型号、缩写、专有名词召回弱。这类词 Embedding 模型很难准确表达语义，换成关键词检索反而秒中。

---

### 8.2 BM25 擅长什么？

参考答案：

> ✅ 精确关键词匹配。本质是"词频 × 词的稀有度（IDF）÷ 文档长度归一化"，是搜索引擎几十年验证过的关键词相关性算法。

---

### 8.3 Hybrid Search 是什么？

参考答案：

> ✅ 同时跑向量检索和关键词（BM25）检索，再把两路结果融合排序。既懂语义又不漏精确关键词，是企业知识库提升准确率的标准做法。

---

### 8.4 RRF 融合的核心思路？

参考答案：

> ✅ 一个文档的最终得分 = 各路结果里 `1/(k+排名)` 之和（k 常取 60）。只看排名不看绝对分数，两路都靠前的文档融合后自然排最前。

---

### 8.5 为什么向量检索和关键词检索适合组合使用？

参考答案：

> ✅ 因为它们的强弱正好互补：向量擅长同义/改写但漏关键词，关键词擅长精确词但不懂同义。组合后覆盖两类场景，整体召回和准确率都更高。
