## 1. 学习内容

### 1.1 RAG 检索的完整链路

先把整条链路画在脑子里：

```text
用户问题
   │  (1) 把问题 Embedding
   ▼
查询向量
   │  (2) 在向量库里算相似度、取 Top-K
   ▼
Top-K 个 Chunk（召回结果）
   │  (3) 拼进 Prompt
   ▼
大模型生成答案
```

本周只做前两步（召回）。第三步（喂给大模型生成）是后面几周的事。今天的焦点就是第 (1)(2) 步质量如何。

小白重点：

> RAG = Retrieval（召回）+ Generation（生成）。召回错了，后面生成得再流畅也是"一本正经地胡说"。所以召回是地基。

---

### 1.2 召回到底在干什么

召回的本质：**把"问题"和"知识片段"放到同一个向量空间里比方向**。

关键点在于——问题和答案文本用的是**同一个 Embedding 模型**。这样"如何退款"这个问题的向量，才会和"退款流程会校验订单状态..."这段文本的向量方向接近。

```python
def retrieve(store, question: str, top_k: int = 3):
    # 问题也要用同一个模型 Embedding
    q_vec = embed([question])[0]
    return store.query(q_vec, top_k=top_k)
```

小白重点：一定用**同一个 Embedding 模型**给问题和文档做向量化。用不同模型，向量空间对不上，召回全乱。

### 1.4 设计测试问题 + 期望命中

测召回，必须先有"标准答案"——每个问题**期望命中哪个 doc**。这就是你的测试集：

```python
test_cases = [
    {"q": "怎么申请退款？",         "expect": "k03"},
    {"q": "支持哪些付款方式？",     "expect": "k02"},
    {"q": "怎么查快递到哪了？",     "expect": "k04"},
    {"q": "优惠券能一起用吗？",     "expect": "k06"},
    {"q": "登录失败太多次会怎样？", "expect": "k08"},
]
```

注意：测试问题**故意不和原文用词一样**（"付款"vs"支付"、"快递"vs"物流"）。这才能真正考验语义召回——如果只是关键词匹配，这些就命中不了。

小白重点：好的召回测试集，问题要用"用户的说法"，而不是"文档里的原话"。否则你测的是关键词匹配，不是语义检索。

---

### 1.5 量化召回质量：Hit@K 和 MRR

不能只看一眼就说"感觉还行"。用两个指标量化。

**Hit@K（命中率）**：Top-K 结果里，有没有包含期望的那个 doc。命中记 1，没命中记 0，取平均。

**MRR（平均倒数排名）**：期望 doc 排在第几位。排第 1 得 1 分，第 2 得 1/2，第 3 得 1/3... 没命中得 0。它比 Hit@K 更严格，能反映"排得够不够靠前"。

```python
def evaluate(store, test_cases, top_k=3):
    hit = 0
    mrr_total = 0.0
    for case in test_cases:
        hits = retrieve(store, case["q"], top_k=top_k)
        ids = [h["id"] for h in hits]
        if case["expect"] in ids:
            hit += 1
            rank = ids.index(case["expect"]) + 1  # 排名从 1 开始
            mrr_total += 1 / rank
        print(f"Q: {case['q']}")
        print(f"   期望: {case['expect']}  召回: {ids}")

    n = len(test_cases)
    print(f"\nHit@{top_k}: {hit}/{n} = {hit/n:.2f}")
    print(f"MRR: {mrr_total/n:.3f}")

evaluate(store, test_cases, top_k=3)
```

| 指标 | 含义 | 越高越好？ |
|---|---|---|
| Hit@K | Top-K 里是否命中期望 doc | 是 |
| MRR | 期望 doc 的排名倒数平均 | 是 |

小白重点：Hit@3 = 1.0 说明"5 个问题都在前 3 命中"；但如果 MRR 才 0.6，说明虽然命中了，但排名不够靠前，还有优化空间。

### 1.6 分析召回失败

如果某个问题没命中，别急着改代码，先看"到底召回了什么"：

```python
hits = retrieve(store, "登录失败太多次会怎样？", top_k=3)
for h in hits:
    print(round(h["distance"], 3), h["metadata"]["title"], "|", h["document"])
```

常见失败原因：

| 现象 | 可能原因 | 明天的优化方向 |
|---|---|---|
| 完全召回错 doc | Chunk 太大，主题混杂 | 切细一点 |
| 期望 doc 排第 4、5 名 | 相似片段太多互相干扰 | 加 metadata 过滤 / 调 Chunk |
| 距离都很接近 | 语料太短、区分度低 | 补充语料 / 换模型 |
| 问题太口语化召不回 | 问题和文档表达差异大 | Day07 预习的 Hybrid Search |

小白重点：召回失败是正常的、有价值的信号。今天把失败 case 记下来，正是明天（Day05 优化日）要动手改的输入。

### 1.7 Top-K 的 K 怎么选

K 太小，可能漏掉正确答案；K 太大，塞进 Prompt 的噪声变多、还费 token。

```text
K = 3 ~ 5   常见起点
K 偏大：召回全但噪声多
K 偏小：精准但可能漏
```

实践做法：先固定 K=3 跑一遍，再试 K=5，对比 Hit@K 变化。如果 K 从 3 加到 5，命中率大涨，说明正确答案常排在 4、5 名——这是 Chunk 或模型需要优化的信号。

## 4. JS/Node.js 类比

| Python / RAG | Node.js / JS 类比 | 说明 |
|---|---|---|
| `retrieve()` | 搜索接口 `search(q)` | 输入 query 返回结果列表 |
| Top-K | `results.slice(0, k)` | 取前 K 条 |
| Hit@K | 搜索"命中率"测试 | 类似接口测试断言 |
| MRR | 搜索排序质量指标 | 关注排名靠不靠前 |
| test_cases | Jest 的测试用例数组 | 输入 + 期望值 |
| evaluate() | 跑测试套件 | 批量断言 + 汇总 |

一句话类比：

> 召回测试 ≈ 给搜索功能写单元测试。`test_cases` 是用例，Hit@K 是断言通过率，只不过判定标准从"精确相等"变成了"期望 doc 在 Top-K 里"。

## 8. 今日自测题

### 8.1 RAG 里的召回是什么？

参考答案：

> ✅ 把用户问题也向量化，在向量库里找出最相似的 Top-K 个 Chunk。它是 RAG 的第一步，决定了后面生成的答案有没有正确的依据。

---

### 8.2 为什么问题和文档要用同一个 Embedding 模型？

参考答案：

> ✅ 只有同一个模型，问题和文档才在同一个向量空间里，方向才可比。用不同模型，向量空间对不上，相似度计算失去意义，召回会乱。

---

### 8.3 Hit@K 和 MRR 的区别？

参考答案：

> ✅ Hit@K 只看 Top-K 里有没有命中期望 doc（命中/不命中）。MRR 还看命中的排名，排得越靠前分越高。MRR 更严格，能反映排序质量。

---

### 8.4 测试问题为什么不该照抄文档原文？

参考答案：

> ✅ 照抄原文只能测出关键词匹配，测不出语义召回能力。要用"用户真实的说法"（如用"付款"而文档写"支付"），才能验证 Embedding 是否真的理解了语义。

---

### 8.5 Top-K 的 K 太大或太小分别有什么问题？

参考答案：

> ✅ K 太小可能漏掉正确答案；K 太大会把无关噪声塞进 Prompt，既影响生成质量又费 token。常见起点是 3~5，需要用测试数据调。
