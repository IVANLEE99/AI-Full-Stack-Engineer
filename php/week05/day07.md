### 1.4 检查薄 Controller 能力

请手写一个 action，并检查它是否只做：

- 鉴权
- 取参
- 基础校验
- 调用 Request/Service
- 返回响应

模板：

```php
<?php

declare(strict_types=1);

public function actionExample(): array
{
    $userId = $this->requireLogin();

    $params = [
        'user_id' => $userId,
        'id' => (int)$this->request->post('id'),
    ];

    $result = $this->someRequest->someMethod($params);

    return $this->success($result);
}
```

检查它没有做：

- 金额计算
- 库存扣减
- 订单状态流转
- 支付状态修改
- 复杂 SQL 查询

---

### 1.5 本周最重要的边界判断

BFF 适合做：

| 适合网关 | 说明 |
|---|---|
| 鉴权入口 | 检查用户身份 |
| 公参注入 | user_id、site_id、lang、trace_id |
| 参数基础校验 | 必填、类型、格式 |
| 接口聚合 | 一个前端接口聚合多个服务 |
| 响应适配 | 返回前端想要的结构 |
| 调用下游服务 | 通过 `*Request` 封装调用 |

BFF 不适合做：

| 不适合网关 | 应该放在哪里 |
|---|---|
| 订单金额计算 | 订单/价格服务 |
| 库存扣减 | 库存/商品服务 |
| 支付状态机 | 支付服务 |
| 优惠券核销 | 营销/优惠券服务 |
| 数据库事务 | 具体业务服务 |
| 复杂风控规则 | 风控服务 |

## 8. 今日自测题

### 8.1 Week 05 的核心验收标准是什么？

参考答案：

> ✅ 不是知道 BFF 这个词，而是能从一个前端 URL 反查到网关 action、Request 客户端和内网服务，并能判断哪些逻辑该留在网关、哪些必须下沉到业务服务。

---

### 8.2 一次完整的 BFF 请求会经过哪些主要环节？

参考答案：

> ✅ 前端请求 → BFF 路由/Controller → 鉴权 + 公参注入 → 薄 action → HTTP Request 客户端 → 内网业务服务。

---

### 8.3 哪些逻辑适合放在 BFF 网关，哪些不适合？

参考答案：

> ✅ 适合放网关：鉴权入口、公参注入、参数基础校验、接口聚合、响应适配、调用下游服务。不适合放网关：订单金额计算、库存扣减、支付状态机、优惠券核销、数据库事务、复杂风控，这些要下沉到对应业务服务。

---

### 8.4 检查一个 action 是否够“薄”，应该看它有没有做什么？

参考答案：

> ✅ 薄 action 只做鉴权、取参、基础校验、调用 Request/Service、返回响应。如果它做了金额计算、库存扣减、订单状态流转、支付状态修改或复杂 SQL，就说明它太“厚”了，逻辑应下沉到下游服务。

---

### 8.5 预习订单域时，为什么金额不能相信前端传来的值？

参考答案：

> ✅ 因为前端参数可以被伪造，如果直接采信前端金额，用户可能篡改价格低价下单。金额必须由后端订单/价格服务重新计算，网关只负责传递用户身份和商品等参数。
