## 1. 学习内容

### 1.1 售后状态为什么重要？

售后单不是静态记录，它会经历多个阶段：

```text
用户申请
  ↓
待审核
  ↓
审核通过
  ↓
处理中
  ↓
退款/补发/退货完成
  ↓
售后完成
```

也可能被拒绝或取消：

```text
待审核 → 审核拒绝
待审核 → 用户取消
处理中 → 退款失败/补发异常
```

状态的作用：

| 作用 | 示例 |
|---|---|
| 控制用户操作 | 用户只能在待审核前取消 |
| 控制客服操作 | 客服只能审核待审核售后单 |
| 控制仓库操作 | 退货待收货时才能确认收货 |
| 控制退款触发 | 审核通过并满足条件后触发退款 |
| 控制前端展示 | 不同状态展示不同按钮和文案 |

---

### 1.2 常见售后状态

实际状态以 `AfterSaleStatusEnum` 为准，常见可能有：

| 状态 | 含义 |
|---|---|
| 待审核 | 用户已申请，等待客服处理 |
| 审核通过 | 客服同意进入后续处理 |
| 审核拒绝 | 客服拒绝申请 |
| 待用户退货 | 需要用户寄回商品 |
| 待仓库收货 | 等待仓库确认收到退货 |
| 退款中 | 正在调用支付退款 |
| 退款成功 | 退款已完成 |
| 补发中 | 正在创建补发/发货任务 |
| 已完成 | 售后流程完成 |
| 已取消 | 用户或系统取消售后 |

---

### 1.3 售后状态和订单状态的关系

售后状态不等于订单状态。

| 售后状态 | 订单可能状态 | 说明 |
|---|---|---|
| 待审核 | 已支付/已发货/已完成 | 用户发起售后 |
| 退款中 | 已支付/售后中 | 正在退钱 |
| 退款成功 | 已退款/部分退款 | 订单可能进入售后完成 |
| 补发中 | 已发货/已完成 | 原订单不一定取消 |
| 售后完成 | 已完成/已退款 | 取决于售后类型 |

小白重点：订单、支付、售后都有状态机，它们互相关联但不能混为一个字段。

---

### 1.4 主线状态流转图

退货退款主线：

```text
用户申请售后
  ↓
待审核
  ├─ 客服拒绝 → 审核拒绝 → 结束
  └─ 客服通过 → 待用户退货
                  ↓
                待仓库收货
                  ↓
                退款中
                  ↓
                退款成功
                  ↓
                售后完成
```

补发主线：

```text
用户申请补发
  ↓
待审核
  ├─ 客服拒绝 → 审核拒绝
  └─ 客服通过 → 补发中 → 补发完成 → 售后完成
```

---

### 1.5 每个状态谁能操作？

| 状态 | 用户可操作 | 客服可操作 | 系统可操作 |
|---|---|---|---|
| 待审核 | 取消、补充凭证 | 通过/拒绝 | 超时提醒 |
| 待用户退货 | 填退货物流 | 查看/催促 | 超时关闭 |
| 待仓库收货 | 查看进度 | 人工干预 | 仓库确认收货 |
| 退款中 | 查看进度 | 查看/重试 | 调用退款/MQ |
| 补发中 | 查看物流 | 修改补发信息 | 推送物流 |
| 已完成 | 查看结果 | 查看记录 | 发送通知 |

权限边界非常重要：普通用户不能调用审核接口，客服也不应直接伪造退款成功。

---

### 1.6 非法状态流转风险

| 非法流转 | 风险 |
|---|---|
| 审核拒绝 → 退款中 | 拒绝后仍退款 |
| 已完成 → 退款中 | 重复退款 |
| 未审核 → 待仓库收货 | 绕过客服审核 |
| 待用户退货 → 退款成功 | 仓库未收货就退款 |
| 补发完成 → 再次补发中 | 重复补发 |

后端必须校验状态流转，不能只靠前端按钮隐藏。

---

### 1.7 TypeScript enum 类比

前端可能有：

```ts
enum AfterSaleStatus {
  PendingReview = 10,
  Approved = 20,
  RefundProcessing = 30,
  Completed = 100,
  Rejected = 900,
}
```

PHP enum / 常量类似：

```php
<?php

enum AfterSaleStatusEnum: int
{
    case PendingReview = 10;
    case Approved = 20;
    case Completed = 100;
    case Rejected = 900;
}
```

后端除了定义枚举，还要控制合法流转。
