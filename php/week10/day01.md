## 1. 学习内容

### 1.1 售后域为什么复杂？

售后可能包含：

- 仅退款
- 退货退款
- 换货
- 补发
- 维修
- 取消售后
- 客服审核
- 仓库收货
- 支付退款
- 用户通知

它连接多个域：

```text
用户域：谁申请售后？
订单域：申请的是哪笔订单？
支付域：是否需要退款？
物流域：是否需要退货/补发？
权限域：谁能审核？
MQ：是否异步通知和处理？
```

小白重点：售后是多个业务域交叉点，所以 Service 很容易变大。

---

### 1.2 AfterSaleService 常见职责

| 职责 | 示例 |
|---|---|
| 创建售后申请 | 用户发起退款/退货/补发 |
| 校验订单归属 | 当前用户是否拥有该订单 |
| 校验可售后状态 | 订单是否已支付/已发货/可退款 |
| 计算可退金额 | 商品金额、运费、优惠、已退金额 |
| 审核售后 | 客服同意/拒绝 |
| 处理退货 | 填写退货物流、仓库收货 |
| 触发退款 | 售后通过后调用支付退款 |
| 更新状态 | 申请中、审核中、退款中、完成 |
| 发送通知 | 通知用户审核结果/退款进度 |
| 查询列表/详情 | 用户端、后台端不同视角 |

---

### 1.3 售后申请基础链路

```text
用户提交售后申请
  ↓
AfterSaleController
  ↓
参数校验 Form
  ↓
AfterSaleService::apply()
  ↓
校验 user_id 与订单归属
  ↓
校验订单/商品是否可售后
  ↓
校验退款金额和凭证图片
  ↓
创建售后单
  ↓
进入待审核状态
  ↓
发送通知或 MQ
```

---

### 1.4 必须关注的风险

| 风险 | 说明 |
|---|---|
| 越权申请 | A 用户申请 B 用户订单售后 |
| 重复售后 | 同一订单商品重复申请 |
| 超额退款 | 退款金额超过可退金额 |
| 状态错误 | 已完成订单被错误回退 |
| 凭证图片风险 | 上传恶意文件或泄露隐私 |
| 审核权限 | 普通用户不能调用审核接口 |
| 退款幂等 | 同一售后不能重复退款 |

---

### 1.5 为什么需要拆分策略？

不同售后类型处理方式不同：

| 类型 | 特点 |
|---|---|
| 仅退款 | 不需要退货物流，重点是退款金额 |
| 退货退款 | 需要用户寄回、仓库收货、再退款 |
| 补发 | 不退款，创建补发/物流任务 |
| 换货 | 可能同时有退回和重新发货 |

如果全部写在 `AfterSaleService` 里，会出现大量 `if processingType == ...`。

策略模式可以把每种类型拆到独立类中。
