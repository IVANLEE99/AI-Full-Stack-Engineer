## 今日目标

用伪代码分别写出「LangGraph 图编排」和「PHP 链式 Node 编排」两种模式，并说清楚它们的 3 处相同和 3 处不同。

今天你要真正掌握这一句话：

> 图编排（LangGraph）和链式 Node 编排（PayService）都在做同一件事——「把一份共享状态，顺着预先定义好的步骤流转下去」。它们最大的差别在于「下一步去哪」这件事：链式是写死的顺序，图编排是靠边（Edge）和条件函数动态决定的。

## 1. 学习内容

### 1.1 先把两种模式各画一张图

昨天和前天你分别接触了两种编排。今天先把它们并排放在一起看。

**模式 A：链式 Node（PHP 支付）**

```text
Context ──▶ [ValidateNode] ──▶ [StockNode] ──▶ [PayNode] ──▶ [NotifyNode] ──▶ 完成
                （顺序是写死的，一个接一个）
```

**模式 B：图编排（LangGraph）**

```text
        ┌──────────────┐
State ─▶│  understand  │
        └──────┬───────┘
               │
        ┌──────▼───────┐      条件边
        │   retrieve   │───────┬────────┐
        └──────┬───────┘       │        │
               │          need_more   enough
        ┌──────▼───────┐       │        │
        │   answer     │◀──────┘        │
        └──────────────┘◀───────────────┘
```

小白重点：

> 链式像「一条直路」，图像「有岔路口的路网」。但两者跑起来时，都在做「拿着状态往下走」这件事。

---

### 1.2 用伪代码写出「链式 Node 编排」

我们先把 PHP 的链式编排抽象成不依赖具体语言的伪代码。

```text
# ============ 链式 Node 编排（Chain）============

# 1. 定义一份贯穿全程的上下文
context = {
    order_id: 123,
    amount: 100,
    channel: "wechat",
}

# 2. 定义一串节点（顺序写死）
nodes = [
    ValidateNode,   # 校验参数
    StockNode,      # 扣库存
    PayNode,        # 调用支付渠道
    NotifyNode,     # 发通知
]

# 3. 引擎：从头到尾依次执行
function run(context, nodes):
    for node in nodes:              # 顺序固定
        result = node.handle(context)
        if result.is_failed():
            return result          # 出错就中断
    return context

run(context, nodes)
```

对应到真实 PHP（简化版）：

```php
<?php

declare(strict_types=1);

foreach ($this->nodes as $node) {
    $result = $node->handle($context);
    if ($result->isFailed()) {
        return $result;
    }
}
```

小白重点：

> 链式编排里，「下一个执行谁」是由数组顺序决定的。你想调整流程，就得改 `nodes` 数组的顺序或内容。

---

### 1.3 用伪代码写出「图编排」

再把 LangGraph 的图编排抽象成伪代码。

```text
# ============ 图编排（Graph）============

# 1. 定义一份共享状态（和链式的 context 是一个意思）
state = {
    question: "退款要多久到账？",
    docs: [],
    answer: "",
}

# 2. 定义节点（函数），每个节点读写 state
function understand(state):  state.intent = parse(state.question); return state
function retrieve(state):    state.docs = search(state.question);  return state
function answer(state):      state.answer = llm(state.docs);       return state

# 3. 把节点注册进图，并用「边」描述连接关系
graph = new Graph()
graph.add_node("understand", understand)
graph.add_node("retrieve", retrieve)
graph.add_node("answer", answer)

graph.set_entry("understand")           # 从哪开始
graph.add_edge("understand", "retrieve")# 普通边：固定连线

# 条件边：下一步去哪，由函数动态决定
function route(state):
    if len(state.docs) == 0:
        return "retrieve"    # 没查到，回去再查
    else:
        return "answer"      # 查到了，去回答

graph.add_conditional_edge("retrieve", route)
graph.add_edge("answer", END)

# 4. 引擎：从入口开始，按边不断跳转，直到 END
app = graph.compile()
result = app.invoke(state)
```

小白重点：

> 图编排里，「下一个执行谁」是由 **Edge（边）** 和 **条件函数** 决定的。你想调整流程，就改边的连接关系，而不用动节点内部代码。

---

### 1.4 逐行对照：3 处相同

把两段伪代码并排看，会发现骨架其实高度一致。

| # | 相同点 | 链式 Node | 图编排 |
|---|--------|-----------|--------|
| 1 | 都有一份「共享状态」贯穿全程 | `context` | `state` |
| 2 | 每一步都是「读状态 → 处理 → 写回状态」 | `node.handle(context)` | `node(state)` |
| 3 | 都有一个「引擎」负责驱动步骤流转 | `run()` 里的 for 循环 | `app.invoke()` |

用一句话概括三处相同：

> 都是「共享状态 + 一组处理步骤 + 一个驱动引擎」。这就是所有编排模式的最小公共骨架。

---

### 1.5 逐行对照：3 处不同

差异集中在「怎么决定下一步」和「结构长什么样」。

| # | 不同点 | 链式 Node | 图编排 |
|---|--------|-----------|--------|
| 1 | 下一步由谁决定 | 数组顺序写死 | Edge + 条件函数动态决定 |
| 2 | 能否回头 / 分支 / 循环 | 天然只能往前走 | 天然支持分支、回环（loop） |
| 3 | 结构形状 | 线性链（一条路） | 有向图（路网，可有岔路） |

补充一处更深的差异：

| 视角 | 链式 Node | 图编排 |
|------|-----------|--------|
| 出错处理 | 通常「立即中断返回」 | 可以「路由到专门的错误节点」再决定 |
| 适合场景 | 步骤固定的业务流（支付、下单） | 需要动态决策的 AI Agent（要不要再检索、要不要调工具） |

小白重点：

> 记住一句话：**链式擅长「确定的流程」，图擅长「不确定、要边走边判断的流程」。** 支付流程步骤基本固定，所以用链；AI Agent 经常要「再想想 / 再查查」，所以用图。

---

### 1.6 为什么 AI Agent 更偏爱图编排

再深入一点，理解「为什么 AI 场景要用图」。

AI Agent 的一个典型循环：

```text
提问 → 检索资料 → 判断资料够不够
                   ├─ 不够 → 回去再检索（循环）
                   └─ 够了 → 生成回答
```

这里有一个「回环」：`检索 → 判断 → 再检索`。

- 链式编排要实现回环，得自己写 while 循环、维护计数器，代码会变脏。
- 图编排里，回环就是「一条指回去的边」，天然支持。

```text
# 图里表达回环，只是加一条边而已
graph.add_conditional_edge("check", route)   # route 可能返回 "retrieve"
```

小白重点：

> AI 流程经常「不知道要循环几次」（比如要检索几轮才够），这种「动态循环 + 动态分支」正是图编排的强项，也是 LangGraph 存在的意义。

## 4. JS/Node.js 类比

| 概念 | 链式 Node（PHP） | 图编排（LangGraph） | JS/Node 类比 |
|------|-----------------|--------------------|-------------|
| 共享状态 | `$context` | `state` | Redux store / Express `req` |
| 单个步骤 | 一个 Node 类 | 一个节点函数 | 一个 middleware |
| 驱动引擎 | for 循环 | `app.invoke()` | Express 的 `next()` 链 |
| 顺序执行 | 数组顺序 | 普通边 | `app.use(a); app.use(b);` |
| 条件分支 | if / 提前 return | 条件边 | 路由中间件 / 状态机库（XState） |

一句话类比：

> 链式 Node ≈ Express 中间件管道（一路 `next()` 到底）；图编排 ≈ 前端状态机库 XState（状态之间靠事件/条件跳转，可回环）。

## 8. 今日自测题

### 8.1 两种编排模式最本质的相同点是什么？

参考答案：

> ✅ 都是「共享状态 + 一组处理步骤 + 一个驱动引擎」。每一步都是「读状态 → 处理 → 写回状态」，靠引擎驱动状态往下流转。

---

### 8.2 两种编排模式最关键的不同点是什么？

参考答案：

> ✅ 「下一步去哪」由谁决定不同。链式是数组顺序写死的；图编排是靠 Edge（边）和条件函数动态决定的，因此图天然支持分支和循环。

---

### 8.3 为什么支付流程适合用链式，而 AI Agent 适合用图？

参考答案：

> ✅ 支付流程步骤固定、顺序确定，链式简单直接就够用。AI Agent 经常要「边走边判断」（比如要不要再检索、要不要调工具），存在次数不定的循环和分支，图编排能自然表达这种动态流程。

---

### 8.4 图编排里的「条件边」相当于链式里的什么？

参考答案：

> ✅ 相当于链式里的 `if / else` 分支判断，或者提前 `return`。区别是图把这个判断抽成了独立的「路由函数」，让流程结构和节点逻辑解耦。

---

### 8.5 用 JS 类比，链式 Node 和图编排分别像什么？

参考答案：

> ✅ 链式 Node 像 Express 中间件管道（一路 next 到底）；图编排像前端状态机库 XState（状态间靠条件/事件跳转，可回环）。
