> 原则:本周把"能跑"升级成"扛得住"。**缓存要防穿透雪崩击穿,并发要控数量、能超时、能取消。**

## 二、本周关键认知

前端很少关心"1 万个请求同时打过来会怎样",后端必须关心。缓存和并发进阶,核心是**在高压下不崩**。

| 前端习惯 | 后端进阶思维 | 关键差异 |
|---|---|---|
| `Promise.all` 一把全发 | Worker Pool 控制并发数 | 无限并发会打爆 DB/内存 |
| 请求超时靠 `AbortController` | `context.WithTimeout` | Go 用 context 贯穿超时和取消 |
| 缓存就是存一下 | 要防穿透/雪崩/击穿 | 缓存用不好,反而放大故障 |
| 重复请求无所谓 | singleflight 合并请求 | 热点 key 失效时防止瞬间打穿 DB |

三个缓存问题一句话记住:
- **穿透**:查一个根本不存在的 key,每次都落 DB。防:缓存空值 + 布隆过滤器。
- **击穿**:某个热点 key 突然失效,瞬间大量请求落 DB。防:singleflight / 加锁。
- **雪崩**:大量 key 同时失效,DB 被压垮。防:过期时间加随机抖动。

## 三、每天学习安排(7天)

### Day 1:缓存穿透 —— 缓存空值 + 参数校验

穿透:请求一个不存在的数据(比如 id=-1),缓存里没有,每次都打 DB。

```go
func (s *ProductService) GetProduct(ctx context.Context, id int) (*Product, error) {
    key := fmt.Sprintf("product:%d", id)

    // 1. 先查缓存
    val, err := s.redis.Get(ctx, key).Result()
    if err == nil {
        if val == "" {
            // 命中"空值缓存",说明确实不存在,直接返回
            return nil, ErrNotFound
        }
        var p Product
        json.Unmarshal([]byte(val), &p)
        return &p, nil
    }

    // 2. 缓存没有,查 DB
    p, err := s.repo.FindByID(id)
    if err != nil {
        if errors.Is(err, gorm.ErrRecordNotFound) {
            // 3. 不存在也缓存一个空值,防止反复穿透(短过期)
            s.redis.Set(ctx, key, "", 60*time.Second)
            return nil, ErrNotFound
        }
        return nil, err
    }

    // 4. 存在则正常缓存
    data, _ := json.Marshal(p)
    s.redis.Set(ctx, key, data, 30*time.Minute)
    return p, nil
}
```

**Day 1 理解要点**
- 穿透的根源是"查不存在的数据",缓存里永远没有
- 缓存空值(存空字符串,短过期)是最简单有效的防线
- 入口再加参数校验(id 必须 > 0),挡掉明显的恶意请求

### Day 2:布隆过滤器思路

缓存空值会占内存,如果攻击者用大量不同的不存在 id 请求,空值也扛不住。布隆过滤器是更省内存的方案。

**核心思路:**
- 布隆过滤器能快速回答"这个 key 一定不存在"或"可能存在"
- 有一定误判率(说存在可能不存在),但**说不存在就一定不存在**
- 请求进来先问布隆过滤器,说"不存在"直接拒绝,根本不查缓存和 DB

```go
import "github.com/bits-and-blooms/bloom/v3"

// 初始化:预计 100 万个元素,误判率 1%
var productFilter = bloom.NewWithEstimates(1000000, 0.01)

// 系统启动时,把所有商品 ID 灌进去
func warmUpFilter(ids []int) {
    for _, id := range ids {
        productFilter.Add([]byte(fmt.Sprintf("%d", id)))
    }
}

// 查询前先过滤
func (s *ProductService) GetWithFilter(ctx context.Context, id int) (*Product, error) {
    key := fmt.Sprintf("%d", id)
    if !productFilter.Test([]byte(key)) {
        // 布隆过滤器说不存在,那就一定不存在,直接返回
        return nil, ErrNotFound
    }
    // 可能存在,走正常的缓存+DB 流程
    return s.GetProduct(ctx, id)
}
```

**Day 2 理解要点**
- 布隆过滤器用位数组 + 多个哈希函数,空间效率极高
- 关键特性:**假阳性存在,假阴性不存在**(说没有就是真没有)
- 新增数据时要同步 `Add` 进过滤器,否则新数据会被误拦

### Day 3:缓存雪崩 —— 过期时间加随机抖动

雪崩:大量 key 设置了相同的过期时间,同一时刻集体失效,请求全落 DB。

```go
import "math/rand"

// 基础过期 30 分钟,再加 0~5 分钟随机抖动
func expireWithJitter(base time.Duration) time.Duration {
    jitter := time.Duration(rand.Intn(300)) * time.Second
    return base + jitter
}

func (s *ProductService) cacheProduct(ctx context.Context, p *Product) {
    key := fmt.Sprintf("product:%d", p.ID)
    data, _ := json.Marshal(p)
    // 每个 key 过期时间都不一样,分散失效
    s.redis.Set(ctx, key, data, expireWithJitter(30*time.Minute))
}
```

**雪崩的另一种成因:Redis 整个挂了。** 应对:
- 加限流/熔断,DB 扛不住时快速失败,别让 DB 也挂
- 关键数据做多级缓存(本地缓存 + Redis)

**Day 3 理解要点**
- 雪崩是"同时失效",破解方法就是"别让它们同时失效"
- 过期时间加随机抖动,几行代码就能显著缓解
- Redis 宕机也算雪崩,要靠限流熔断保护 DB

### Day 4:缓存击穿 —— singleflight 合并请求

击穿:某个**热点** key 过期的瞬间,成千上万请求同时发现缓存没了,一起去查 DB。

`singleflight` 让同一时刻对同一个 key 的多个请求,只有一个真正去查,其余的等结果。

```go
import "golang.org/x/sync/singleflight"

type ProductService struct {
    redis *redis.Client
    repo  *ProductRepo
    sf    singleflight.Group
}

func (s *ProductService) GetHot(ctx context.Context, id int) (*Product, error) {
    key := fmt.Sprintf("product:%d", id)

    // 先查缓存
    if val, err := s.redis.Get(ctx, key).Result(); err == nil {
        var p Product
        json.Unmarshal([]byte(val), &p)
        return &p, nil
    }

    // 缓存没有,用 singleflight 合并并发请求
    // 同一个 key 同时进来 1000 个请求,只有 1 个真正执行 fn,其余等它的结果
    result, err, _ := s.sf.Do(key, func() (interface{}, error) {
        p, err := s.repo.FindByID(id)
        if err != nil {
            return nil, err
        }
        data, _ := json.Marshal(p)
        s.redis.Set(ctx, key, data, expireWithJitter(30*time.Minute))
        return p, nil
    })
    if err != nil {
        return nil, err
    }
    return result.(*Product), nil
}
```

**Day 4 理解要点**
- 击穿针对的是"单个热点 key",区别于雪崩的"大量 key"
- singleflight 的 `Do` 会让同 key 的并发请求共享一次执行结果
- 第三个返回值 `shared` 表示结果是否被多个调用共享(true 说明起到了合并作用)

### Day 5:Worker Pool 模式

前端 `Promise.all` 一次发 1 万个请求会打爆浏览器和服务器。后端处理 1 万条任务,要用 Worker Pool 控制同时干活的协程数。

```go
package main

import (
    "fmt"
    "sync"
)

// 处理 1 万条任务,但最多 10 个协程同时干
func processTasks(tasks []int, workerNum int) []int {
    taskCh := make(chan int, len(tasks))
    resultCh := make(chan int, len(tasks))

    var wg sync.WaitGroup

    // 启动固定数量的 worker
    for i := 0; i < workerNum; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for task := range taskCh {
                // 模拟处理:平方
                resultCh <- task * task
            }
        }()
    }

    // 投递任务
    for _, t := range tasks {
        taskCh <- t
    }
    close(taskCh) // 关闭后 worker 的 range 会自然结束

    // 等所有 worker 干完,再关结果通道
    wg.Wait()
    close(resultCh)

    // 收集结果
    results := make([]int, 0, len(tasks))
    for r := range resultCh {
        results = append(results, r)
    }
    return results
}

func main() {
    tasks := make([]int, 10000)
    for i := range tasks {
        tasks[i] = i + 1
    }
    results := processTasks(tasks, 10) // 1 万任务,10 个 worker
    fmt.Println("处理完成,结果数:", len(results))
}
```

**Day 5 理解要点**
- Worker Pool = 固定数量的协程 + 一个任务通道,消费完为止
- `close(taskCh)` 后,worker 的 `for range` 会自动退出
- `sync.WaitGroup` 等所有 worker 结束,再关结果通道,避免向已关闭通道写入

### Day 6:协程池 ants + context 超时取消

手写 Worker Pool 能理解原理,生产里常用现成的 `ants` 协程池,更省心。

```go
import "github.com/panjf2000/ants/v2"

func runWithAnts(tasks []int) {
    var wg sync.WaitGroup
    // 创建容量 10 的池
    pool, _ := ants.NewPool(10)
    defer pool.Release()

    for _, t := range tasks {
        wg.Add(1)
        task := t // 捕获循环变量
        _ = pool.Submit(func() {
            defer wg.Done()
            fmt.Println("处理:", task*task)
        })
    }
    wg.Wait()
}
```

**context 超时 + 取消**(批处理不能无限跑下去):

```go
func batchWithTimeout(tasks []int) {
    // 整个批处理最多跑 5 秒
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    resultCh := make(chan int)

    go func() {
        for _, t := range tasks {
            select {
            case <-ctx.Done():
                // 超时或被取消,立刻停止
                fmt.Println("批处理中止:", ctx.Err())
                return
            case resultCh <- t * t:
            }
        }
        close(resultCh)
    }()

    for {
        select {
        case <-ctx.Done():
            fmt.Println("总超时,退出")
            return
        case r, ok := <-resultCh:
            if !ok {
                fmt.Println("全部处理完成")
                return
            }
            _ = r
        }
    }
}
```

**Day 6 理解要点**
- `ants` 复用协程,避免频繁创建销毁,适合海量小任务
- `context.WithTimeout` 是 Go 控制超时的标准姿势,记得 `defer cancel()`
- 每个可能阻塞的地方都要 `select { case <-ctx.Done(): }`,才能真正响应取消

## 五、常见踩坑提醒

| 坑 | 说明 |
|---|---|
| 只防穿透不缓存空值 | 不存在的 key 反复打 DB,要缓存空值 |
| 所有 key 同一过期时间 | 会雪崩,必须加随机抖动 |
| WaitGroup 没等完就关通道 | 向已关闭通道写入会 panic |
| 循环变量没捕获 | goroutine 里直接用循环变量会取到最后一个值 |
| context 忘了 cancel | 造成资源泄露,`defer cancel()` 必写 |
| 阻塞操作不监听 ctx.Done | context 取消了协程还在跑,超时形同虚设 |
