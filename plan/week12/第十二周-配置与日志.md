# 第十二周详细学习内容:配置与日志

> 主题:配置管理与结构化日志
> 目标:让项目支持多环境配置切换,并输出可用于线上排障的结构化日志。
> 原则:配置和日志是"上线前提",不是可选项。**先让日志能定位问题,再谈性能优化。**

---

## 一、本周核心目标

| 目标 | 说明 |
|---|---|
| 配置外置 | 用 Viper 读取 yaml/env,代码里不再写死配置 |
| 多环境 | 一套代码支持 dev/prod 切换,靠环境变量决定加载哪份配置 |
| 结构化日志 | 用 slog 输出 JSON 日志,带级别、时间、字段 |
| 链路追踪 | 中间件注入 trace_id,一次请求的日志能串起来 |

---

## 二、本周关键认知

你在前端也见过 `.env`、`process.env.NODE_ENV`,后端思路是一样的,只是更严格:

| 前端习惯 | 后端(Go)做法 | 关键差异 |
|---|---|---|
| `.env` + `import.meta.env` | Viper 读 yaml/env/环境变量 | 支持多来源、多环境合并 |
| `console.log` | 结构化日志(slog/zap) | 输出 JSON,机器可解析、可检索 |
| 打印字符串拼接 | 日志带 key-value 字段 | 便于按字段过滤(如按 trace_id) |
| 前端没有链路概念 | trace_id 贯穿一次请求 | 排障时能捞出一整条链路 |

三句话记住本周:

- **配置不写死**:换环境只改配置文件或环境变量,不改代码。
- **日志要能查**:线上出问题,你只有日志。日志查不到,等于没日志。
- **一次请求一个 trace_id**:没有它,高并发下日志根本分不清是谁的请求。

## 三、每天学习安排(7天)

### Day 1:为什么要外置配置 + Viper 入门

**痛点**:配置写死在代码里,换环境就要改代码、重新编译,还容易把生产密钥提交到 git。

**安装 Viper**
```bash
go get github.com/spf13/viper
```

**准备配置文件** `config/config.yaml`
```yaml
app:
  name: my-api
  port: 8080
database:
  dsn: "root:123456@tcp(127.0.0.1:3306)/dev_db"
  max_open_conns: 10
log:
  level: debug
```

**读取配置**
```go
package main

import (
    "fmt"

    "github.com/spf13/viper"
)

func main() {
    viper.SetConfigName("config") // 文件名(不带扩展名)
    viper.SetConfigType("yaml")
    viper.AddConfigPath("./config") // 搜索路径

    if err := viper.ReadInConfig(); err != nil {
        panic(fmt.Errorf("读取配置失败: %w", err))
    }

    fmt.Println("端口:", viper.GetInt("app.port"))
    fmt.Println("DSN:", viper.GetString("database.dsn"))
}
```

**Day 1 理解要点**
- Viper 类似前端的配置加载器,负责把文件读进内存
- `viper.GetInt("app.port")` 用点号访问嵌套字段
- 配置文件不进 git(加到 `.gitignore`),只提交 `config.example.yaml`

### Day 2:配置映射到结构体 + 多环境切换

**痛点**:到处用 `viper.GetString("...")` 字符串 key,容易拼错、没有类型提示。

**映射到结构体(推荐)**
```go
type Config struct {
    App struct {
        Name string `mapstructure:"name"`
        Port int    `mapstructure:"port"`
    } `mapstructure:"app"`
    Database struct {
        DSN          string `mapstructure:"dsn"`
        MaxOpenConns int    `mapstructure:"max_open_conns"`
    } `mapstructure:"database"`
    Log struct {
        Level string `mapstructure:"level"`
    } `mapstructure:"log"`
}

func LoadConfig() (*Config, error) {
    var cfg Config
    if err := viper.Unmarshal(&cfg); err != nil {
        return nil, err
    }
    return &cfg, nil
}
```

**多环境切换:用环境变量决定加载哪份文件**

准备 `config.dev.yaml` 和 `config.prod.yaml`,再按 `APP_ENV` 选择:
```go
func setupConfig() (*Config, error) {
    env := os.Getenv("APP_ENV")
    if env == "" {
        env = "dev" // 默认开发环境
    }

    viper.SetConfigName("config." + env) // config.dev / config.prod
    viper.SetConfigType("yaml")
    viper.AddConfigPath("./config")

    // 允许用环境变量覆盖:DATABASE_DSN 覆盖 database.dsn
    viper.AutomaticEnv()
    viper.SetEnvKeyReplacer(strings.NewReplacer(".", "_"))

    if err := viper.ReadInConfig(); err != nil {
        return nil, err
    }
    return LoadConfig()
}
```

**启动方式**
```bash
APP_ENV=dev  go run main.go   # 开发
APP_ENV=prod go run main.go   # 生产
```

**Day 2 理解要点**
- `mapstructure` tag 类似 json tag,负责字段名映射
- 结构体配置有类型检查和 IDE 补全,比字符串 key 安全
- `AutomaticEnv` 让环境变量能覆盖文件配置,生产密钥就走环境变量,不进文件

### Day 3:结构化日志入门(slog)

**痛点**:`fmt.Println` 打出来的是一坨文本,机器没法按字段检索,线上排障很痛。

Go 1.21+ 标准库自带 `log/slog`,不用引第三方就能输出 JSON 日志。

```go
package main

import (
    "log/slog"
    "os"
)

func main() {
    // JSON handler:输出结构化日志
    logger := slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
        Level: slog.LevelDebug,
    }))
    slog.SetDefault(logger)

    // 带字段的日志(key-value 成对出现)
    slog.Info("用户登录成功",
        "user_id", 1001,
        "ip", "192.168.1.1",
    )
    slog.Warn("库存偏低", "product_id", 55, "stock", 3)
    slog.Error("下单失败", "order_id", "A100", "err", "库存不足")
}
```

输出(每行一个 JSON,便于日志系统采集):
```json
{"time":"2026-07-02T10:00:00Z","level":"INFO","msg":"用户登录成功","user_id":1001,"ip":"192.168.1.1"}
```

**Day 3 理解要点**
- 结构化日志 = 每条日志是 JSON,字段可被检索和聚合
- `slog.Info/Warn/Error` 后面跟成对的 key-value
- 开发时可用 `slog.NewTextHandler` 输出更易读的文本,生产用 JSON

### Day 4:日志分级 + 生产/开发不同输出

**日志级别(从低到高)**:Debug < Info < Warn < Error

- **Debug**:调试细节,生产环境一般关闭
- **Info**:正常业务流水(启动、请求完成)
- **Warn**:能继续但要注意(重试、降级)
- **Error**:出错了,需要人关注

**按环境切换级别和格式**
```go
func NewLogger(env, level string) *slog.Logger {
    var lv slog.Level
    switch level {
    case "debug":
        lv = slog.LevelDebug
    case "warn":
        lv = slog.LevelWarn
    case "error":
        lv = slog.LevelError
    default:
        lv = slog.LevelInfo
    }

    opts := &slog.HandlerOptions{Level: lv}

    var handler slog.Handler
    if env == "prod" {
        // 生产:JSON,给日志系统采集
        handler = slog.NewJSONHandler(os.Stdout, opts)
    } else {
        // 开发:文本,人看着舒服
        handler = slog.NewTextHandler(os.Stdout, opts)
    }
    return slog.New(handler)
}
```

**Day 4 理解要点**
- 生产环境把级别设成 Info 或 Warn,避免 Debug 日志刷屏、拖慢磁盘
- 级别过滤在 handler 层完成,低于阈值的日志直接丢弃
- 别在循环里打 Info 日志,量太大会拖垮性能

### Day 5:trace_id 与链路追踪中间件

**痛点**:高并发下,10 个请求的日志混在一起,你根本分不清哪条日志属于哪次请求。

**方案**:每次请求生成一个唯一 trace_id,放进 `context`,后续所有日志都带上它。

**生成 trace_id 的中间件(Gin 版)**
```go
package middleware

import (
    "context"

    "github.com/gin-gonic/gin"
    "github.com/google/uuid"
)

type ctxKey string

const TraceIDKey ctxKey = "trace_id"

func TraceID() gin.HandlerFunc {
    return func(c *gin.Context) {
        // 优先用上游传来的,没有就生成
        traceID := c.GetHeader("X-Trace-Id")
        if traceID == "" {
            traceID = uuid.NewString()
        }

        // 存进请求 context,并回写响应头
        ctx := context.WithValue(c.Request.Context(), TraceIDKey, traceID)
        c.Request = c.Request.WithContext(ctx)
        c.Header("X-Trace-Id", traceID)

        c.Next()
    }
}
```

**从 context 取出 trace_id 打日志**
```go
func LoggerFromCtx(ctx context.Context) *slog.Logger {
    logger := slog.Default()
    if v := ctx.Value(TraceIDKey); v != nil {
        logger = logger.With("trace_id", v)
    }
    return logger
}

// 在 handler / service 里这样用
func GetUser(c *gin.Context) {
    log := LoggerFromCtx(c.Request.Context())
    log.Info("开始查询用户", "user_id", 1001)
    // ... 业务逻辑
    log.Info("查询用户完成")
}
```

现在同一次请求的所有日志都带同一个 `trace_id`,排障时按它一过滤,整条链路就出来了。

**Day 5 理解要点**
- trace_id 通过 `context` 在函数间传递,不用挨个函数加参数
- 中间件在请求最前面生成,后续 handler、service 都能取到
- 微服务之间通过 HTTP header 透传 trace_id,实现跨服务链路

### Day 6:实战 —— 给项目接入配置与日志

把 Day 1-5 的东西整合进你前几周的 Gin 项目。目标结构:

```
myproject/
├── config/
│   ├── config.dev.yaml
│   └── config.prod.yaml
├── internal/
│   ├── config/config.go      # 配置加载
│   ├── logger/logger.go      # 日志初始化
│   └── middleware/trace.go   # trace_id 中间件
└── main.go
```

**main.go 整合**
```go
func main() {
    // 1. 加载配置
    cfg, err := config.Setup()
    if err != nil {
        panic(err)
    }

    // 2. 初始化日志(依赖配置里的环境和级别)
    logger := logger.New(cfg.Env, cfg.Log.Level)
    slog.SetDefault(logger)

    slog.Info("服务启动",
        "app", cfg.App.Name,
        "port", cfg.App.Port,
        "env", cfg.Env,
    )

    // 3. 注册中间件
    r := gin.New()
    r.Use(gin.Recovery())
    r.Use(middleware.TraceID())
    r.Use(middleware.RequestLog()) // 记录每个请求的耗时、状态码

    // 4. 路由
    r.GET("/users/:id", handler.GetUser)

    addr := fmt.Sprintf(":%d", cfg.App.Port)
    if err := r.Run(addr); err != nil {
        slog.Error("服务退出", "err", err)
    }
}
```

**请求日志中间件(记录耗时和状态码)**
```go
func RequestLog() gin.HandlerFunc {
    return func(c *gin.Context) {
        start := time.Now()
        c.Next() // 执行后续处理

        log := LoggerFromCtx(c.Request.Context())
        log.Info("请求完成",
            "method", c.Request.Method,
            "path", c.Request.URL.Path,
            "status", c.Writer.Status(),
            "cost_ms", time.Since(start).Milliseconds(),
        )
    }
}
```

**Day 6 理解要点**
- 配置和日志在 `main` 里最早初始化,后续全项目复用
- 中间件顺序有讲究:trace_id 要在请求日志之前
- `RequestLog` 把每个请求的耗时、状态码都记下来,是最基础的可观测性

## 五、常见踩坑提醒

| 坑 | 说明 |
|---|---|
| 密钥提交到 git | 配置文件含密码就别进 git,用环境变量或 `.gitignore` |
| 结构体字段没导出 | `mapstructure` 只能填充首字母大写的导出字段 |
| 生产开 Debug 日志 | 日志量暴增,拖慢磁盘和性能,生产设 Info/Warn |
| trace_id 没透传 | 只在 handler 生成、service 拿不到,要靠 context 传递 |
| 循环里打 Info 日志 | 高频日志会成为性能瓶颈,循环内改用 Debug 或不打 |
| 忘了初始化日志就用 | `slog.SetDefault` 要在最早执行,否则用的是默认配置 |
