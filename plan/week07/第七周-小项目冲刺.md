## 二、本周关键认知

你前面学的都是"零件",本周是第一次把它们装成"整机"。几个前端不熟的后端概念,先建立认知:

| 概念 | 一句话理解 | 前端类比 |
|---|---|---|
| 事务 transaction | 一组操作要么全成功,要么全回滚 | 类似"批量操作,失败就整体撤销" |
| 一致性 | 扣了库存就必须生成订单,不能只做一半 | 前端没有对应物,这是后端核心 |
| 超卖 | 100 件库存卖出了 101 单 | 并发下的经典 bug |
| 乐观锁 | 假设不会冲突,更新时用版本号/库存条件校验 | 类似前端的"提交时带上原始值比对" |
| 悲观锁 | 假设一定冲突,先锁住这行再操作 | 类似"加把锁,别人先等着" |

记住一句话:**读操作可以用缓存和并发提速,写操作必须用事务和锁保正确。**

**核心表设计**
```go
package model

import "time"

// 商品
type Product struct {
    ID        uint      `gorm:"primaryKey" json:"id"`
    Name      string    `gorm:"size:100;not null" json:"name"`
    Price     int64     `gorm:"not null" json:"price"`  // 单位:分,避免浮点误差
    Stock     int       `gorm:"not null" json:"stock"`  // 库存
    Version   int       `gorm:"not null;default:0" json:"-"` // 乐观锁版本号
    CreatedAt time.Time `json:"created_at"`
    UpdatedAt time.Time `json:"updated_at"`
}

// 订单
type Order struct {
    ID        uint      `gorm:"primaryKey" json:"id"`
    OrderNo   string    `gorm:"size:32;uniqueIndex;not null" json:"order_no"`
    UserID    uint      `gorm:"index;not null" json:"user_id"`
    ProductID uint      `gorm:"not null" json:"product_id"`
    Quantity  int       `gorm:"not null" json:"quantity"`
    Amount    int64     `gorm:"not null" json:"amount"` // 总金额(分)
    Status    int       `gorm:"not null;default:0" json:"status"` // 0待支付 1已支付 2已取消
    CreatedAt time.Time `json:"created_at"`
    UpdatedAt time.Time `json:"updated_at"`
}

// 订单状态常量
const (
    OrderStatusPending  = 0 // 待支付
    OrderStatusPaid     = 1 // 已支付
    OrderStatusCanceled = 2 // 已取消
)
```

### Day 2:GORM 事务基础

事务是本周的命根子。GORM 有两种写法,先掌握手动 `Begin/Commit/Rollback`,再用 `db.Transaction` 简化。

**手动事务(看清全流程)**
```go
func (r *orderRepo) CreateWithTx(order *model.Order, productID uint, qty int) error {
    tx := r.db.Begin() // 开启事务
    if tx.Error != nil {
        return tx.Error
    }
    // 用 defer + recover 兜底,panic 也能回滚
    defer func() {
        if p := recover(); p != nil {
            tx.Rollback()
        }
    }()

    // 1. 扣库存
    if err := tx.Model(&model.Product{}).
        Where("id = ?", productID).
        Update("stock", gorm.Expr("stock - ?", qty)).Error; err != nil {
        tx.Rollback()
        return err
    }

    // 2. 创建订单
    if err := tx.Create(order).Error; err != nil {
        tx.Rollback()
        return err
    }

    return tx.Commit().Error // 全部成功才提交
}
```

**闭包事务(推荐,少写样板)**
```go
func (r *orderRepo) CreateOrder(order *model.Order, productID uint, qty int) error {
    return r.db.Transaction(func(tx *gorm.DB) error {
        if err := tx.Model(&model.Product{}).
            Where("id = ?", productID).
            Update("stock", gorm.Expr("stock - ?", qty)).Error; err != nil {
            return err // 返回 error 自动 Rollback
        }
        if err := tx.Create(order).Error; err != nil {
            return err
        }
        return nil // 返回 nil 自动 Commit
    })
}
```

**Day 2 理解要点**
- 闭包事务里:`return err` 自动回滚,`return nil` 自动提交,不用手写 Commit
- 事务里所有操作必须用 `tx`,不能用原来的 `r.db`,否则不在同一事务
- `defer recover` 是防止 panic 导致事务悬挂的保险

### Day 3:防超卖(乐观锁)

上面的扣库存有个致命 bug:并发下会超卖。100 库存,两个请求同时读到 100,各自扣 1,结果卖了 2 单但库存还剩 99。

**乐观锁:更新时带库存条件**
```go
func (r *orderRepo) DeductStockOptimistic(tx *gorm.DB, productID uint, qty int) error {
    // 关键:WHERE 里加 stock >= qty,数据库层面保证不会扣成负数
    result := tx.Model(&model.Product{}).
        Where("id = ? AND stock >= ?", productID, qty).
        Update("stock", gorm.Expr("stock - ?", qty))

    if result.Error != nil {
        return result.Error
    }
    // RowsAffected == 0 说明库存不足,没扣成功
    if result.RowsAffected == 0 {
        return errors.New("库存不足")
    }
    return nil
}
```

**版本号方式(更通用的乐观锁)**
```go
func (r *orderRepo) DeductWithVersion(tx *gorm.DB, p *model.Product, qty int) error {
    result := tx.Model(&model.Product{}).
        Where("id = ? AND version = ?", p.ID, p.Version).
        Updates(map[string]interface{}{
            "stock":   gorm.Expr("stock - ?", qty),
            "version": p.Version + 1,
        })
    if result.RowsAffected == 0 {
        return errors.New("并发冲突,请重试")
    }
    return nil
}
```

**Day 3 理解要点**
- 乐观锁核心就是 `WHERE` 里加条件,让数据库帮你挡住并发
- `RowsAffected == 0` 是判断"更新是否生效"的关键信号
- 库存这种简单场景用 `stock >= qty` 就够,不用非上版本号

### Day 4:防超卖(悲观锁)+ 状态流转

**悲观锁:先锁行再操作**
```go
func (r *orderRepo) DeductStockPessimistic(tx *gorm.DB, productID uint, qty int) error {
    var product model.Product
    // SELECT ... FOR UPDATE,锁住这一行,别的事务要改得等
    if err := tx.Clauses(clause.Locking{Strength: "UPDATE"}).
        First(&product, productID).Error; err != nil {
        return err
    }
    if product.Stock < qty {
        return errors.New("库存不足")
    }
    product.Stock -= qty
    return tx.Save(&product).Error
}
```

**订单状态流转(状态机)**
```go
// 定义合法的状态转移
var allowedTransitions = map[int][]int{
    model.OrderStatusPending: {model.OrderStatusPaid, model.OrderStatusCanceled},
    model.OrderStatusPaid:    {}, // 已支付不能再变
    model.OrderStatusCanceled: {},
}

func canTransition(from, to int) bool {
    for _, s := range allowedTransitions[from] {
        if s == to {
            return true
        }
    }
    return false
}

// 支付订单
func (s *orderService) Pay(orderNo string) error {
    return s.db.Transaction(func(tx *gorm.DB) error {
        var order model.Order
        if err := tx.Where("order_no = ?", orderNo).First(&order).Error; err != nil {
            return err
        }
        if !canTransition(order.Status, model.OrderStatusPaid) {
            return fmt.Errorf("订单状态 %d 不能变为已支付", order.Status)
        }
        return tx.Model(&order).Update("status", model.OrderStatusPaid).Error
    })
}
```

**Day 4 理解要点**
- 悲观锁用 `FOR UPDATE`,适合冲突激烈的场景,但性能开销大
- 乐观锁适合冲突少的场景,大多数电商库存用乐观锁就够
- 状态流转要用"状态机"约束,禁止 `已取消 → 已支付` 这种非法跳转

### Day 5:串起完整下单链路(Service 层)

把 JWT 拿用户、校验、扣库存、建订单、写 Redis 全部串起来。

```go
func (s *orderService) CreateOrder(userID, productID uint, qty int) (*model.Order, error) {
    var order *model.Order

    err := s.db.Transaction(func(tx *gorm.DB) error {
        // 1. 查商品(拿价格)
        var product model.Product
        if err := tx.First(&product, productID).Error; err != nil {
            return errors.New("商品不存在")
        }

        // 2. 乐观锁扣库存
        result := tx.Model(&model.Product{}).
            Where("id = ? AND stock >= ?", productID, qty).
            Update("stock", gorm.Expr("stock - ?", qty))
        if result.Error != nil {
            return result.Error
        }
        if result.RowsAffected == 0 {
            return errors.New("库存不足")
        }

        // 3. 创建订单
        order = &model.Order{
            OrderNo:   genOrderNo(),
            UserID:    userID,
            ProductID: productID,
            Quantity:  qty,
            Amount:    product.Price * int64(qty),
            Status:    model.OrderStatusPending,
        }
        if err := tx.Create(order).Error; err != nil {
            return err
        }
        return nil
    })
    if err != nil {
        return nil, err
    }

    // 4. 事务成功后再写缓存(缓存失败不影响下单)
    s.cacheOrder(order)
    return order, nil
}

// 生成订单号:时间戳 + 随机
func genOrderNo() string {
    return fmt.Sprintf("%d%04d", time.Now().UnixNano()/1e6, rand.Intn(10000))
}
```

**Handler 层(拿 JWT 里的 userID)**
```go
func (h *OrderHandler) Create(c *gin.Context) {
    userID := c.GetUint("user_id") // JWT 中间件里写进去的
    var req struct {
        ProductID uint `json:"product_id" binding:"required"`
        Quantity  int  `json:"quantity" binding:"required,min=1"`
    }
    if err := c.ShouldBindJSON(&req); err != nil {
        c.JSON(400, gin.H{"error": "参数错误"})
        return
    }
    order, err := h.svc.CreateOrder(userID, req.ProductID, req.Quantity)
    if err != nil {
        c.JSON(400, gin.H{"error": err.Error()})
        return
    }
    c.JSON(200, order)
}
```

**Day 5 理解要点**
- 缓存写在**事务外**:数据库是唯一真相源,缓存只是加速
- 金额在 Service 层算,别信前端传的金额
- 参数校验用 Gin 的 `binding` 标签,`min=1` 挡住 0 和负数

**要补的接口**
```
POST /register        注册
POST /login           登录(返回 JWT)
GET  /products        商品列表
POST /orders          下单(需登录,扣库存 + 建订单)
POST /orders/:no/pay  支付(状态流转)

GET  /orders          我的订单
```

**取消订单要回补库存(又一个事务)**
```go
func (s *orderService) Cancel(orderNo string) error {
    return s.db.Transaction(func(tx *gorm.DB) error {
        var order model.Order
        if err := tx.Where("order_no = ?", orderNo).First(&order).Error; err != nil {
            return err
        }
        if !canTransition(order.Status, model.OrderStatusCanceled) {
            return errors.New("当前状态不能取消")
        }
        // 改状态
        if err := tx.Model(&order).Update("status", model.OrderStatusCanceled).Error; err != nil {
            return err
        }
        // 回补库存
        return tx.Model(&model.Product{}).
            Where("id = ?", order.ProductID).
            Update("stock", gorm.Expr("stock + ?", order.Quantity)).Error
    })
}
```

**Day 6 理解要点**
- 取消订单也是"一组操作":改状态 + 回补库存,必须在一个事务里
- 每个写操作都问自己:如果中途失败,数据会不会脏?会脏就上事务

### Day 7:并发压测 + 复盘

用并发脚本验证防超卖是否真的生效。

```go
// 并发下单测试:100 库存,开 200 个 goroutine 抢
func TestNoOversell(t *testing.T) {
    var wg sync.WaitGroup
    successCount := int64(0)

    for i := 0; i < 200; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            _, err := svc.CreateOrder(1, productID, 1)
            if err == nil {
                atomic.AddInt64(&successCount, 1)
            }
        }()
    }
    wg.Wait()

    // 成功数必须 <= 100,超了就是超卖
    if successCount > 100 {
        t.Fatalf("超卖了!成功 %d 单,但库存只有 100", successCount)
    }
    t.Logf("成功下单 %d 单,无超卖", successCount)
}
```

## 五、常见踩坑提醒

| 坑 | 说明 |
|---|---|
| 事务里用了 `r.db` 而不是 `tx` | 不在同一事务,回滚无效,必须用 `tx` |
| 先查库存再扣,中间不加锁 | 经典超卖,要用 `WHERE stock >= ?` 或 `FOR UPDATE` |
| 忘记判断 `RowsAffected` | 乐观锁更新失败但没报错,以为成功了 |
| 缓存写在事务内 | 缓存失败会导致下单回滚,得不偿失 |
| 金额用 float64 | 浮点误差,`0.1+0.2 != 0.3`,一律用 int64 存分 |
| 信任前端传的金额/价格 | 价格必须后端查库计算 |
