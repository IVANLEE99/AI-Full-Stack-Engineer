# 第十四周详细学习内容:联调与自动化测试

> 主题:接口测试、前后端联调与端到端冒烟
> 目标:能自动验证核心业务主链路,前后端联调顺畅,回归有保障。
> 原则:单元测试保证"函数对",集成/端到端测试保证"链路通"。**联调不靠手点,靠脚本。**

---

## 一、本周核心目标

| 目标 | 说明 |
|---|---|
| API 测试 | 用 `httptest` 不启动真实端口测 HTTP 接口 |
| 集成测试 | 打通 handler → service → 真实/测试数据库的完整链路 |
| 端到端冒烟 | 用 Playwright 跑前端主流程,验证前后端联调 |
| 结果断言 | 同时校验接口响应和数据库落库结果 |

---

## 二、本周关键认知

- **测试金字塔**:底层是大量单元测试(第13周),中层是集成测试,顶层是少量端到端测试。越往上越慢越脆,数量越少。
- **`httptest` 是 Go 的杀手锏**:不用真的监听端口,直接把请求喂给 handler,毫秒级完成,适合测整条 HTTP 链路。
- **集成测试要连"真实依赖"**:单元测试 mock 数据库,集成测试反而要连真库(用 Docker 起一个测试库或 SQLite),验证 SQL 真的能跑。
- **前端优势用起来**:你熟悉 Playwright/Cypress,冒烟测试直接用前端技能写,验证"用户点得通"比后端断言更贴近真实。
- **联调的本质是契约对齐**:字段名、类型、状态码、错误格式,前后端要一致。自动化测试就是把契约固化成可执行代码。

## 三、每天学习安排(7天)

### Day 1:httptest 测试 HTTP 接口

`httptest` 让你不启动服务器就能测 handler。核心是 `httptest.NewRecorder()` 录制响应。

```go
package api

import (
    "encoding/json"
    "net/http"
    "net/http/httptest"
    "strings"
    "testing"
)

// 被测 handler
func CreateTodoHandler(w http.ResponseWriter, r *http.Request) {
    var body struct {
        Title string `json:"title"`
    }
    if err := json.NewDecoder(r.Body).Decode(&body); err != nil || body.Title == "" {
        http.Error(w, "title 不能为空", http.StatusBadRequest)
        return
    }
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusCreated)
    json.NewEncoder(w).Encode(map[string]any{"id": 1, "title": body.Title})
}

func TestCreateTodoHandler(t *testing.T) {
    // 构造请求
    req := httptest.NewRequest(http.MethodPost, "/todos",
        strings.NewReader(`{"title":"学测试"}`))
    // 录制响应
    rec := httptest.NewRecorder()

    CreateTodoHandler(rec, req)

    if rec.Code != http.StatusCreated {
        t.Fatalf("期望状态码 201,实际 %d", rec.Code)
    }

    var resp map[string]any
    json.NewDecoder(rec.Body).Decode(&resp)
    if resp["title"] != "学测试" {
        t.Errorf("期望 title=学测试,实际 %v", resp["title"])
    }
}
```

**Day 1 理解要点**
- `httptest.NewRequest` 造请求,`NewRecorder` 录响应,直接调 handler
- `rec.Code` 是状态码,`rec.Body` 是响应体
- 这类测试极快,可以覆盖大量参数校验、错误分支

### Day 2:测试带路由的完整服务

真实项目 handler 挂在路由上(Gin/chi/标准库 mux)。用 `router.ServeHTTP` 测整个路由。

```go
package api

import (
    "net/http"
    "net/http/httptest"
    "testing"

    "github.com/gin-gonic/gin"
)

func setupRouter() *gin.Engine {
    gin.SetMode(gin.TestMode)
    r := gin.New()
    r.GET("/ping", func(c *gin.Context) {
        c.JSON(http.StatusOK, gin.H{"msg": "pong"})
    })
    return r
}

func TestPingRoute(t *testing.T) {
    router := setupRouter()

    req := httptest.NewRequest(http.MethodGet, "/ping", nil)
    rec := httptest.NewRecorder()

    // 把请求喂给整个路由,走完中间件 + handler
    router.ServeHTTP(rec, req)

    if rec.Code != http.StatusOK {
        t.Fatalf("期望 200,实际 %d", rec.Code)
    }
    if !strings.Contains(rec.Body.String(), "pong") {
        t.Errorf("响应不含 pong: %s", rec.Body.String())
    }
}
```

**Day 2 理解要点**
- `router.ServeHTTP(rec, req)` 会走完整个中间件链,更接近真实
- Gin 记得设 `gin.TestMode`,减少无关日志
- 把 `setupRouter` 抽出来,测试和 `main` 复用同一套路由

### Day 3:集成测试(连真实数据库)

集成测试连真库,验证 SQL、事务、约束真的生效。用 SQLite 内存库最轻量,无需额外服务。

```go
package repo

import (
    "database/sql"
    "testing"

    _ "github.com/mattn/go-sqlite3"
)

// 每个测试起一个干净的内存库
func newTestDB(t *testing.T) *sql.DB {
    db, err := sql.Open("sqlite3", ":memory:")
    if err != nil {
        t.Fatal(err)
    }
    _, err = db.Exec(`CREATE TABLE users (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        name TEXT NOT NULL,
        email TEXT UNIQUE NOT NULL
    )`)
    if err != nil {
        t.Fatal(err)
    }
    // 测试结束自动关闭
    t.Cleanup(func() { db.Close() })
    return db
}

func TestInsertUser(t *testing.T) {
    db := newTestDB(t)

    res, err := db.Exec("INSERT INTO users(name,email) VALUES(?,?)", "Alice", "a@x.com")
    if err != nil {
        t.Fatalf("插入失败: %v", err)
    }
    id, _ := res.LastInsertId()
    if id != 1 {
        t.Errorf("期望 id=1,实际 %d", id)
    }

    // 验证唯一约束真的生效
    _, err = db.Exec("INSERT INTO users(name,email) VALUES(?,?)", "Bob", "a@x.com")
    if err == nil {
        t.Error("期望唯一约束报错,实际成功了")
    }
}
```

**Day 3 理解要点**
- 集成测试连真库,能发现 mock 测不出的 SQL 语法、约束问题
- `:memory:` SQLite 每次全新,互不污染,速度快
- `t.Cleanup` 注册收尾函数,比 `defer` 更适合测试
- 生产库是 MySQL/PG 时,可用 Docker 起测试库(见 Day 4)

### Day 4:用 Docker 起测试依赖 + 主链路集成测试

生产用 MySQL/PostgreSQL 时,本地/CI 用 Docker 起一个临时库跑集成测试。

```bash
# 起一个临时 PostgreSQL(测完就删)
docker run -d --name test-pg \
  -e POSTGRES_PASSWORD=test \
  -e POSTGRES_DB=appdb \
  -p 5433:5432 postgres:16

# 测试跑完清理
docker rm -f test-pg
```

主链路集成测试:一次请求验证 HTTP 响应 + 数据库落库。

```go
func TestCreateUser_EndToEnd(t *testing.T) {
    db := newTestDB(t)          // Day 3 的内存库
    router := setupRouterWithDB(db) // 路由注入真实 db

    body := `{"name":"Alice","email":"a@x.com"}`
    req := httptest.NewRequest(http.MethodPost, "/users", strings.NewReader(body))
    rec := httptest.NewRecorder()
    router.ServeHTTP(rec, req)

    // 1. 断言接口响应
    if rec.Code != http.StatusCreated {
        t.Fatalf("期望 201,实际 %d: %s", rec.Code, rec.Body.String())
    }

    // 2. 断言数据库真的写进去了
    var count int
    db.QueryRow("SELECT COUNT(*) FROM users WHERE email=?", "a@x.com").Scan(&count)
    if count != 1 {
        t.Errorf("期望库里有 1 条记录,实际 %d", count)
    }
}
```

**Day 4 理解要点**
- 集成测试双重断言:接口对 + 数据落库对,缺一不可
- Docker 起测试库适合 CI,内存库适合本地快跑
- 路由要支持注入 db,别在 handler 里全局写死连接

### Day 5:Playwright 端到端冒烟(发挥前端优势)

Playwright 驱动真实浏览器跑前端主流程,是最贴近用户的验证。你有前端底子,直接上手。

```bash
# 前端项目里安装
npm init playwright@latest
```

```ts
// tests/smoke.spec.ts —— 核心业务冒烟
import { test, expect } from '@playwright/test';

test('用户能登录并创建 Todo', async ({ page }) => {
  // 1. 打开首页
  await page.goto('http://localhost:5173');

  // 2. 登录
  await page.fill('input[name="email"]', 'a@x.com');
  await page.fill('input[name="password"]', '123456');
  await page.click('button:has-text("登录")');

  // 3. 等待跳转,确认登录成功
  await expect(page.locator('.user-name')).toHaveText('Alice');

  // 4. 创建一条 Todo
  await page.fill('input[placeholder="输入待办"]', '学 Playwright');
  await page.click('button:has-text("添加")');

  // 5. 断言列表里出现了
  await expect(page.locator('.todo-item')).toContainText('学 Playwright');
});
```

```bash
npx playwright test          # 无头运行
npx playwright test --ui     # 带界面调试,看每步做了什么
```

**Day 5 理解要点**
- 冒烟测试只覆盖"最核心的几条主链路",不追全覆盖
- Playwright 跑真浏览器,能抓到前后端字段不匹配、CORS、状态码等联调问题
- `--ui` 模式可视化回放,调试联调问题非常直观

### Day 6-7:实战——为关键流程搭端到端测试

**目标**:给你的项目搭一套自动化测试,覆盖 1-2 条核心主链路。

分层落地:

1. **API 层(Go httptest)**:核心接口的成功 + 失败分支
2. **集成层(Go + 测试库)**:注册/下单等写库主链路,双重断言
3. **端到端层(Playwright)**:1 条最重要的用户旅程冒烟

一键跑全部测试的脚本 `test.sh`:

```bash
#!/bin/bash
set -e   # 任一步失败立即退出

echo "==> 后端单元 + 集成测试"
go test ./... -cover

echo "==> 前端端到端冒烟"
cd frontend
npx playwright test

echo "==> 全部测试通过"
```

联调检查清单(前后端字段契约对齐):

| 检查项 | 说明 |
|---|---|
| 字段名 | 前后端 JSON key 一致(注意驼峰/下划线) |
| 状态码 | 成功 2xx、参数错 4xx、服务端 5xx 约定清楚 |
| 错误格式 | 统一 `{code, message}` 结构,前端好处理 |
| 时间/数字类型 | 时间戳还是字符串?金额是分还是元?约定死 |

**Day 6-7 理解要点**
- 三层测试各司其职,别用端到端测所有细节(太慢太脆)
- `set -e` 让脚本遇错即停,适合本地和 CI 复用
- 联调问题 80% 是契约不一致,测试把契约固化下来

## 五、常见踩坑提醒

| 坑 | 说明 |
|---|---|
| 端到端测太多细节 | 端到端慢又脆,只测核心主链路,细节交给单元测试 |
| 集成测试共用一个库 | 数据互相污染,每个测试用独立/干净的库 |
| 忘记断言数据库 | 只看接口返回 200 不够,要确认真的落库了 |
| Playwright 选择器写死文本 | 文案一改就挂,优先用 `data-testid` 稳定属性 |
| CI 里连不上 localhost | Docker 网络下服务名和端口要对,别照搬本地地址 |
| 测试依赖执行顺序 | 每个测试要能独立跑,不依赖前一个的结果 |
