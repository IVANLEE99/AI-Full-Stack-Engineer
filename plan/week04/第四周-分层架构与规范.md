# 第四周详细学习内容: 分层架构与规范

> 主题: 分层架构 / 规范
> 目标: 把第 3 周已经能跑的数据库项目,重构成真正可维护、可复用的 Go 后端骨架,让你开始从“会写接口”升级到“会搭项目”。
> 原则: 本周不追求功能更多,追求结构更清晰。**先把项目边界理顺,再继续堆业务。好的分层不是为了好看,是为了后面改需求不崩。**

---

## 一、本周核心目标

| 目标 | 说明 |
|---|---|
| 理解分层职责 | 明确 Handler / Service / Repository 各自负责什么 |
| 建立标准目录结构 | 用 `cmd/`、`internal/`、`configs/` 等组织项目 |
| 统一返回与错误码 | 所有接口返回结构一致,错误可预期 |
| 沉淀可复用骨架 | 为第 8 周后的第二个项目做准备 |

---

## 二、前端视角的思维转换

你之前可能更习惯“一个页面一个接口文件”或“一个模块一个 service”。Go 后端里,分层的核心不是按页面拆,而是按职责拆。

| 你熟悉的前端写法 | 后端分层的做法 | 关键差异 |
|---|---|---|
| 页面里直接调 API | Handler 只接请求,不写业务细节 | 入口和业务分离 |
| `services/user.ts` 里既调接口又拼逻辑 | Service 只处理业务规则 | 业务逻辑统一放中间层 |
| Axios 封装里顺手做转换 | Repository 只处理数据库读写 | 数据访问独立抽离 |
| 页面自己 toast 报错 | 后端统一错误码和消息 | 客户端能稳定消费错误 |
| 每个项目目录都随手搭 | 标准骨架可反复复用 | 结构开始成为资产 |

记住一句话: **分层不是为了“高级”,而是为了以后改需求时,你知道该去哪一层改。**

## 三、每天学习安排(7天)

### Day 1: 先搞清楚每一层到底干什么

先别急着改代码,先把职责模型想明白。

**推荐分层**
```text
Handler      接收 HTTP 请求,参数绑定,返回响应
Service      写业务逻辑,组合多个操作
Repository   只负责数据库读写
Model        定义数据结构
Response     统一响应结构
Middleware   通用中间件
```

**错误示例: 所有逻辑写在 Handler 里**
```go
func CreateProduct(c *gin.Context) {
    var req CreateProductReq
    if err := c.ShouldBindJSON(&req); err != nil {
        c.JSON(400, gin.H{"error": err.Error()})
        return
    }

    // 这里直接拼业务逻辑 + 数据库写入 + 返回
    if req.Price <= 0 {
        c.JSON(400, gin.H{"error": "price must be > 0"})
        return
    }

    p := Product{Name: req.Name, Price: req.Price}
    DB.Create(&p)
    c.JSON(200, p)
}
```

**更合理的思路**
```text
Handler:
- 绑定参数
- 调 Service
- 返回统一响应

Service:
- 校验业务规则
- 调 Repository

Repository:
- DB.Create / DB.Find / DB.Update
```

**Day 1 理解要点**
- Handler 不该知道太多业务细节
- Repository 不该关心 HTTP 请求和响应
- Service 是“业务规则中心”,是你后端价值最集中的地方

### Day 2: 搭标准目录结构

今天开始整理项目目录。建议先按这个骨架来:

```text
myapp/
├── cmd/
│   └── server/
│       └── main.go
├── internal/
│   ├── handler/
│   ├── service/
│   ├── repository/
│   ├── model/
│   ├── middleware/
│   └── response/
├── configs/
├── go.mod
└── README.md
```

**为什么这样分**
- `cmd/server/main.go`: 程序入口清晰
- `internal/`: 项目内部实现,不暴露给外部包
- `configs/`: 配置文件或配置说明
- `README.md`: 项目运行说明

**最小入口文件** `cmd/server/main.go`
```go
package main

import (
    "github.com/gin-gonic/gin"
)

func main() {
    r := gin.Default()
    r.GET("/ping", func(c *gin.Context) {
        c.JSON(200, gin.H{"message": "pong"})
    })
    r.Run(":8080")
}
```

**Day 2 理解要点**
- `cmd/` 是程序入口,未来你甚至可以有 `cmd/server`、`cmd/worker`
- `internal/` 适合放不希望被外部引用的核心实现
- 目录本身就是文档,结构清楚能省很多沟通成本

### Day 3: 抽 Handler / Service / Repository

今天开始真正拆代码。

**Model** `internal/model/product.go`
```go
package model

type Product struct {
    ID    uint    `json:"id" gorm:"primaryKey"`
    Name  string  `json:"name" gorm:"size:100;not null"`
    Price float64 `json:"price" gorm:"not null"`
    Stock int     `json:"stock" gorm:"default:0"`
}
```

**Repository** `internal/repository/product_repository.go`
```go
package repository

import (
    "gorm.io/gorm"
    "myapp/internal/model"
)

type ProductRepository struct {
    db *gorm.DB
}

func NewProductRepository(db *gorm.DB) *ProductRepository {
    return &ProductRepository{db: db}
}

func (r *ProductRepository) Create(product *model.Product) error {
    return r.db.Create(product).Error
}

func (r *ProductRepository) List() ([]model.Product, error) {
    var products []model.Product
    err := r.db.Order("id desc").Find(&products).Error
    return products, err
}
```

**Service** `internal/service/product_service.go`
```go
package service

import (
    "errors"
    "myapp/internal/model"
    "myapp/internal/repository"
)

type ProductService struct {
    repo *repository.ProductRepository
}

func NewProductService(repo *repository.ProductRepository) *ProductService {
    return &ProductService{repo: repo}
}

func (s *ProductService) Create(name string, price float64, stock int) (*model.Product, error) {
    if price <= 0 {
        return nil, errors.New("price must be greater than 0")
    }

    product := &model.Product{
        Name:  name,
        Price: price,
        Stock: stock,
    }

    if err := s.repo.Create(product); err != nil {
        return nil, err
    }
    return product, nil
}

func (s *ProductService) List() ([]model.Product, error) {
    return s.repo.List()
}
```

**Handler** `internal/handler/product_handler.go`
```go
package handler

import (
    "net/http"
    "myapp/internal/service"
    "github.com/gin-gonic/gin"
)

type ProductHandler struct {
    service *service.ProductService
}

func NewProductHandler(service *service.ProductService) *ProductHandler {
    return &ProductHandler{service: service}
}

type CreateProductReq struct {
    Name  string  `json:"name" binding:"required"`
    Price float64 `json:"price" binding:"required"`
    Stock int     `json:"stock" binding:"gte=0"`
}

func (h *ProductHandler) Create(c *gin.Context) {
    var req CreateProductReq
    if err := c.ShouldBindJSON(&req); err != nil {
        c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
        return
    }

    product, err := h.service.Create(req.Name, req.Price, req.Stock)
    if err != nil {
        c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
        return
    }

    c.JSON(http.StatusOK, product)
}

func (h *ProductHandler) List(c *gin.Context) {
    products, err := h.service.List()
    if err != nil {
        c.JSON(http.StatusInternalServerError, gin.H{"error": "query failed"})
        return
    }
    c.JSON(http.StatusOK, products)
}
```

**Day 3 理解要点**
- Repository 只做数据访问
- Service 只做业务规则
- Handler 只做 HTTP 层处理
- 同一个模块按这三层拆,后面需求变复杂时很稳

### Day 4: 统一响应结构

如果每个接口返回都不一样,前端会很痛苦。后端也难维护。

**统一响应结构** `internal/response/response.go`
```go
package response

type APIResponse struct {
    Code    int         `json:"code"`
    Message string      `json:"message"`
    Data    interface{} `json:"data,omitempty"`
}
```

**统一成功/失败函数**
```go
package response

import "github.com/gin-gonic/gin"

func Success(c *gin.Context, data interface{}) {
    c.JSON(200, APIResponse{
        Code:    0,
        Message: "ok",
        Data:    data,
    })
}

func Fail(c *gin.Context, code int, message string) {
    c.JSON(200, APIResponse{
        Code:    code,
        Message: message,
    })
}
```

**Handler 使用方式**
```go
if err := c.ShouldBindJSON(&req); err != nil {
    response.Fail(c, 40001, "参数错误")
    return
}

response.Success(c, product)
```

**Day 4 理解要点**
- 后端统一响应结构,前端处理会稳定很多
- 业务失败不一定非得用 HTTP 4xx/5xx,团队内部很多项目会固定 200 + 业务 code
- 关键是统一,不要一会儿 `{"error":...}` 一会儿 `{"message":...}`

### Day 5: 统一错误码设计

今天重点是把“报错”设计得像项目,不是像 demo。

**错误码常量** `internal/response/code.go`
```go
package response

const (
    CodeSuccess       = 0
    CodeInvalidParam  = 40001
    CodeUnauthorized  = 40002
    CodeNotFound      = 40004
    CodeInternalError = 50000
)
```

**统一错误消息映射**
```go
package response

var codeMessage = map[int]string{
    CodeSuccess:       "ok",
    CodeInvalidParam:  "参数错误",
    CodeUnauthorized:  "未登录或登录已过期",
    CodeNotFound:      "资源不存在",
    CodeInternalError: "服务器内部错误",
}

func Message(code int) string {
    if msg, ok := codeMessage[code]; ok {
        return msg
    }
    return "未知错误"
}
```

**改进版 Fail**
```go
func FailByCode(c *gin.Context, code int) {
    c.JSON(200, APIResponse{
        Code:    code,
        Message: Message(code),
    })
}
```

**Day 5 理解要点**
- 错误码要稳定,不要今天一个明天一个
- 前端最喜欢的是“可枚举、可判断”的错误
- 错误码系统一旦形成,后面联调效率会高很多

### Day 6: 手动依赖注入(比 wire 更适合当前阶段)

你现在不需要先学 `wire`,手动依赖注入更清楚。

**main.go 里组装依赖**
```go
package main

import (
    "github.com/gin-gonic/gin"
    "gorm.io/driver/mysql"
    "gorm.io/gorm"

    "myapp/internal/handler"
    "myapp/internal/repository"
    "myapp/internal/service"
)

func main() {
    db, err := gorm.Open(mysql.Open("root:root123@tcp(127.0.0.1:3306)/todo_db?charset=utf8mb4&parseTime=True&loc=Local"), &gorm.Config{})
    if err != nil {
        panic(err)
    }

    productRepo := repository.NewProductRepository(db)
    productService := service.NewProductService(productRepo)
    productHandler := handler.NewProductHandler(productService)

    r := gin.Default()
    api := r.Group("/api/v1")
    {
        api.POST("/products", productHandler.Create)
        api.GET("/products", productHandler.List)
    }

    r.Run(":8080")
}
```

**Day 6 理解要点**
- 依赖注入本质就是“把依赖一层层传进去”,不是某个框架的专利
- 手动注入最适合你当前阶段,清晰且够用
- 等项目特别大了,再考虑 `wire` 这类自动化工具

## 五、常见踩坑提醒

| 坑 | 说明 |
|---|---|
| 为分层而分层 | 目标是职责清晰,不是把一个简单函数强行拆成 10 层 |
| Handler 写太多业务逻辑 | 这是最常见的问题,要及时抽到 Service |
| Repository 里写业务判断 | Repository 只管查库写库,别混业务规则 |
| 错误码乱定义 | 必须统一命名和编号范围 |
| 目录结构过度复杂 | 当前阶段够清晰就行,别一上来学大厂过度设计 |
| 过早引入 wire | 你现在更需要理解依赖关系,不是追工具 |
