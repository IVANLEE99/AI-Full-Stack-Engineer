# 第八周详细学习内容:第一个可展示项目(收尾)

> 主题:第一个可展示项目(收尾)
> 目标:把第七周的订单系统收尾成一个能对外演示、能写进简历的完整项目。
> 原则:本周不加新功能,只做**交付**。**文档、联调、Git 规范、README 才是把"能跑的代码"变成"可展示项目"的关键。**

---

## 一、本周核心目标

| 目标 | 说明 |
|---|---|
| 联调收尾 | 把所有接口串起来自测一遍,补齐边界处理 |
| API 文档 | 用 swaggo 自动生成 Swagger 文档 |
| 接口集合 | 整理 Postman/Apifox 集合,一键跑全流程 |

---

## 二、本周关键认知

前端做完页面要交付,后端做完接口同样要交付。区别在于:后端的"界面"就是**文档和接口**。

| 交付物 | 作用 | 前端类比 |
|---|---|---|
| Swagger 文档 | 别人不看代码就知道怎么调接口 | 类似组件库的 Storybook |
| Postman 集合 | 一键跑通全流程,方便演示和回归 | 类似 E2E 测试脚本 |
| README | 项目说明书,怎么跑、有什么功能 | 类似项目的落地页 |
| 规范 commit | 让 git log 变成可读的开发历史 | 类似有意义的 PR 标题 |

记住一句话:**代码是给机器跑的,文档和 commit 是给人看的,两者都做好才叫交付。**

## 三、每天学习安排(7天)

### Day 1:接口联调自测 + 补边界

先把整条链路手动跑一遍,记录所有问题。

**用 curl 跑完整流程**
```bash
# 1. 注册
curl -X POST localhost:8080/register \
  -H "Content-Type: application/json" \
  -d '{"username":"tom","password":"123456"}'

# 2. 登录拿 token
TOKEN=$(curl -s -X POST localhost:8080/login \
  -H "Content-Type: application/json" \
  -d '{"username":"tom","password":"123456"}' | jq -r .token)

# 3. 下单(带 token)
curl -X POST localhost:8080/orders \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"product_id":1,"quantity":2}'

# 4. 查我的订单
curl localhost:8080/orders -H "Authorization: Bearer $TOKEN"
```

**统一响应格式(联调时暴露的第一个问题:格式不一致)**
```go
// response/response.go
package response

import "github.com/gin-gonic/gin"

type Resp struct {
    Code int         `json:"code"`
    Msg  string      `json:"msg"`
    Data interface{} `json:"data,omitempty"`
}

func OK(c *gin.Context, data interface{}) {
    c.JSON(200, Resp{Code: 0, Msg: "success", Data: data})
}

func Fail(c *gin.Context, code int, msg string) {
    c.JSON(200, Resp{Code: code, Msg: msg})
}
```

**Day 1 理解要点**
- 联调第一步永远是"手动跑一遍",问题会自己冒出来
- 统一响应格式是可展示项目的基本要求,别一个接口一个样
- 边界要补:token 过期、库存不足、重复下单都要有清晰返回

### Day 2:安装 swaggo,给接口加注释

swaggo 能从代码注释自动生成 Swagger 文档,不用手写。

**安装**
```bash
go install github.com/swaggo/swag/cmd/swag@latest
go get github.com/swaggo/gin-swagger
go get github.com/swaggo/files
```

**给 main 加全局注释**
```go
// @title           订单系统 API
// @version         1.0
// @description     前端转 Go 学习项目:订单系统
// @host            localhost:8080
// @BasePath        /
// @securityDefinitions.apikey BearerAuth
// @in header
// @name Authorization
func main() {
    // ...
}
```

**给接口加注释**
```go
// Create 创建订单
// @Summary      下单
// @Description  扣库存并创建订单(需登录)
// @Tags         订单
// @Accept       json
// @Produce      json
// @Security     BearerAuth
// @Param        body body CreateOrderReq true "下单参数"
// @Success      200 {object} response.Resp
// @Failure      400 {object} response.Resp
// @Router       /orders [post]
func (h *OrderHandler) Create(c *gin.Context) {
    // ...
}

type CreateOrderReq struct {
    ProductID uint `json:"product_id" example:"1"`
    Quantity  int  `json:"quantity" example:"2"`
}
```

**Day 2 理解要点**
- Swagger 注释就是"写在代码上方的特殊注释",swag 会扫描它们
- `@Security BearerAuth` 让文档里能填 token 测试
- 注释和代码放一起,改接口时顺手改文档,不容易漏

### Day 3:生成并挂载 Swagger UI

**生成文档**
```bash
swag init   # 在项目根目录执行,生成 docs/ 目录
```

**挂载到路由**
```go
package router

import (
    swaggerFiles "github.com/swaggo/files"
    ginSwagger "github.com/swaggo/gin-swagger"
    _ "order-system/docs" // 导入生成的 docs
    "github.com/gin-gonic/gin"
)

func Setup(r *gin.Engine) {
    // 访问 http://localhost:8080/swagger/index.html
    r.GET("/swagger/*any", ginSwagger.WrapHandler(swaggerFiles.Handler))

    // ... 其他路由
}
```

**验证**
```bash
go run main.go
# 浏览器打开 http://localhost:8080/swagger/index.html
# 能看到所有接口,能在页面上直接测试
```

**Day 3 理解要点**
- `swag init` 每次改了注释都要重跑,文档才会更新
- `_ "order-system/docs"` 前面的下划线是"只执行 init,不直接用"
- Swagger UI 是可展示项目的门面,演示时直接打开它

### Day 4:整理 Postman/Apifox 集合

Swagger 适合看,Postman/Apifox 适合"一键跑全流程"。

**组织结构**
```
订单系统集合
├── 环境变量: base_url = localhost:8080, token = (自动填)
├── 01 注册
├── 02 登录         → 测试脚本自动把 token 存到环境变量
├── 03 商品列表
├── 04 下单
├── 05 支付
├── 06 我的订单
└── 07 取消订单
```

**登录接口的自动存 token 脚本(Postman Tests 标签)**
```javascript
// 登录成功后自动把 token 存进环境变量,后续接口自动带上
const res = pm.response.json();
if (res.data && res.data.token) {
    pm.environment.set("token", res.data.token);
}
pm.test("登录成功", () => {
    pm.expect(res.code).to.eql(0);
});
```

**后续接口的 Authorization 头**
```
Authorization: Bearer {{token}}
```

**Day 4 理解要点**
- 集合按调用顺序编号,演示时从上到下点一遍就是完整流程
- 登录自动存 token,后面接口不用手动复制粘贴
- 导出集合 JSON 放进仓库,别人 import 就能用

### Day 5:写 README + 项目自查

README 是别人认识你项目的第一眼,认真写。

**README 模板**
```markdown
# 订单系统

前端转 Go 学习项目。实现了用户注册登录、商品管理、下单、
支付、取消的完整电商下单流程,重点解决并发下单的超卖问题。

## 技术栈
- Web 框架:Gin
- ORM:GORM + MySQL
- 缓存:Redis
- 认证:JWT
- 文档:Swagger (swaggo)

## 核心特性
- JWT 鉴权,密码 bcrypt 加密
- 下单用数据库事务保证一致性
- 乐观锁防超卖(200 并发抢 100 库存,不超卖)
- 订单状态机约束,禁止非法状态跳转

## 快速开始
​```bash
# 1. 启动 MySQL 和 Redis
# 2. 改 config/config.yaml 里的数据库连接
# 3. 运行
go run main.go
# 4. 访问 API 文档
open http://localhost:8080/swagger/index.html
​```

## 项目结构
​```
handler/     HTTP 处理
service/     业务逻辑(事务在这层)
repository/  数据访问
model/       数据模型
middleware/  JWT 中间件
​```
```

**项目自查清单**
```
□ 所有接口响应格式统一
□ 敏感信息(密码、密钥)不硬编码,走配置/环境变量
□ 密码存的是 bcrypt hash,不是明文
□ 有 .gitignore,不提交二进制、.env、IDE 配置
□ go.mod / go.sum 完整,别人能直接 go mod download
□ Swagger 能打开,接口能测
□ README 里的启动步骤真的能跑通
```

**Day 5 理解要点**
- README 先写"这项目是啥、解决什么问题",再写怎么跑
- 自查密码是否明文、密钥是否硬编码,这是展示项目的红线
- `.gitignore` 要有,别把编译产物和密钥提交上去

### Day 6:Git 规范提交 + 整理历史

好的 git log 本身就是项目质量的体现。

**Conventional Commits 规范**
```bash
# 格式:<type>: <描述>
git commit -m "feat: 实现下单接口和乐观锁防超卖"
git commit -m "fix: 修复取消订单未回补库存的问题"
git commit -m "docs: 添加 Swagger 注释和 README"
git commit -m "refactor: 抽取统一响应格式"
git commit -m "test: 补充并发下单压测"
```

**常用 type**
```
feat     新功能
fix      修 bug
docs     文档
refactor 重构(不改功能)
test     测试
chore    杂项(配置、依赖)
```

**推送到远程(新分支)**
```bash
git checkout -b feature/order-system
git add .
git commit -m "feat: 完成订单系统第一版"
git push -u origin feature/order-system
# 在 GitHub 上开 PR
```

**Day 6 理解要点**
- commit message 用 `type: 描述`,让 git log 一眼看懂
- 一次 commit 只做一件事,别把 10 个改动塞一个 commit
- 推到新分支再开 PR,别直接推 main

## 五、常见踩坑提醒

| 坑 | 说明 |
|---|---|
| 改了注释忘了 `swag init` | 文档不更新,一定要重新生成 |
| 密码明文存库 | 展示项目的硬伤,必须用 bcrypt |
| 密钥硬编码在代码里 | 走配置文件或环境变量,别提交到 git |
| commit 全是 "update" | git log 没信息量,用规范的 type 前缀 |
| README 步骤跑不通 | 自己按 README 从零跑一遍,别想当然 |
| 把编译产物提交上去 | 配好 .gitignore |
