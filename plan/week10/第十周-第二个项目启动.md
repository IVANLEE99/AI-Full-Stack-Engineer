# 第十周详细学习内容: 第二个项目启动

> 主题: 第二个项目启动
> 目标: 用第 8 周沉淀下来的后台骨架,在 2-3 天内快速搭出第二个小项目,验证你的项目结构、分层方式、鉴权方案、缓存策略是否真的可复用。
> 原则: 本周不是学一个全新技术点,而是**验证你前 9 周学到的东西能不能复用**。少做炫技,重点放在项目启动效率、骨架质量和快速交付能力上。

---

## 一、本周核心目标

| 目标 | 说明 |
|---|---|
| 选定第二个业务场景 | 从内容管理 / 预约系统 / 记账系统等场景中选一个 |
| 复用已有骨架 | 直接套用分层结构、统一响应、JWT、缓存、中间件 |
| 2-3 天搭出可用后端 | 把“从零开发”变成“从骨架迭代” |
| 找出骨架不足 | 记录哪些地方还不够通用、哪些设计需要调整 |

---

## 二、本周关键认知

很多人学到这里会继续堆新知识,但真正的工程化分水岭在这周。

你现在要从“我会做一个项目”升级到“我能快速启动第二个项目”。

| 第一阶段的你 | 本周开始的你 | 差别 |
|---|---|---|
| 先想怎么写代码 | 先想怎么复用骨架 | 从写功能转向搭系统 |
| 每个项目都从头搭 | 共用启动模板 | 开发效率明显提升 |
| 目录结构只服务当前项目 | 目录结构开始服务未来项目 | 架构开始有复用价值 |
| 只关注功能是否跑通 | 同时关注是否可复制 | 工程思维开始建立 |

记住一句话: **第二个项目的目标不是证明你会写更多代码,而是证明你第一个项目的骨架不是一次性脚手架。**

---

## 三、每天学习安排(7天)

### Day 1: 选题 + 定范围

本周先别写代码,先定项目范围。你的目标不是做大,而是做快。

**推荐题目(三选一)**
1. 内容管理系统 CMS
2. 预约系统 Booking
3. 个人记账系统 Ledger

**建议标准**
- 至少有 3 个核心实体
- 至少有登录态
- 至少有 1 个列表页对应的接口
- 至少有 1 个“状态流转”或“业务动作”

**示例: 预约系统的最小范围**
```text
实体:
- 用户 User
- 门店 Store
- 预约 Booking

功能:
- 用户登录
- 创建预约
- 查询预约列表
- 取消预约
```

**Day 1 产出**
写一个简单 PRD 草稿:
```md
# 第二个项目选题

项目名称: 预约系统
目标用户: 门店管理员 / 普通用户
核心实体: User, Store, Booking
本周只做:
- 登录
- 创建预约
- 预约列表
- 取消预约
不做:
- 支付
- 消息通知
- 后台运营分析
```

**Day 1 理解要点**
- 范围必须小,否则你会把“验证骨架复用性”做成“重新做一个大项目”
- 这一周不追求复杂业务,追求启动速度和结构稳定

---

### Day 2: 复制骨架 + 改业务命名

今天开始动手。最重要的是:不要从空目录重新搭。

**目标目录结构(示例)**
```text
booking-system/
├── cmd/
│   └── server/
│       └── main.go
├── internal/
│   ├── handler/
│   ├── service/
│   ├── repository/
│   ├── model/
│   ├── middleware/
│   └── response/
├── configs/
├── go.mod
└── README.md
```

**main.go 示例**
```go
package main

import (
    "github.com/gin-gonic/gin"
)

func main() {
    r := gin.Default()

    api := r.Group("/api/v1")
    {
        api.POST("/login", func(c *gin.Context) {
            c.JSON(200, gin.H{"message": "login"})
        })
    }

    r.Run(":8080")
}
```

**Day 2 理解要点**
- 今天的目标不是把业务写完,而是“把项目骨架立起来并能启动”
- 如果你需要改很多公共代码才能适配新项目,说明骨架还不够通用

---

### Day 3: 定义模型与接口

今天做两件事:
1. 先定数据结构
2. 再定接口

**示例: Booking 模型**
```go
package model

type BookingStatus string

const (
    BookingPending   BookingStatus = "pending"
    BookingConfirmed BookingStatus = "confirmed"
    BookingCancelled BookingStatus = "cancelled"
)

type Booking struct {
    ID      uint          `json:"id"`
    UserID  uint          `json:"user_id"`
    StoreID uint          `json:"store_id"`
    Date    string        `json:"date"`
    Status  BookingStatus `json:"status"`
}
```

**接口清单示例**
```text
POST   /api/v1/login
POST   /api/v1/bookings
GET    /api/v1/bookings
PATCH  /api/v1/bookings/:id/cancel
```

**Day 3 理解要点**
- 先定模型和接口,能减少后面反复改结构
- 新项目能不能快,很大程度取决于命名是否清晰、实体边界是否合理

---

### Day 4: 快速完成 CRUD 主链路

今天重点是:别过度设计,先把主链路跑通。

**Handler 示例**
```go
package handler

import (
    "net/http"

    "github.com/gin-gonic/gin"
)

type CreateBookingReq struct {
    StoreID uint   `json:"store_id" binding:"required"`
    Date    string `json:"date" binding:"required"`
}

func CreateBooking(c *gin.Context) {
    var req CreateBookingReq
    if err := c.ShouldBindJSON(&req); err != nil {
        c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
        return
    }

    c.JSON(http.StatusOK, gin.H{
        "message":  "booking created",
        "store_id": req.StoreID,
        "date":     req.Date,
    })
}
```

**Day 4 理解要点**
- 先打通接口,再补细节
- 如果你能在一天内把 3-4 个接口拉起来,说明骨架复用是成功的

---

### Day 5: 复用登录 / 鉴权 / 中间件

这天重点不是新写 JWT,而是直接复用旧项目里的鉴权逻辑。

**复用思路**
- 登录 handler 可复用 70%-80%
- JWT 签发函数可完全复用
- 鉴权中间件可完全复用
- 统一响应结构可完全复用

**中间件挂载示例**
```go
authorized := r.Group("/api/v1")
authorized.Use(JWTAuthMiddleware())
{
    authorized.POST("/bookings", CreateBooking)
    authorized.GET("/bookings", ListBookings)
    authorized.PATCH("/bookings/:id/cancel", CancelBooking)
}
```

**Day 5 理解要点**
- 能复用的就别重写
- 如果 JWT、中间件、响应结构无法无痛迁移,骨架就还不成熟

---

### Day 6: 记录骨架问题并抽象优化

今天开始做“复盘式开发”。

你需要记录这些问题:
- 哪些目录结构太死板
- 哪些中间件耦合旧项目命名
- 哪些 response / error code 设计不通用
- 哪些配置项应该抽到公共层

**建议输出一个文档** `skeleton-review.md`
```md
# 骨架复用复盘

## 能直接复用的
- JWT 签发
- 鉴权中间件
- 统一响应结构
- 路由分组模式

## 需要改进的
- 错误码命名仍偏向订单系统
- repository 接口定义不够统一
- config 层缺少环境变量说明
```

**Day 6 理解要点**
- 骨架不是写出来就结束,而是要通过第二个项目不断打磨
- 真正可复用的系统,一定是“被第二次使用后改出来的”

---

### Day 7: 小结与阶段验证

本周最后一天,你要回答两个问题:
1. 第二个项目是不是比第一个项目启动快很多?
2. 你的骨架能不能支持未来第 3 个、第 4 个项目?

**阶段产出要求**
- 第二个项目目录已建立
- 至少 3 个核心接口跑通
- 登录和鉴权链路正常
- 产出骨架复盘文档

## 五、常见踩坑提醒

| 坑 | 说明 |
|---|---|
| 一开始就想做大而全 | 本周目标是验证骨架,不是重做大项目 |
| 新项目又从零搭目录 | 这会让你失去复用验证的意义 |
| 为了复用强行抽象 | 抽象太早会让代码更绕,先跑起来再抽 |
| 业务命名沿用旧项目 | 会让骨架看起来像拷贝拼凑,要及时重命名 |
| 第二个项目范围过大 | 2-3 天无法出成果,就说明范围控制失败 |
